# xv6: 一个简单的类Unix操作系统

# 第 1 章

## 操作系统接口

操作系统的工作是在多个程序之间共享一台计算机，并提供比硬件本身支持的更有用的服务集。操作系统管理和抽象底层硬件，这样，例如，文字处理器就不必关心使用哪种类型的磁盘硬件。操作系统在多个程序之间共享硬件，以便它们可以（或看起来可以）同时运行。最后，操作系统为程序之间的交互提供了受控的方式，以便它们可以共享数据或协同工作。

操作系统通过一个接口向用户程序提供服务。设计一个好的接口非常困难。一方面，我们希望接口简单而狭窄，因为这使得实现更正确。另一方面，我们可能倾向于为应用程序提供许多复杂的功能。解决这种矛盾的关键是设计依赖于少数可以组合以提供广泛通用性的机制的接口。

本书使用一个操作系统作为具体例子来说明操作系统的概念。这个操作系统 xv6 提供了由肯·汤普逊和丹尼斯·里奇引入的 Unix 操作系统 [15] 的基本接口，并模仿了 Unix 的内部设计。Unix 提供了一个狭窄的接口，其机制组合得很好，提供了令人惊讶的通用性。这个接口非常成功，以至于现代操作系统——BSD、Linux、macOS、Solaris，甚至在较小程度上包括 Microsoft Windows——都具有类 Unix 接口。理解 xv6 是理解这些系统及其他许多系统的一个良好开端。

如图 1.1 所示，xv6 采用内核的传统形式，内核是一个为运行中的程序提供服务的特殊程序。每个运行中的程序称为一个进程，它包含指令、数据和栈的内存。指令实现程序的计算。数据是计算所操作的变量。栈组织程序的函数调用。一台给定的计算机通常有许多进程，但只有一个内核。

当一个进程需要调用内核服务时，它会发起一个系统调用，即操作系统接口中的一个调用。系统调用进入内核；内核执行服务并返回。因此，一个进程在用户空间和内核空间之间交替执行。

内核利用 CPU¹ 提供的硬件保护机制来确保每个在用户空间执行的进程只能访问其自己的内存。内核

Shell 是一个普通程序，它从用户读取命令并执行它们。Shell 是一个用户程序而不是内核的一部分，这一事实说明了系统调用接口的强大之处：shell 并没有什么特别之处。这也意味着 shell 很容易被替换；因此，现代 Unix 系统有多种 shell 可供选择，每种都有自己独特的用户界面和脚本功能。xv6 shell 是对 Unix Bourne shell 本质的一个简单实现。它的实现可以在 (user/sh.c:1) 找到。

### 1.1 进程与内存

一个 xv6 进程由用户空间内存（指令、数据和栈）和内核私有的每进程状态组成。Xv6 对进程进行分时共享：它透明地在等待执行的一组进程之间切换可用的 CPU。当一个进程没有执行时，xv6 保存其 CPU 寄存器，并在下次运行该进程时恢复它们。内核为每个进程关联一个进程标识符（PID）。

一个进程可以使用 `fork` 系统调用来创建一个新进程。Fork 使新进程拥有与调用进程完全相同的内存内容（包括指令和数据）。Fork 在原始进程和新进程中都有返回值。在原始进程中，fork 返回新进程的 PID。在新进程中，fork 返回零。原始进程和新进程通常被称为父进程和子进程。

例如，考虑以下用 C 编程语言 [6] 编写的程序片段：

```c
int pid = fork();
```

```c
if (pid > 0) {
    printf("parent: child=%d\n", pid);
    pid = wait((int*)0);
```

`exit` 系统调用导致调用进程停止执行并释放资源，如内存和打开的文件。`exit` 接受一个整数状态参数，通常 0 表示成功，1 表示失败。`wait` 系统调用返回当前进程的一个已退出（或被杀死）的子进程的 PID，并将子进程的退出状态复制到传递给 `wait` 的地址；如果调用者的子进程都没有退出，`wait` 会等待其中一个退出。如果调用者没有子进程，`wait` 立即返回 -1。如果父进程不关心子进程的退出状态，它可以向 `wait` 传递一个 0 地址。



在该示例中，输出行
```
parent: child=1234
child: exiting
```
可能以任意顺序出现（甚至交错出现），具体取决于父进程和子进程谁先执行到它的 `printf` 调用。在子进程退出后，父进程的 `wait` 返回，导致父进程打印
```
parent: child 1234 is done
```
尽管子进程初始时拥有与父进程相同的内存内容，但父进程和子进程使用不同的内存和不同的寄存器执行：在一个进程中改变变量不会影响另一个进程。例如，当 `wait` 的返回值在父进程中被存入 `pid` 时，它不会改变子进程中的变量 `pid`。子进程中的 `pid` 值仍然为零。

`exec` 系统调用用从文件系统中存储的文件加载的新内存映像替换调用进程的内存。该文件必须具有特定的格式，指定文件的哪部分包含指令、哪部分是数据、从哪条指令开始执行等。Xv6 使用 ELF 格式，第 3 章将更详细地讨论它。当 `exec` 成功时，它不会返回到调用程序；相反，从文件加载的指令从 ELF 头中声明的入口点开始执行。`exec` 接受两个参数：包含可执行文件的文件名和一个字符串参数数组。例如：

```c
char *argv[3];
argv[0] = "echo";
argv[1] = "hello";
argv[2] = 0;
exec("/bin/echo", argv);
printf("exec error\n");
```

这个片段用运行中的 `/bin/echo` 程序实例替换调用程序，参数列表为 `echo hello`。大多数程序忽略参数数组的第一个元素，该元素通常约定为程序的名称。

xv6 shell 使用上述调用来代表用户运行程序。shell 的主要结构很简单；见 `main` (user/sh.c:145)。主循环使用 `getcmd` 从用户读取一行输入。然后它调用 `fork`，这会创建 shell 进程的一个副本。父进程调用 `wait`，而子进程运行命令。例如，如果用户在 shell 中键入了 "echo hello"，`runcmd` 将以 "echo hello" 作为参数被调用。`runcmd` (user/sh.c:58) 运行实际的命令。对于 "echo hello"，它将调用 `exec` (user/sh.c:78)。如果 `exec` 成功，那么子进程将执行来自 `echo` 的指令，而不是 `runcmd`。在某个时刻，`echo` 将调用 `exit`，这将导致父进程从 `main` 中的 `wait` 返回 (user/sh.c:145)。

您可能会好奇为什么 `fork` 和 `exec` 不合并为一个单独的调用；我们稍后会看到 shell 在其 I/O 重定向的实现中利用了这种分离。为了避免创建重复进程然后立即替换它（使用 `exec`）的浪费，操作系统内核通过使用虚拟内存技术（如写时复制（见 4.6 节））来优化这种情况下 `fork` 的实现。



### 1.2 I/O 与文件描述符

文件描述符是一个小整数，表示一个进程可以从中读取或写入的内核管理对象。一个进程可以通过打开文件、目录或设备，或者通过创建管道，或者通过复制现有描述符来获得文件描述符。为简单起见，我们通常将文件描述符所指的对象称为“文件”；文件描述符接口抽象了文件、管道和设备之间的差异，使它们看起来都像字节流。我们将输入和输出称为 I/O。

在内部，xv6 内核使用文件描述符作为每个进程表的索引，因此每个进程都有一个从零开始的私有文件描述符空间。按照惯例，进程从文件描述符 0（标准输入）读取，向文件描述符 1（标准输出）写入输出，并向文件描述符 2（标准错误）写入错误消息。正如我们将看到的，shell 利用这个约定来实现 I/O 重定向和管道。Shell 确保它始终有三个打开的文件描述符 (user/sh.c:151)，默认情况下是控制台的文件描述符。

`read` 和 `write` 系统调用从由文件描述符命名的打开文件中读取字节和写入字节。调用 `read(fd, buf, n)` 从文件描述符 `fd` 读取最多 `n` 个字节，将它们复制到 `buf` 中，并返回读取的字节数。每个指向文件的文件描述符都有一个与之关联的偏移量。`read` 从当前文件偏移量读取数据，然后将该偏移量向前移动读取的字节数：后续的 `read` 将返回第一次 `read` 返回的字节之后的字节。当没有更多字节可读时，`read` 返回 0 表示文件结束。

调用 `write(fd, buf, n)` 将 `buf` 中的 `n` 个字节写入文件描述符 `fd`，并返回写入的字节数。只有在发生错误时，写入的字节数才会少于 `n`。与 `read` 一样，`write` 在当前文件偏移量处写入数据，然后将该偏移量向前移动写入的字节数：每次 `write` 从前一次结束的地方开始写入。

以下程序片段（构成了 `cat` 程序的本质）将数据从其标准输入复制到其标准输出。如果发生错误，它会向标准错误写入一条消息。

```c
char buf[512];
int n;
for (;;) {
    n = read(0, buf, sizeof buf);
    if (n == 0)
        break;
    if (n < 0) {
        fprintf(2, "read error\n");
        exit(1);
    }
    if (write(1, buf, n) != n) {
        fprintf(2, "write error\n");
        exit(1);
    }
}
```



在代码片段中需要注意的重要一点是，`cat` 不知道它是从文件、控制台还是管道读取。同样，`cat` 也不知道它是输出到控制台、文件还是其他任何地方。文件描述符的使用以及文件描述符 0 是输入、文件描述符 1 是输出的约定，使得 `cat` 的实现变得简单。

`close` 系统调用释放一个文件描述符，使其可以被未来的 `open`、`pipe` 或 `dup` 系统调用（见下文）重新使用。新分配的文件描述符始终是当前进程中编号最小的未使用描述符。

文件描述符和 `fork` 的交互使得 I/O 重定向易于实现。`Fork` 复制父进程的文件描述符表及其内存，因此子进程开始时拥有与父进程完全相同的打开文件。`exec` 系统调用替换调用进程的内存，但保留其文件表。这种行为允许 shell 通过 fork、在子进程中重新打开选定的文件描述符、然后调用 `exec` 来运行新程序，从而实现 I/O 重定向。以下是 shell 为命令 `cat < input.txt` 运行的简化代码：

```c
char *argv[2];
argv[0] = "cat";
argv[1] = 0;
if (fork() == 0) {
    close(0);
    open("input.txt", O_RDONLY);
    exec("cat", argv);
}
```

在子进程关闭文件描述符 0 之后，`open` 保证为新打开的 `input.txt` 使用该文件描述符：0 将是最小可用的文件描述符。然后 `cat` 执行时，文件描述符 0（标准输入）指向 `input.txt`。父进程的文件描述符不会被此序列改变，因为它只修改了子进程的描述符。

xv6 shell 中的 I/O 重定向代码正是以这种方式工作的 (user/sh.c:82)。回想一下，在代码的这一点，shell 已经 fork 了子 shell，并且 `runcmd` 将调用 `exec` 来加载新程序。

`open` 的第二个参数由一组标志组成，以位表示，用于控制 `open` 的行为。可能的值在文件控制（fcntl）头文件 (kernel/fcntl.h:1-5) 中定义：`O_RDONLY`、`O_WRONLY`、`O_RDWR`、`O_CREATE` 和 `O_TRUNC`，它们指示 `open` 以只读、只写、读写方式打开文件，如果文件不存在则创建文件，并将文件截断为零长度。

现在应该清楚为什么将 `fork` 和 `exec` 分开是有帮助的：在两者之间，shell 有机会重定向子进程的 I/O，而不会干扰主 shell 的 I/O 设置。

===== 第 15 页 =====

可以想象一个假设的组合 `forkexec` 系统调用，但使用这样的调用进行 I/O 重定向的选项似乎很笨拙。Shell 可以在调用 `forkexec` 之前修改自己的 I/O 设置（然后再撤销这些修改）；或者 `forkexec` 可以将 I/O 重定向的指令作为参数；或者（最不吸引人）像 `cat` 这样的每个程序都必须被教导自己进行 I/O 重定向。

尽管 `fork` 复制了文件描述符表，但每个底层的文件偏移量在父进程和子进程之间是共享的。考虑这个例子：

```c
if (fork() == 0) {
    write(1, "hello ", 6);
    exit(0);
} else {
    wait(0);
    write(1, "world\n", 6);
}
```

在这个片段的最后，连接到文件描述符 1 的文件将包含数据 "hello world"。父进程中的 `write`（由于 `wait`，仅在子进程完成后运行）从子进程的 `write` 结束的地方继续。这种行为有助于从 shell 命令序列（如 `(echo hello; echo world) > output.txt`）产生顺序输出。

`dup` 系统调用复制一个现有的文件描述符，返回一个新的指向相同底层 I/O 对象的文件描述符。两个文件描述符共享一个偏移量，就像 `fork` 复制的文件描述符一样。这是将 "hello world" 写入文件的另一种方式：

```c
fd = dup(1);
write(1, "hello ", 6);
write(fd, "world\n", 6);
```

两个文件描述符如果通过一系列的 `fork` 和 `dup` 调用从同一个原始文件描述符派生而来，则它们共享偏移量。否则，文件描述符不共享偏移量，即使它们是为同一个文件调用 `open` 的结果。`Dup` 允许 shell 实现像这样的命令：`ls existing-file non-existing-file > tmp1 2>&1`。这里的 `2>&1` 告诉 shell 给命令一个文件描述符 2，它是描述符 1 的副本。现有文件的名称和针对不存在文件的错误消息都将出现在文件 `tmp1` 中。xv6 shell 不支持错误文件描述符的 I/O 重定向，但您现在知道如何实现它了。

文件描述符是一个强大的抽象，因为它们隐藏了它们所连接对象的细节：写入文件描述符 1 的进程可能正在写入文件、像控制台这样的设备或管道。

### 1.3 管道

管道是一个小的内核缓冲区，以一对文件描述符的形式暴露给进程，一个用于读，一个用于写。将数据写入管道的一端可以使这些数据从管道的另一端读取。管道为进程提供了一种通信方式。

===== 第 16 页 =====

以下示例代码运行 `wc` 程序，其标准输入连接到管道的读端。

```c
int p[2];
char *argv[2];
argv[0] = "wc";
argv[1] = 0;
pipe(p);
if (fork() == 0) {
    close(0);
    dup(p[0]);
    close(p[0]);
    close(p[1]);
    exec("/bin/wc", argv);
} else {
    close(p[0]);
    write(p[1], "hello world\n", 12);
    close(p[1]);
}
```

程序调用 `pipe`，它创建一个新管道，并将读和写文件描述符记录在数组 `p` 中。在 `fork` 之后，父进程和子进程都有指向该管道的文件描述符。子进程调用 `close` 和 `dup` 使文件描述符零指向管道的读端，关闭数组 `p` 中的文件描述符，并调用 `exec` 运行 `wc`。当 `wc` 从其标准输入读取时，它从管道读取。父进程关闭管道的读端，向管道写入数据，然后关闭写端。

如果没有数据可用，在管道上的 `read` 会等待数据被写入，或者等待所有指向写端的文件描述符被关闭；在后一种情况下，`read` 将返回 0，就像到达数据文件的末尾一样。`read` 会阻塞直到不可能有新数据到达，这是子进程在执行上述 `wc` 之前关闭管道写端的重要原因之一：如果 `wc` 的某个文件描述符指向了管道的写端，`wc` 将永远看不到文件结束。

xv6 shell 实现管道，例如 `grep fork sh.c | wc -l`，其方式类似于上述代码 (user/sh.c:100)。子进程创建一个管道来连接管道的左端和右端。然后它为管道的左端调用 `fork` 和 `runcmd`，并为管道的右端调用 `fork` 和 `runcmd`，然后等待两者完成。管道的右端可能是一个本身包含管道的命令（例如 `a | b | c`），它会进一步 fork 出两个新的子进程（一个用于 `b`，一个用于 `c`）。因此，shell 可能创建一个进程树。这棵树的叶子是命令，内部节点是等待左右子进程完成的进程。

原则上，可以让内部节点运行管道的左端，但这样做正确会使实现复杂化。考虑仅仅做以下修改：更改 `sh.c`，使其不为 `p->left` 进行 fork，而是在内部进程中运行 `runcmd(p->left)`。那么，

### 1.4 文件系统

xv6 文件系统提供数据文件（包含未解释的字节数组）和目录（包含指向数据文件和其他目录的命名引用）。目录形成一棵树，从一个称为根目录的特殊目录开始。像 `/a/b/c` 这样的路径指的是根目录 `/` 下名为 `a` 的目录中名为 `b` 的目录内名为 `c` 的文件或目录。不以 `/` 开头的路径相对于调用进程的当前目录进行计算，当前目录可以通过 `chdir` 系统调用更改。以下两个代码片段打开相同的文件（假设所有涉及的目录都存在）：

### 1.5 现实世界

Unix 将“标准”文件描述符、管道以及在其上进行操作的便捷 shell 语法相结合，是编写通用可重用程序的一项重大进步。这个想法催生了一种“软件工具”文化，这是 Unix 强大和流行的主要原因，而 shell 是第一个所谓的“脚本语言”。Unix 系统调用接口至今仍在 BSD、Linux 和 macOS 等系统中延续。

Unix 系统调用接口已通过可移植操作系统接口（POSIX）标准进行了标准化。Xv6 不符合 POSIX：它缺少许多系统调用（包括像 `lseek` 这样的基本调用），并且它提供的许多系统调用与标准不同。我们对 xv6 的主要目标是简单和清晰，同时提供一个简单的类 Unix 系统调用接口。有几个人通过添加更多的系统调用和一个简单的 C 库来扩展 xv6，以便运行基本的 Unix 程序。然而，现代内核提供了比 xv6 多得多的系统调用和更多种类的内核服务。例如，它们支持网络、窗口系统、用户级线程、许多设备的驱动程序等等。现代内核不断快速发展，并提供许多超越 POSIX 的功能。

Unix 使用单一的文件名和文件描述符接口集统一了对多种类型资源（文件、目录和设备）的访问。这个想法可以扩展到更多种类的资源；一个很好的例子是 Plan 9 [14]，它将“资源即文件”的概念应用于网络、图形等。然而，大多数 Unix 衍生操作系统并未遵循这条路线。

文件系统和文件描述符是强大的抽象。即便如此，操作系统接口还有其他模型。Multics（Unix 的前身）以一种使其看起来像内存的方式抽象了文件存储，产生了一种截然不同的接口风格。Multics 设计的复杂性直接影响到了 Unix 的设计者，他们试图构建更简单的东西。

Xv6 没有提供用户的概念或保护一个用户免受另一个用户侵害的概念；用 Unix 的术语来说，所有 xv6 进程都以 root 身份运行。

本书研究 xv6 如何实现其类 Unix 接口，但这些思想和方法不仅适用于 Unix。任何操作系统都必须将进程多路复用到底层硬件上，隔离进程，并提供受控的进程间通信机制。在学习 xv6 之后，您应该能够研究其他更复杂的操作系统。

# 2 操作系统组织

操作系统的一个关键要求是同时支持多项活动。例如，使用第 1 章描述的系统调用接口，一个进程可以通过 `fork` 启动新进程。操作系统必须在这些进程之间对计算机的资源进行分时共享。例如，即使进程数多于硬件 CPU 数，操作系统也必须确保所有进程都有机会执行。操作系统还必须安排进程之间的隔离。也就是说，如果一个进程有缺陷并且发生故障，它不应该影响不依赖于该故障进程的进程。然而，完全隔离又过于严格，因为进程应该可以有目的地进行交互；管道就是一个例子。因此，操作系统必须满足三个要求：多路复用、隔离和交互。

本章概述了如何组织操作系统以实现这三个要求。事实证明有多种方法可以做到这一点，但本文关注的是以宏内核为中心的主流设计，许多 Unix 操作系统都使用这种设计。本章还概述了 xv6 进程，这是 xv6 中的隔离单元，以及 xv6 启动时第一个进程的创建。

Xv6 在多核 RISC-V 微处理器上运行，其大部分底层功能（例如其进程实现）是特定于 RISC-V 的。RISC-V 是一个 64 位 CPU，xv6 是用“LP64”C 编写的，这意味着 C 编程语言中的 long(L) 和指针(P) 是 64 位，但它是 32 位的。本书假设读者对某种体系结构进行过一些机器级编程，并在遇到 RISC-V 特定概念时进行介绍。RISC-V 的有用参考是“RISC-V 读本：开放架构图集”[13]。用户级 ISA [2] 和特权架构 [1] 是官方规范。

完整计算机中的 CPU 周围环绕着支持硬件，其中大部分以 I/O 接口的形式存在。Xv6 是为 qemu 的“machine virt”选项模拟的支持硬件而编写的。这包括 RAM、包含启动代码的 ROM、连接到用户键盘/屏幕的串行连接以及用于存储的磁盘。

### 2.1 抽象物理资源

遇到操作系统时人们可能问的第一个问题是，到底为什么需要它？也就是说，可以将图 1.2 中的系统调用实现为一个库，应用程序与之链接。在这个方案中，每个应用程序甚至可以拥有自己根据其需求定制的库。应用程序可以直接与硬件资源交互，并以最适合应用程序的方式使用这些资源（例如，实现高或可预测的性能）。一些用于嵌入式设备或实时系统的操作系统就是以这种方式组织的。

这种库方法的缺点是，如果有多个应用程序在运行，这些应用程序必须是行为良好的。例如，每个应用程序必须定期放弃 CPU，以便其他应用程序可以运行。如果所有应用程序互相信任并且没有缺陷，这种协作式分时方案也许可以接受。但更典型的情况是应用程序之间互不信任，并且存在缺陷，因此人们通常希望获得比协作式方案更强的隔离。

为了实现强隔离，禁止应用程序直接访问敏感的硬件资源，并将这些资源抽象为服务，这很有帮助。例如，Unix 应用程序仅通过文件系统的 `open`、`read`、`write` 和 `close` 系统调用与存储交互，而不是直接读写磁盘。这为应用程序提供了路径名的便利，并允许操作系统（作为接口的实现者）管理磁盘。即使不关心隔离，有意交互的程序（或者只是希望互相不干扰的程序）也可能发现文件系统是比直接使用磁盘更方便的抽象。

类似地，Unix 透明地在进程之间切换硬件 CPU，根据需要保存和恢复寄存器状态，这样应用程序就不必意识到时间共享。这种透明性允许操作系统共享 CPU，即使某些应用程序处于无限循环中。

再举一个例子，Unix 进程使用 `exec` 来构建其内存映像，而不是直接与物理内存交互。这允许操作系统决定将进程放在内存中的何处；如果内存紧张，操作系统甚至可以将进程的部分数据存储在磁盘上。`Exec` 还为用户提供了使用文件系统存储可执行程序映像的便利。

Unix 进程之间的许多交互形式是通过文件描述符进行的。文件描述符不仅抽象了许多细节（例如，管道或文件中的数据存储在哪里），而且它们还以简化交互的方式进行定义。例如，如果管道中的一个应用程序失败，内核会为管道中的下一个进程生成一个文件结束信号。

图 1.2 中的系统调用接口经过精心设计，既提供了程序员的便利，又提供了强隔离的可能性。Unix 接口不是抽象资源的唯一方式，但它已被证明是非常好的一种。

### 2.2 用户模式、监督模式与系统调用

强隔离需要在应用程序和操作系统之间建立一个硬边界。如果应用程序出错，我们不希望操作系统失败或其他应用程序

===== 第 23 页 =====

失败。相反，操作系统应该能够清理失败的应用程序并继续运行其他应用程序。为了实现强隔离，操作系统必须安排应用程序不能修改（甚至读取）操作系统的数据结构和指令，并且应用程序不能访问其他进程的内存。

CPU 为强隔离提供硬件支持。例如，RISC-V 有三种 CPU 执行指令的模式：机器模式、监督模式和用户模式。在机器模式下执行的指令具有完全的权限；CPU 从机器模式启动。机器模式主要用于配置计算机。Xv6 在机器模式下执行几行代码，然后切换到监督模式。

在监督模式下，CPU 被允许执行特权指令：例如，启用和禁用中断、读取和写入保存页表地址的寄存器等。如果用户模式下的应用程序试图执行特权指令，那么 CPU 不会执行该指令，而是切换到监督模式，以便监督模式下的代码可以终止该应用程序，因为它做了不应该做的事情。第 1 章的图 1.1 说明了这种组织。应用程序只能执行用户模式指令（例如，加法等），被称为在用户空间中运行，而监督模式下的软件也可以执行特权指令，被称为在内核空间中运行。在内核空间（或监督模式）中运行的软件称为内核。

想要调用内核函数（例如 xv6 中的 `read` 系统调用）的应用程序必须过渡到内核；应用程序不能直接调用内核函数。CPU 提供了一条特殊指令，将 CPU 从用户模式切换到监督模式，并在内核指定的入口点进入内核。（RISC-V 为此目的提供了 `ecall` 指令。）一旦 CPU 切换到监督模式，内核就可以验证系统调用的参数（例如，检查传递给系统调用的地址是否属于应用程序的内存），决定是否允许应用程序执行请求的操作（例如，检查应用程序是否被允许写入指定的文件），然后拒绝或执行它。控制进入监督模式的入口点对内核来说很重要；如果应用程序可以决定内核入口点，恶意应用程序就可以，例如，在跳过参数验证的点进入内核。

### 2.3 内核组织

一个关键的设计问题是操作系统的哪部分应该在监督模式下运行。一种可能性是整个操作系统驻留在内核中，因此所有系统调用的实现都在监督模式下运行。这种组织称为宏内核。

在这种组织中，整个操作系统以完全的硬件特权运行。这种组织很方便，因为操作系统设计者不需要决定操作系统的哪部分不需要完全的硬件特权。此外，操作系统的不同部分更容易协作。例如，一个操作系统可能有一个缓冲区缓存，可以被文件系统和虚拟内存系统共享。

宏内核组织的一个缺点是操作系统不同部分之间的接口通常很复杂（正如我们在本文其余部分将看到的），因此

操作系统开发者很容易犯错。在宏内核中，一个错误是致命的，因为监督模式下的错误通常会导致内核失败。如果内核失败，计算机停止工作，因此所有应用程序也都会失败。计算机必须重新启动才能再次运行。

为了降低内核中出错的风险，操作系统设计者可以最小化在监督模式下运行的操作系统代码量，并在用户模式下执行操作系统的大部分代码。这种内核组织称为微内核。

在微内核中，内核接口由一些底层函数组成，用于启动应用程序、发送消息、访问设备硬件等。这种组织允许内核相对简单，因为大部分操作系统驻留在用户级服务器中。

在现实世界中，宏内核和微内核都很流行。许多 Unix 内核是宏内核。例如，Linux 有一个宏内核，尽管一些操作系统功能作为用户级服务器运行（例如，窗口系统）。Linux 为操作系统密集型应用程序提供了高性能，部分原因是内核子系统可以紧密集成。

像 Minix、L4 和 QNX 这样的操作系统被组织为带有服务器的微内核，并在嵌入式环境中得到广泛部署。L4 的一个变体 seL4 非常小，以至于它已被验证了内存安全性和其他安全属性 [7]。

操作系统开发者之间关于哪种组织更好存在很多争论，并且没有确凿的证据支持哪一种。此外，这在很大程度上取决于“更好”的含义：更快的性能、更小的代码量、内核的可靠性、整个操作系统（包括用户级服务）的可靠性等等。

还有一些实际考虑因素可能比哪种组织的问题更重要。有些操作系统是微内核，但出于性能原因，它们在核心态中运行一些用户级服务。有些操作系统是宏内核，因为它们是那样开始的，并且没有动力转向纯粹的微内核组织，因为新特性可能比重写现有操作系统以适应微内核设计更重要。

### 2.5 进程概述

xv6（与其他 Unix 操作系统一样）中的隔离单元是进程。进程抽象阻止一个进程破坏或窥探另一个进程的内存、CPU、文件描述符等。它还阻止一个进程破坏内核本身，这样进程就无法破坏内核的隔离机制。内核必须谨慎地实现进程抽象，因为有缺陷或恶意的应用程序可能欺骗内核或硬件做坏事（例如，绕过隔离）。内核用来实现进程的机制包括用户/监督模式标志、地址空间和线程的时间切片。

为了帮助实施隔离，进程抽象向程序提供了它拥有自己私有机器的错觉。一个进程为程序提供了一个看似私有的内存系统或地址空间，其他进程无法读取或写入。一个进程还为程序提供了一个看似自己的 CPU 来执行程序的指令。

Xv6 使用页表（由硬件实现）为每个进程提供其自己的地址空间。RISC-V 页表将虚拟地址（RISC-V 指令操作的地址）转换为物理地址（CPU 芯片发送到主存的地址）。

Xv6 为每个进程维护一个单独的页表，该页表定义了该进程的地址空间。如图 2.3 所示，地址空间包括从虚拟地址零开始的进程用户内存。首先是指令，接着是全局变量，然后是栈，最后是进程可以根据需要扩展的“堆”区域（用于 `malloc`）。有许多因素限制了进程地址空间的最大大小：RISC-V 上的指针是 64 位宽；硬件在页表中查找虚拟地址时只使用低 39 位；并且 xv6 只使用这 39 位中的 38 位。因此，最大地址是 \(2^{38}-1 = 0x3fffffffff\)，这是

xv6 内核为每个进程维护许多状态信息，这些信息被收集到一个 `struct proc` (kernel/proc.h:86) 中。一个进程最重要的内核状态是其页表、内核栈和运行状态。我们将使用符号 `p->xxx` 来指代 `proc` 结构的元素；例如，`p->pagetable` 是指向进程页表的指针。

每个进程都有一个执行线程（简称线程）来执行进程的指令。一个线程可以被挂起，稍后恢复。为了在进程之间透明地切换，内核挂起当前运行的线程并恢复另一个进程的线程。线程的大部分状态（局部变量、函数调用返回地址）存储在线程的栈上。每个进程有两个栈：用户栈和内核栈 (`p->kstack`)。当进程执行用户指令时，只有其用户栈在使用，而内核栈是空的。当进程进入内核时（由于系统调用或中断），内核代码在进程的内核栈上执行；当进程在内核中时，其用户栈仍然保存着数据，但未被主动使用。进程的线程在使用其用户栈和内核栈之间交替。内核栈是独立的（并且受用户代码保护），这样即使进程破坏了其用户栈，内核也能执行。

一个进程可以通过执行 RISC-V `ecall` 指令来进行系统调用。这条指令提高硬件特权级别并将程序计数器更改为内核定义的入口点。入口点的代码切换到内核栈并执行实现系统调用的内核指令。当系统调用完成时，内核切换回用户栈并通过调用 `sret` 指令返回到用户空间，该指令降低硬件特权级别并在系统调用指令之后立即恢复执行用户指令。进程的线程可以在内核中“阻塞”以等待 I/O，并在 I/O 完成时从停止的地方恢复执行。

`p->state` 指示进程是已分配、就绪、运行、等待 I/O 还是退出。

`p->pagetable` 保存进程的页表，其格式符合 RISC-V 硬件的预期。Xv6 在用户空间中执行该进程时，会使分页硬件使用该进程的 `p->pagetable`。进程的页表也充当记录分配给进程存储其内存的物理页地址的账簿。

总之，一个进程捆绑了两个设计思想：一个地址空间，给进程拥有自己内存的错觉；以及一个线程，给进程拥有自己 CPU 的错觉。在 xv6 中，一个进程由一个地址空间和一个线程组成。在真正的操作系统中，一个进程可能有多个线程以利用多个 CPU。

### 2.6 代码：启动 xv6，第一个进程与系统调用

为了使 xv6 更具体，我们将概述内核如何启动并运行第一个进程。后续章节将更详细地描述本概述中出现的机制。

===== 第 28 页 =====

当 RISC-V 计算机通电时，它会进行自我初始化并运行存储在只读存储器中的引导加载程序。引导加载程序将 xv6 内核加载到内存中。然后，在机器模式下，CPU 从 `_entry` (kernel/entry.S:7) 开始执行 xv6。RISC-V 启动时分页硬件是禁用的：虚拟地址直接映射到物理地址。

加载程序将 xv6 内核加载到物理地址 `0x80000000` 处的内存中。它将内核放在 `0x80000000` 而不是 `0x0` 的原因是地址范围 `0x0 : 0x80000000` 包含 I/O 设备。

`_entry` 处的指令设置了一个栈，以便 xv6 可以运行 C 代码。Xv6 在文件 `start.c` (kernel/start.c:11) 中为初始栈 `stack0` 声明了空间。`_entry` 处的代码将栈指针寄存器 `sp` 加载为地址 `stack0+4096`，即栈顶，因为 RISC-V 上的栈是向下增长的。现在内核有了栈，`_entry` 调用到 `start` (kernel/start.c:21) 处的 C 代码。

函数 `start` 执行一些仅在机器模式下允许的配置，然后切换到监督模式。为了进入监督模式，RISC-V 提供了 `mret` 指令。该指令最常用于从先前的从监督模式到机器模式的调用返回。`start` 并非从这样的调用返回，而是设置状态，就好像有过一次调用一样：它在寄存器 `mstatus` 中将先前的特权模式设置为监督模式，通过将 `main` 的地址写入寄存器 `mepc` 来设置返回地址为 `main`，通过向页表寄存器 `satp` 写入 0 来禁用监督模式下的虚拟地址转换，并将所有中断和异常委托给监督模式。

在跳转到监督模式之前，`start` 还执行一项任务：它对时钟芯片进行编程以产生时钟中断。完成这些准备工作后，`start` 通过调用 `mret` “返回”到监督模式。这导致程序计数器更改为 `main` (kernel/main.c:11)。

在 `main` (kernel/main.c:11) 初始化了几个设备和子系统之后，它通过调用 `userinit` (kernel/proc.c:226) 创建第一个进程。第一个进程执行一个用 RISC-V 汇编编写的小程序，`initcode.S` (user/initcode.S:3) 将 `exec` 系统调用的编号 `SYS_EXEC` (kernel/syscall.h:8) 加载到寄存器 `a7` 中，然后调用 `ecall` 重新进入内核。

内核使用寄存器 `a7` 中的编号在 `syscall` (kernel/syscall.c:133) 中调用所需的系统调用。系统调用表 (kernel/syscall.c:108) 将 `SYS_EXEC` 映射到 `sys_exec`，内核会调用它。正如我们在第 1 章中看到的，`exec` 用新程序（这里是 `/init`）替换当前进程的内存和寄存器。

一旦内核完成 `exec`，它返回到 `/init` 进程的用户空间。`Init` (user/init.c:15) 在需要时创建一个新的控制台设备文件，然后将其作为文件描述符 0、1 和 2 打开。然后它在控制台上启动一个 shell。系统已启动。

### 2.7 安全模型

您可能想知道操作系统如何处理有缺陷或恶意的代码。由于应对恶意行为比处理意外错误严格得多，因此将此主题视为与安全相关是合理的。以下是操作系统设计中典型安全假设和目标的高级视图。

===== 第 29 页 =====

操作系统必须假设进程的用户级代码会尽力破坏内核或其他进程。用户代码可能试图解引用其允许地址空间之外的指针；它可能试图执行任何 RISC-V 指令，即使那些不是为用户代码设计的；它可能试图读取和写入任何 RISC-V 控制寄存器；它可能试图直接访问设备硬件；它可能向系统调用传递巧妙的值，试图诱骗内核崩溃或做傻事。内核的目标是限制每个用户进程，使其所能做的只是读/写/执行自己的用户内存，使用 32 个通用 RISC-V 寄存器，并以系统调用预期允许的方式影响内核和其他进程。内核必须阻止任何其他行为。这通常是内核设计中的绝对要求。

对内核自身代码的期望则大不相同。内核代码被假定是由善意且细心的程序员编写的。内核代码应该是没有缺陷的，当然也不包含任何恶意内容。这个假设影响了我们分析内核代码的方式。例如，有许多内部内核函数（例如自旋锁），如果内核代码错误地使用它们，会导致严重问题。在检查任何特定的内核代码片段时，我们想说服自己它的行为是正确的。然而，我们假设内核代码通常编写正确，并遵循所有关于使用内核自身函数和数据结构的规则。在硬件层面，假设 RISC-V CPU、RAM、磁盘等按文档所述运行，没有硬件缺陷。

当然，在现实生活中事情并非如此简单。很难防止精明的用户代码通过消耗受内核保护的资源（磁盘空间、CPU 时间、进程表槽位等）来使系统无法使用（或导致其崩溃）。编写无缺陷的代码或设计无缺陷的硬件通常是不可能的；如果恶意用户代码的编写者意识到内核或硬件缺陷，他们会利用这些缺陷。在内核中设计针对自身可能存在缺陷的防护措施是值得的：断言、类型检查、栈保护页等。最后，用户代码和内核代码之间的区别有时是模糊的：某些特权的用户级进程可能提供基本服务并有效地成为操作系统的一部分，并且在某些操作系统中，特权的用户代码可以将新代码插入内核（例如 Linux 的可加载内核模块）。

### 2.8 现实世界

大多数操作系统都采用了进程的概念，并且大多数进程看起来与 xv6 的进程相似。然而，现代操作系统支持一个进程内的多个线程，以允许单个进程利用多个 CPU。在一个进程中支持多个线程涉及许多 xv6 没有的机制，包括潜在的接口更改（例如 Linux 的 `clone`，`fork` 的一个变体），以控制线程共享进程的哪些方面。

# 3 页表

页表是最流行的机制，操作系统通过它为每个进程提供自己的私有地址空间和内存。页表决定了内存地址的含义，以及物理内存的哪些部分可以访问。它们允许 xv6 隔离不同进程的地址空间，并将它们多路复用到单个物理内存上。页表是一种流行的设计，因为它们提供了一个间接级别，允许操作系统执行许多技巧。xv6 执行了一些技巧：在多个地址空间中映射相同的内存（一个跳板页面），以及用未映射的页面保护内核栈和用户栈。本章的其余部分解释了 RISC-V 硬件提供的页表以及 xv6 如何使用它们。

### 3.1 分页硬件

提醒一下，RISC-V 指令（用户和内核）操作的是虚拟地址。机器的 RAM 或物理内存，使用物理地址进行索引。RISC-V 页表硬件通过将每个虚拟地址映射到一个物理地址来连接这两种地址。

xv6 在 Sv39 RISC-V 上运行，这意味着 64 位虚拟地址中只有低 39 位被使用；高 25 位不被使用。在这种 Sv39 配置中，一个 RISC-V 页表在逻辑上是一个有 \(2^{27}\)（134,217,728）个页表条目（PTE）的数组。每个 PTE 包含一个 44 位的物理页号（PPN）和一些标志位。分页硬件通过使用 39 位中的高 27 位在页表中索引以找到一个 PTE，并制作一个 56 位的物理地址，其高 44 位来自 PTE 中的 PPN，其低 12 位从原始虚拟地址复制而来。图 3.1 使用将页表视为简单 PTE 数组的逻辑视图展示了这个过程（更完整的情况见图 3.2）。页表使操作系统能够以 4096 \((2^{12})\) 字节的对齐块（称为页面）的粒度控制虚拟地址到物理地址的转换。

在 Sv39 RISC-V 中，虚拟地址的高 25 位不用于转换。物理地址也有增长空间：PTE 格式中为物理页号留出了再增加 10 位的空间。RISC-V 的设计者根据技术预测选择了这些数字。\(2^{39}\) 字节是 512 GB，这对于运行在 RISC-V 计算机上的应用程序来说应该足够了。

每个 PTE 包含标志位，告诉分页硬件如何使用关联的虚拟地址。`PTE_V` 指示 PTE 是否存在：如果未设置，则对该页的引用会导致异常（即不允许）。`PTE_R` 控制是否允许指令读取该页。`PTE_W` 控制是否允许指令写入该页。`PTE_X` 控制 CPU 是否可以将该页的内容解释为指令并执行它们。`PTE_U` 控制用户模式下的指令是否允许访问该页；如果未设置 `PTE_U`，则该 PTE 只能在监督模式下使用。图 3.2 展示了这一切是如何工作的。标志位和所有其他页面硬件相关的结构在 (kernel/riscv.h) 中定义。

为了告诉硬件使用页表，内核必须将根页表页的物理地址写入 `satp` 寄存器。每个 CPU 都有自己的 `satp`。一个 CPU 将使用其自己的 `satp` 指向的页表来转换后续指令生成的所有地址。每个 CPU 都有自己的 `satp`，以便不同的 CPU 可以运行不同的进程，每个进程都有由其自己的页表描述的私有地址空间。

通常，内核会将所有物理内存映射到其页表中，以便它可以使用加载/存储指令读写物理内存中的任何位置。由于页目录位于物理内存中，内核可以通过使用标准存储指令将 PTE 的内容写入页目录中 PTE 的虚拟地址来对其进行编程。

### 3.2 内核地址空间

Xv6 为每个进程维护一个页表，描述每个进程的用户地址空间，以及一个描述内核地址空间的单独页表。内核配置其地址空间的布局，以便在可预测的虚拟地址处访问物理内存和各种硬件资源。图 3.3 显示了此布局如何将内核虚拟地址映射到物理地址。文件 (kernel/memlayout.h) 声明了 xv6 内核内存布局的常量。

QEMU 模拟了一台计算机，该计算机包含从物理地址 `0x80000000` 开始、至少持续到 `0x86400000` 的 RAM（物理内存），xv6 称之为 `PHYSTOP`。QEMU 模拟还包括诸如磁盘接口之类的 I/O 设备。QEMU 将设备接口以内存映射控制寄存器的形式暴露给软件，这些寄存器位于物理地址空间的 `0x80000000` 以下。内核可以通过读/写这些特殊的物理地址来与设备交互；这样的读和写与设备硬件通信，而不是与 RAM 通信。第 4 章解释了 xv6 如何与设备交互。

内核使用“直接映射”来访问 RAM 和内存映射的设备寄存器；也就是说，将资源映射到等于物理地址的虚拟地址。例如，内核本身在内核虚拟地址空间和物理内存中都位于 `KERNBASE = 0x80000000`。直接映射简化了读写物理内存的内核代码。例如，当 `fork` 为子进程分配用户内存时，分配器返回该内存的物理地址；`fork` 在将父进程的用户内存复制到子进程时，直接将该地址用作虚拟地址。

有几个内核虚拟地址不是直接映射的：

- **跳板页**。它映射在虚拟地址空间的顶部；用户页表也有这个相同的映射。第 4 章讨论了跳板页的作用，但我们在这里看到了页表的一个有趣用例：一个物理页（保存跳板代码）在内核的虚拟地址空间中被映射两次：一次在虚拟地址空间的顶部，一次通过直接映射。
- **内核栈页**。每个进程都有其自己的内核栈，它被映射到高位，以便在其下方 xv6 可以留下一个未映射的保护页。保护页的 PTE 是无效的（即未设置 `PTE_V`），因此如果内核溢出内核栈，它很可能会导致异常并且内核会崩溃。没有保护页，溢出的栈将覆盖其他内核内存，导致错误操作。崩溃更可取。

虽然内核通过高位内存映射使用其栈，但内核也可以通过直接映射地址访问它们。另一种设计可能只有直接映射，并在直接映射地址处使用栈。然而，在这种安排中，提供保护页将涉及取消映射否则会指向物理内存的虚拟地址，这将导致难以使用。

内核将跳板页和内核文本的页面映射为具有 `PTE_R` 和 `PTE_X` 权限。内核从这些页面读取并执行指令。内核将其他页面映射为具有 `PTE_R` 和 `PTE_W` 权限，以便它可以读取和写入这些页面中的内存。保护页的映射是无效的。

### 3.3 代码：创建地址空间

大多数用于操作地址空间和页表的 xv6 代码驻留在 `vm.c` (kernel/vm.c:1) 中。中心数据结构是 `pagetable_t`，它实际上是一个指向 RISC-V

在启动序列的早期，`main` 调用 `kvminit` (kernel/vm.c:54) 来使用 `kvmmake` (kernel/vm.c:20) 创建内核页表。此调用发生在 xv6 在 RISC-V 上启用分页之前，因此地址直接指向物理内存。`Kvmmake` 首先分配一个物理内存页来保存根页表页。然后它调用 `kvmmap` 来安装内核所需的转换。这些转换包括内核的指令和数据、直到 `PHYSTOP` 的物理内存以及实际上是设备的内存范围。`Proc_mapstacks` (kernel/proc.c:33) 为每个进程分配一个内核栈。它调用 `kvmmap` 将每个栈映射到由 `KSTACK` 生成的虚拟地址，这为无效的栈保护页留出了空间。

`kvmmap` (kernel/vm.c:127) 调用 `mappages` (kernel/vm.c:138)，后者将一定范围的虚拟地址映射到相应范围的物理地址，将映射安装到页表中。它对范围内的每个虚拟地址单独执行此操作，按页间隔。对于每个要映射的虚拟地址，`mappages` 调用 `walk` 以找到该地址的 PTE 的地址。然后它初始化 PTE 以保存相关的物理页号、所需的权限（`PTE_W`、`PTE_X` 和/或 `PTE_R`）以及 `PTE_V` 以将 PTE 标记为有效 (kernel/vm.c:153)。

`walk` (kernel/vm.c:81) 在查找虚拟地址的 PTE 时模仿了 RISC-V 分页硬件（见图 3.2）。`walk` 每次 9 位地遍历三级页表。它使用虚拟地址的每一级的 9 位来找到下一级页表或最终页面的 PTE (kernel/vm.c:87)。如果 PTE 无效，则所需的页面尚未分配；如果 `alloc` 参数被设置，`walk` 分配一个新的页表页并将其物理地址放入 PTE。它返回树中最底层的 PTE 的地址 (kernel/vm.c:97)。

上述代码依赖于物理内存被直接映射到内核虚拟地址空间。例如，当 `walk` 向下遍历页表的层级时，它从一个 PTE 中取出下一级页表的（物理）地址 (kernel/vm.c:89)，然后使用该地址作为虚拟地址来获取下一级的 PTE (kernel/vm.c:87)。

`main` 调用 `kvminithart` (kernel/vm.c:62) 来安装内核页表。它将根页表页的物理地址写入寄存器 `satp`。此后，CPU 将使用内核页表转换地址。由于内核使用恒等映射，下一条指令的（现在为）虚拟地址将映射到正确的物理内存地址。

每个 RISC-V CPU 将页表条目缓存在转译后备缓冲区（TLB）中，当 xv6 更改页表时，它必须告诉 CPU 使相应的缓存 TLB 条目无效。如果不这样做，那么稍后的某个时候 TLB 可能会使用一个旧的缓存映射，指向一个在此期间已分配给另一个进程的物理页，结果，一个进程可能能够涂鸦另一个进程的内存。RISC-V 有一条指令 `sfence.vma`，用于刷新当前 CPU 的 TLB。Xv6 在重新加载 `satp` 寄存器后在 `kvminithart` 中执行 `sfence.vma`，并在切换回用户空间之前切换到用户页表的跳板代码中执行 `sfence.vma` (kernel/trampoline.S:79)。

### 3.4 物理内存分配

内核必须在运行时为页表、用户内存、内核栈和管道缓冲区分配和释放物理内存。

xv6 使用内核结束之后和 `PHYSTOP` 之间的物理内存进行运行时分配。它一次分配和释放整个 4096 字节的页面。它通过将空闲页面本身串成一个链表来跟踪哪些页面是空闲的。分配包括从链表中移除一个页面；释放包括将释放的页面添加到链表中。

### 3.5 代码：物理内存分配器

分配器位于 `kalloc.c` (kernel/kalloc.c:1) 中。分配器的数据结构是一个空闲物理内存页的链表，可供分配。每个空闲页的链表元素是一个 `struct run` (kernel/kalloc.c:17)。分配器从哪里获取内存来保存该数据结构？它将每个空闲页的 `run` 结构存储在该空闲页本身中，因为那里没有存储其他东西。空闲列表由一个自旋锁保护 (kernel/kalloc.c:21-24)。列表和锁被包装在一个结构体中，以明确锁保护结构体中的字段。现在，忽略锁以及对 `acquire` 和 `release` 的调用；第 6 章将详细讨论锁。

函数 `main` 调用 `kinit` 来初始化分配器 (kernel/kalloc.c:27)。`kinit` 初始化空闲列表，使其包含内核结束之后和 `PHYSTOP` 之间的每个页面。xv6 应该通过解析硬件提供的配置信息来确定可用的物理内存量。相反，xv6 假设机器有 128 兆字节的 RAM。`kinit` 调用 `freerange` 通过每页调用 `kfree` 将内存添加到空闲列表。一个 PTE 只能引用在 4096 字节边界上对齐（是 4096 的倍数）的物理地址，因此 `freerange` 使用 `PGROUNDUP` 来确保它只释放对齐的物理地址。分配器开始时没有内存；这些对 `kfree` 的调用给它一些内存来管理。

分配器有时将地址视为整数以对其执行算术运算（例如，在 `freerange` 中遍历所有页面），有时将地址用作指针来读写内存（例如，操作存储在每个页面中的 `run` 结构）；地址的这种双重用途是分配器代码充满 C 类型转换的主要原因。另一个原因是释放和分配本质上会改变内存的类型。

函数 `kfree` (kernel/kalloc.c:47) 首先将正在释放的内存中的每个字节设置为值 1。这将导致在释放后使用内存的代码（使用“悬空引用”）读取垃圾而不是旧的有效内容；希望这将导致此类代码更快地出错。然后 `kfree` 将页面放到空闲列表的前面：它将 `pa` 转换为指向 `struct run` 的指针，将空闲列表的旧起始地址记录在 `r->next` 中，并将空闲列表设置为 `r`。`kalloc` 移除并返回空闲列表中的第一个元素。

### 3.6 进程地址空间

每个进程都有一个独立的页表，当 xv6 在进程之间切换时，它也会更改页表。如图 2.3 所示，进程的用户内存从虚拟地址零开始，可以增长到 `MAXVA` (kernel/riscv.h:360)，原则上允许进程寻址 256 GB 的内存。

当进程向 xv6 请求更多用户内存时，xv6 首先使用 `kalloc` 分配物理页。然后它将 PTE 添加到进程的页表中，指向新的物理页。Xv6 在这些 PTE 中设置 `PTE_W`、`PTE_X`、`PTE_R`、`PTE_U` 和 `PTE_V` 标志。大多数进程不使用整个用户地址空间；xv6 在未使用的 PTE 中使 `PTE_V` 保持清除状态。

我们在这里看到了使用页表的一些很好的例子。首先，不同进程的页表将用户地址转换为不同的物理内存页，因此每个进程都有私有的用户内存。其次，每个进程将其内存视为从零开始的连续虚拟地址，而进程的物理内存可以是不连续的。第三，内核在用户地址空间的顶部映射了一个带有跳板代码的页面，因此单个物理内存页出现在所有地址空间中。

### 3.7 代码：sbrk

`Sbrk` 是进程缩减或增长其内存的系统调用。该系统调用由函数 `growproc` (kernel/proc.c:253) 实现。根据 `n` 是正数还是负数，`growproc` 调用 `uvmalloc` 或 `uvmdealloc`。`uvmalloc` (kernel/vm.c:221) 使用 `kalloc` 分配物理内存，并使用 `mappages` 将 PTE 添加到用户页表。`uvmdealloc` 调用 `uvmunmap` (kernel/vm.c:166)，后者使用 `walk` 查找 PTE 并使用 `kfree` 释放它们引用的物理内存。

### 3.8 代码：exec

`Exec` 是创建地址空间的用户部分的系统调用。它从文件系统中存储的文件初始化地址空间的用户部分。`Exec` (kernel/exec.c:13) 使用 `namei` (kernel/exec.c:26) 打开命名的二进制路径，这将在第 8 章中解释。然后，它读取 ELF 头。Xv6 应用程序使用广泛使用的 ELF 格式描述，定义在 (kernel/elf.h) 中。一个 ELF 二进制文件由一个 ELF 头 `struct elfhdr` (kernel/elf.h:6) 组成，后跟一系列程序段头 `struct proghdr` (kernel/elf.h:25)。每个 `proghdr` 描述应用程序的一个必须加载到内存中的段；xv6 程序只有一个程序段头，但其他系统可能将指令和数据分开在不同的段中。

第一步是快速检查文件可能包含一个 ELF 二进制文件。一个 ELF 二进制文件以四字节的“魔数” \(0\times 7\mathrm{F}\)、'E'、'L'、'F' 或 `ELF_MAGIC` (kernel/elf.h:3) 开头。如果 ELF 头具有正确的魔数，`exec` 假定二进制文件是格式良好的。

`Exec` 使用 `proc_pagetable` (kernel/exec.c:38) 分配一个没有用户映射的新页表，使用 `uvmalloc` (kernel/exec.c:52) 为每个 ELF 段分配内存，并使用 `loadseg` (kernel/exec.c:10) 将每个段加载到内存中。`loadseg` 使用 `walkaddr` 找到分配的内存的物理地址，用于写入 ELF 段的每一页，并使用 `readi` 从文件中读取。

`Exec` 在栈页正下方放置一个不可访问的页，以便尝试使用超过一页的程序将引发缺页故障。这个不可访问的页也允许 `exec` 处理过大的参数；在这种情况下，`exec` 用于将参数复制到栈的 `copyout` (kernel/vm.c:347) 函数将注意到目标页面不可访问，并将返回 -1。

在准备新内存映像期间，如果 `exec` 检测到错误（如无效的程序段），它会跳转到标签 `bad`，释放新映像，并返回 -1。`Exec` 必须等到确定系统调用将成功后才能释放旧映像：如果旧映像消失了，系统调用就无法向它返回 -1。`exec` 中的错误情况只发生在映像创建期间。一旦映像完成，`exec` 可以提交新的页表 (kernel/exec.c:113) 并释放旧的页表 (kernel/exec.c:117)。

`Exec` 从 ELF 文件加载字节到 ELF 文件指定的地址。用户或进程可以将他们想要的任何地址放入 ELF 文件。因此 `exec` 是危险的，因为 ELF 文件中的地址可能意外地或有意地指向内核。对于不小心的内核来说，后果可能是崩溃，也可能是对内核隔离机制的恶意破坏（即安全漏洞）。Xv6 执行了许多检查来避免这些风险。例如，`if(ph.vaddr + ph.memsz < ph.vaddr)` 检查总和是否溢出 64 位整数。危险在于，用户可以构建一个 ELF 二进制文件，其 `ph.vaddr` 指向一个用户选择的地址，并且 `ph.memsz` 足够大，以至于总和溢出到 `0x1000`，这看起来像一个有效值。在 xv6 的旧版本中，用户地址空间也包含内核（但在用户模式下不可读/写），用户可以选择一个对应于内核内存的地址，从而将数据从 ELF 二进制文件复制到内核中。在 xv6 的 RISC-V 版本中，这不可能发生，因为内核有自己的独立页表；`loadseg` 加载到进程的页表中，而不是内核的页表。

内核开发者很容易忽略一个关键检查，而现实世界的内核有很长的历史，其中缺少的检查可能被用户程序利用来获取内核特权。xv6 很可能没有完全完成对提供给内核的用户级数据的验证，恶意用户程序可能能够利用这一点来绕过 xv6 的隔离。

### 3.9 现实世界

像大多数操作系统一样，xv6 使用分页硬件进行内存保护和映射。大多数操作系统通过结合分页和缺页异常（我们将在第 4 章讨论）来使用比 xv6 更复杂的分页技术。

Xv6 因内核使用虚拟地址和物理地址之间的直接映射，以及假设物理 RAM 位于地址 `0x8000000`（内核期望加载的位置）而得到简化。这在 QEMU 中有效，但在真实硬件上结果证明这是一个坏主意；真实硬件将 RAM 和设备放在不可预测的物理地址，因此（例如）在 `0x8000000` 可能没有 RAM，而 xv6 期望在那里存储内核。更严格的内核设计利用页表将任意的硬件物理内存布局转换为可预测的内核虚拟地址布局。

RISC-V 支持物理地址级别的保护，但 xv6 不使用该特性。

在内存很多的机器上，使用 RISC-V 对“超大页面”的支持可能是有意义的。当物理内存较小时，小页面有意义，可以以细粒度进行分配和换出到磁盘。例如，如果一个程序只使用 8 KB 内存，给它一个完整的 4 MB 超大物理内存页是浪费的。在 RAM 很多的机器上，较大的页面更有意义，并且可以减少页表操作的开销。

xv6 内核缺乏能为小对象提供内存的类似 `malloc` 的分配器，这阻止了内核使用需要动态分配的复杂数据结构。

内存分配是一个长期的热门话题，基本问题是如何有效利用有限的内存以及为未知的未来请求做准备 [8]。如今人们更关心速度而不是空间效率。此外，更复杂的内核可能会分配许多不同大小的小块，而不是（像 xv6 那样）只分配 4096 字节的块；真正的内核分配器需要处理小型和大型分配。

# 4 陷阱与系统调用

有三种事件会导致 CPU 放弃正常的指令执行，并强制将控制权转移给处理该事件的特殊代码。一种情况是系统调用，当用户程序执行 `ecall` 指令请求内核为其做某事时发生。另一种情况是异常：指令（用户或内核）执行了非法操作，例如除以零或使用了无效的虚拟地址。第三种情况是设备中断，当设备发出需要注意的信号时发生，例如当磁盘硬件完成读或写请求时。

本书使用术语**陷阱**来泛指这些情况。通常，无论陷阱发生时正在执行什么代码，稍后都需要恢复，并且不应该意识到发生了什么特殊的事情。也就是说，我们通常希望陷阱是透明的；这对于设备中断尤其重要，被中断的代码通常不会期望发生中断。通常的序列是：陷阱强制将控制权转移到内核；内核保存寄存器和其他状态，以便可以恢复执行；内核执行适当的处理程序代码（例如，系统调用实现或设备驱动程序）；内核恢复保存的状态并从陷阱返回；原始代码从中断的地方继续执行。

Xv6 在内核中处理所有陷阱；陷阱不会传递给用户代码。在内核中处理陷阱对于系统调用来说是很自然的。这对于中断也是有意义的，因为隔离要求只有内核被允许使用设备，并且因为内核是一个方便的机制，可以在多个进程之间共享设备。对于异常来说也是有意义的，因为 xv6 通过杀死违规程序来响应用户空间的所有异常。

xv6 陷阱处理分四个阶段：RISC-V CPU 采取的硬件操作、一些为内核 C 代码做准备的汇编指令、一个决定如何处理陷阱的 C 函数，以及系统调用或设备驱动程序服务例程。虽然三种陷阱类型的共性表明内核可以用一个单一的代码路径处理所有陷阱，但事实证明，为三种不同的情况（来自用户空间的陷阱、来自内核空间的陷阱和时钟中断）使用单独的代码很方便。处理陷阱的内核代码（汇编或 C）通常称为**处理程序**；第一个处理程序指令通常用汇编（而不是 C）编写，有时称为**向量**。

### 4.1 RISC-V 陷阱机制

每个 RISC-V CPU 都有一组控制寄存器，内核写入这些寄存器以告诉 CPU 如何处理陷阱，并且内核可以读取这些寄存器以了解发生的陷阱。RISC-V 文档包含完整说明 [1]。`riscv.h` (kernel/riscv.h:1) 包含 xv6 使用的定义。以下是最重要寄存器的概述：

- `stvec`：内核在此处写入其陷阱处理程序的地址；RISC-V 跳转到 `stvec` 中的地址来处理陷阱。
- `sepc`：当陷阱发生时，RISC-V 将程序计数器保存于此（因为 `pc` 随后被 `stvec` 中的值覆盖）。`sret`（从陷阱返回）指令将 `sepc` 复制到 `pc`。内核可以写入 `sepc` 以控制 `sret` 的去向。
- `scause`：RISC-V 在此处放入一个数字，描述陷阱的原因。
- `sscratch`：内核在此处放置一个值，该值在陷阱处理程序的最开始会很方便。
- `sstatus`：`sstatus` 中的 `SIE` 位控制是否启用设备中断。如果内核清除 `SIE`，RISC-V 将推迟设备中断，直到内核设置 `SIE`。`SPP` 位指示陷阱来自用户模式还是监督模式，并控制 `sret` 返回到哪种模式。

上述寄存器与在监督模式下处理的陷阱有关，它们不能在用户模式下读取或写入。有一组类似的用于在机器模式下处理的陷阱的控制寄存器；xv6 仅在时钟中断的特殊情况下使用它们。

多核芯片上的每个 CPU 都有一套自己的这些寄存器，并且任何给定时间可能有多个 CPU 在处理陷阱。

当需要强制产生陷阱时，RISC-V 硬件对所有陷阱类型（时钟中断除外）执行以下操作：

1. 如果陷阱是设备中断，并且 `sstatus` 的 `SIE` 位被清除，则不执行以下任何操作。
2. 通过清除 `sstatus` 中的 `SIE` 位来禁用中断。
3. 将 `pc` 复制到 `sepc`。
4. 将当前模式（用户或监督）保存在 `sstatus` 的 `SPP` 位中。
5. 设置 `scause` 以反映陷阱的原因。
6. 将模式设置为监督模式。
7. 将 `stvec` 复制到 `pc`。

===== 第 45 页 =====

8. 开始在新的 `pc` 处执行。

请注意，CPU 不会切换到内核页表，不会切换到内核中的栈，并且除了 `pc` 之外不保存任何寄存器。内核软件必须执行这些任务。CPU 在陷阱期间只做最少工作的一个原因是为软件提供灵活性；例如，某些操作系统在某些情况下会省略页表切换以提高陷阱性能。

值得思考的是，上述步骤中是否有任何可以省略的，也许是为了寻求更快的陷阱。尽管在某些情况下更简单的序列可以工作，但通常省略许多步骤是危险的。例如，假设 CPU 没有切换程序计数器。那么来自用户空间的陷阱可以在仍然运行用户指令的同时切换到监督模式。那些用户指令可能会破坏用户/内核隔离，例如通过修改 `satp` 寄存器以指向允许访问所有物理内存的页表。因此，CPU 切换到内核指定的指令地址（即 `stvec`）非常重要。

### 4.2 来自用户空间的陷阱

Xv6 根据陷阱是在内核中还是在用户代码中执行，以不同的方式处理陷阱。以下是在用户代码中发生陷阱的情况；第 4.5 节描述了在内核代码中发生陷阱的情况。

如果用户程序进行系统调用（`ecall` 指令）、做了非法操作，或者设备中断，则可能在用户空间执行期间发生陷阱。来自用户空间的陷阱的高级路径是：`uservec` (kernel/trampoline.S:16)，然后是 `usertrap` (kernel/trap.c:37)；返回时，`usertrapret` (kernel/trap.c:90) 然后 `userret` (kernel/trampoline.S:88)。

xv6 陷阱处理设计的一个主要约束是，RISC-V 硬件在强制陷阱时不会切换页表。这意味着 `stvec` 中的陷阱处理程序地址必须在用户页表中具有有效的映射，因为陷阱处理代码开始执行时该页表是生效的。此外，xv6 的陷阱处理代码需要切换到内核页表；为了能够在切换后继续执行，内核页表也必须具有 `stvec` 指向的处理程序的映射。

Xv6 使用**跳板页**来满足这些要求。跳板页包含 `uservec`，即 `stvec` 指向的 xv6 陷阱处理代码。跳板页在每个进程的页表中都映射在地址 `TRAMPOLINE` 处，该地址位于虚拟地址空间的末尾，以便它高于程序为自己使用的内存。跳板页也在内核页表中的地址 `TRAMPOLINE` 处映射。见图 2.3 和图 3.3。因为跳板页映射在用户页表中，并带有 `PTE_U` 标志，陷阱可以在监督模式下开始在那里执行。因为跳板页在内核地址空间的相同地址处映射，陷阱处理程序可以在切换到内核页表后继续执行。

`uservec` 陷阱处理程序的代码位于 `trampoline.S` (kernel/trampoline.S:16) 中。当 `uservec` 开始时，所有 32 个寄存器都包含被中断的用户代码拥有的值。这 32 个值需要保存在内存中的某个地方，以便在陷阱返回到用户空间时可以恢复它们。存储到内存需要使用一个寄存器来保存地址，但在这一点上没有可用的通用寄存器！幸运的是，RISC-V 以 `sscratch` 寄存器的形式提供了帮助。

`uservec` 开头的 `csrrw` 指令交换 `a0` 和 `sscratch` 的内容。现在用户代码的 `a0` 保存在 `sscratch` 中；`uservec` 有一个寄存器（`a0`）可以玩；并且 `a0` 包含内核先前放在 `sscratch` 中的值。

`uservec` 的下一个任务是保存 32 个用户寄存器。在进入用户空间之前，内核将 `sscratch` 设置为指向一个每进程的**陷阱帧**结构，该结构（除其他外）有空间保存 32 个用户寄存器 (kernel/proc.h:44)。因为 `satp` 仍然指向用户页表，`uservec` 需要陷阱帧映射在用户地址空间中。在创建每个进程时，xv6 为进程的陷阱帧分配一个页面，并安排它始终映射在用户虚拟地址 `TRAPFRAME` 处，该地址略低于 `TRAMPOLINE`。进程的 `p->trapframe` 也指向陷阱帧，但是在其物理地址处，以便内核可以通过内核页表使用它。

因此，在交换 `a0` 和 `sscratch` 之后，`a0` 保存一个指向当前进程陷阱帧的指针。`uservec` 现在将所有用户寄存器保存在那里，包括从 `sscratch` 读取的用户 `a0`。

陷阱帧包含当前进程的内核栈地址、当前 CPU 的 `hartid`、`usertrap` 函数的地址以及内核页表的地址。`uservec` 检索这些值，将 `satp` 切换到内核页表，并调用 `usertrap`。

`usertrap` 的工作是确定陷阱的原因，处理它，然后返回 (kernel/trap.c:37)。它首先更改 `stvec`，以便内核中的陷阱将由 `kernelvec` 而不是 `uservec` 处理。它保存 `sepc` 寄存器（保存的用户程序计数器），因为 `usertrap` 可能调用 `yield` 切换到另一个进程的内核线程，而那个进程可能返回到用户空间，在此过程中它会修改 `sepc`。如果陷阱是系统调用，`usertrap` 调用 `syscall` 来处理它；如果是设备中断，则调用 `devintr`；否则是异常，内核会杀死违规进程。系统调用路径将保存的用户程序计数器加四，因为 RISC-V 在系统调用的情况下，将程序指针留在指向 `ecall` 指令的位置，但用户代码需要从后续指令开始恢复执行。在返回的路上，`usertrap` 检查进程是否已被杀死或应该让出 CPU（如果此陷阱是时钟中断）。

### 4.3 代码：调用系统调用

第 2 章以 `initcode.S` 调用 `exec` 系统调用结束 (user/initcode.S:11)。让我们看看用户调用是如何到达内核中 `exec` 系统调用的实现的。

`initcode.S` 将 `exec` 的参数放在寄存器 `a0` 和 `a1` 中，并将系统调用号放在 `a7` 中。系统调用号与 `syscalls` 数组中的条目匹配，`syscalls` 是一个函数指针表 (kernel/syscall.c:108)。`ecall` 指令陷入内核，并导致 `uservec`、`usertrap` 然后 `syscall` 执行，正如我们上面看到的。

`syscall` (kernel/syscall.c:133) 从陷阱帧中保存的 `a7` 中检索系统调用号，并使用它来索引 `syscalls`。对于第一个系统调用，`a7` 包含 `SYS_exec` (kernel/syscall.h:8)，导致调用系统调用实现函数 `sys_exec`。

当 `sys_exec` 返回时，`syscall` 将其返回值记录在 `p->trapframe->a0` 中。这将导致原始的用户空间调用 `exec()` 返回该值，因为 RISC-V 上的 C 调用约定将返回值放在 `a0` 中。系统调用通常返回负数表示错误，返回零或正数表示成功。如果系统调用号无效，`syscall` 打印错误并返回 -1。

### 4.4 代码：系统调用参数

内核中的系统调用实现需要找到用户代码传递的参数。因为用户代码调用系统调用包装函数，参数最初位于 RISC-V C 调用约定放置它们的位置：在寄存器中。内核陷阱代码将用户寄存器保存到当前进程的陷阱帧中，内核代码可以在那里找到它们。内核函数 `argint`、`argaddr` 和 `argfd` 从陷阱帧中以整数、指针或文件描述符的形式检索第 `n` 个系统调用参数。它们都调用 `argraw` 来检索适当的已保存用户寄存器 (kernel/syscall.c:35)。

一些系统调用传递指针作为参数，内核必须使用这些指针来读取或写入用户内存。例如，`exec` 系统调用向内核传递一个指向用户空间中字符串参数的指针数组。这些指针带来两个挑战。首先，用户程序可能有缺陷或恶意，可能向内核传递一个无效的指针或旨在诱骗内核访问内核内存而不是用户内存的指针。其次，xv6 内核页表映射与用户页表映射不同，因此内核不能使用普通指令从用户提供的地址加载或存储。

内核实现了一些函数，可以安全地在用户提供的地址之间传输数据。`fetchstr` 是一个例子 (kernel/syscall.c:25)。像 `exec` 这样的文件系统调用使用 `fetchstr` 从用户空间检索字符串文件名参数。`fetchstr` 调用 `copyinstr` 来完成艰难的工作。

`copyinstr` (kernel/vm.c:398) 将最多 `max` 字节从用户页表 `pagetable` 中的虚拟地址 `srcva` 复制到 `dst`。由于 `pagetable` 不是当前的页表，`copyinstr` 使用 `walkaddr`（它调用 `walk`）在 `pagetable` 中查找 `srcva`，得到物理地址

`pa0`。内核将每个物理 RAM 地址映射到相应的内核虚拟地址，因此 `copyinstr` 可以直接将字符串字节从 `pa0` 复制到 `dst`。`walkaddr` (kernel/vm.c:104) 检查用户提供的虚拟地址是否属于进程的用户地址空间，因此程序无法诱骗内核读取其他内存。类似地，函数 `copyout` 将数据从内核复制到用户提供的地址。

### 4.5 来自内核空间的陷阱

Xv6 根据执行的是用户代码还是内核代码，对 CPU 陷阱寄存器的配置略有不同。当内核在 CPU 上执行时，内核将 `stvec` 指向 `kernelvec` 处的汇编代码 (kernel/kernelvec.S:10)。由于 xv6 已经在内核中，`kernelvec` 可以依赖 `satp` 被设置为内核页表，并依赖栈指针指向一个有效的内核栈。`kernelvec` 将所有 32 个寄存器压入栈，稍后将从中恢复它们，以便被中断的内核代码可以不受干扰地恢复。

`kernelvec` 将寄存器保存在被中断的内核线程的栈上，这是有道理的，因为寄存器值属于该线程。如果陷阱导致切换到不同的线程，这一点尤其重要——在这种情况下，陷阱实际上将从新线程的栈返回，将被中断线程的保存寄存器安全地留在其栈上。

在保存寄存器之后，`kernelvec` 跳转到 `kerneltrap` (kernel/trap.c:134)。`kerneltrap` 为两种类型的陷阱做准备：设备中断和异常。它调用 `devintr` (kernel/trap.c:177) 来检查并处理前者。如果陷阱不是设备中断，那么它必定是异常，并且如果发生在 xv6 内核中，这总是一个致命错误；内核调用 `panic` 并停止执行。

如果 `kerneltrap` 是由于时钟中断而被调用的，并且正在运行一个进程的内核线程（而不是调度器线程），则 `kerneltrap` 调用 `yield` 让其他线程有机会运行。在某个时刻，这些线程中的一个将让出 CPU，并让我们的线程及其 `kerneltrap` 再次恢复。第 7 章解释了 `yield` 中发生的事情。

当 `kerneltrap` 的工作完成时，它需要返回到被陷阱中断的任何代码。因为 `yield` 可能会扰乱 `sepc` 和 `sstatus` 中的先前模式，`kerneltrap` 在开始时保存它们。现在它恢复这些控制寄存器并返回到 `kernelvec` (kernel/kernelvec.S:48)。`kernelvec` 从栈中弹出保存的寄存器并执行 `sret`，它将 `sepc` 复制到 `pc` 并恢复被中断的内核代码。

值得思考一下，如果 `kerneltrap` 由于时钟中断而调用了 `yield`，陷阱返回是如何发生的。

Xv6 当 CPU 从用户空间进入内核时，将该 CPU 的 `stvec` 设置为 `kernelvec`；您可以在 `usertrap` (kernel/trap.c:29) 中看到这一点。有一个时间窗口，内核已经开始执行但 `stvec` 仍然设置为 `uservec`，并且在这个窗口期间不发生设备中断至关重要。幸运的是，RISC-V 在开始处理陷阱时总是禁用中断，并且 xv6 在设置 `stvec` 之后才重新启用它们。

### 4.6 缺页异常

Xv6 对异常的反应相当无趣：如果异常发生在用户空间，内核会杀死违规进程。如果异常发生在内核中，内核会崩溃。真正的操作系统通常以更有趣的方式做出反应。

例如，许多内核使用缺页异常来实现**写时复制（COW）fork**。为了解释写时复制 fork，考虑第 3 章中描述的 xv6 的 `fork`。`fork` 使子进程的初始内存内容与 `fork` 发生时父进程的内存内容相同。xv6 使用 `uvmcopy` (kernel/vm.c:301) 实现 `fork`，它为子进程分配物理内存并将父进程的内存复制到其中。如果子进程和父进程可以共享父进程的物理内存，效率会更高。然而，直接实现这一点是行不通的，因为父进程和子进程会对共享的栈和堆的写入会干扰彼此的执行。

通过适当使用页表权限和缺页异常，父进程和子进程可以安全地共享物理内存。当使用的虚拟地址在页表中没有映射，或者映射的 `PTE_V` 标志被清除，或者映射的权限位（`PTE_R`、`PTE_W`、`PTE_X`、`PTE_U`）禁止正在尝试的操作时，CPU 会引发缺页异常。RISC-V 区分三种缺页异常：加载缺页（当加载指令无法转换其虚拟地址时）、存储缺页（当存储指令无法转换其虚拟地址时）和指令缺页（当程序计数器中的地址无法转换时）。`scause` 寄存器指示缺页异常的类型，`stval` 寄存器包含无法转换的地址。

COW fork 的基本计划是让父进程和子进程最初共享所有物理页，但每个都将其映射为只读（清除 `PTE_W` 标志）。父进程和子进程可以从共享的物理内存中读取。如果任何一个写入给定的页面，RISC-V CPU 会引发缺页异常。内核的陷阱处理程序通过分配一个新的物理内存页，并将故障地址映射到的物理页复制到其中来响应。内核更改故障进程页表中相关的 PTE，使其指向副本并允许读写，然后恢复故障进程在引发故障的指令处执行。因为 PTE 允许写入，重新执行的指令现在将顺利执行而不会引发故障。写时复制需要簿记来帮助确定何时可以释放物理页，因为每个物理页可能根据 fork、缺页异常、exec 和 exit 的历史被不同数量的页表引用。这种簿记允许一个重要的优化：如果一个进程发生存储缺页，并且该物理页仅从该进程的页表引用，则无需复制。

写时复制使 `fork` 更快，因为 `fork` 不需要复制内存。部分内存稍后会在写入时被复制，但通常情况下，大部分内存永远不需要复制。一个常见的例子是 `fork` 后跟 `exec`：`fork` 后可能只写入几个页面，但随后子进程的 `exec` 会释放从父进程继承的大部分内存。写时复制 fork 消除了复制这些内存的需要。此外，COW fork 是透明的：应用程序无需修改即可从中受益。

页表和缺页异常的组合除了 COW fork 之外，还开辟了广泛的有趣可能性。另一个广泛使用的特性称为**惰性分配**，它有两个部分。首先，当应用程序通过调用 `sbrk` 请求更多内存时，内核会记下

大小的增加，但不分配物理内存，也不为新的虚拟地址范围创建 PTE。其次，当在这些新地址之一上发生缺页异常时，内核分配一个物理内存页并将其映射到页表中。与 COW fork 一样，内核可以对应用程序透明地实现惰性分配。

由于应用程序通常请求的内存多于它们实际需要的，惰性分配是一个胜利：对于应用程序从未使用的页面，内核根本不需要做任何工作。此外，如果应用程序请求大幅增长地址空间，那么没有惰性分配的 `sbrk` 代价很高：如果应用程序请求 1 GB 内存，内核必须分配并清零 262,144 个 4096 字节的页面。惰性分配允许将这种成本分摊到一段时间内。另一方面，惰性分配会带来缺页异常的额外开销，这涉及内核/用户态转换。操作系统可以通过每次缺页异常分配一批连续的页面（而不是一个页面）以及为这种缺页异常专门化内核入口/出口代码来降低这种成本。

另一个利用缺页异常的广泛使用的特性是**按需分页**。在 `exec` 中，xv6 急切地将应用程序的所有文本和数据加载到内存中。由于应用程序可能很大，并且从磁盘读取成本高昂，因此这种启动成本可能会让用户明显感觉到：当用户从 shell 启动一个大型应用程序时，用户可能需要很长时间才能看到响应。为了提高响应时间，现代内核为用户地址空间创建页表，但将页面的 PTE 标记为无效。在缺页异常时，内核从磁盘读取页面的内容，并将其映射到用户地址空间。与 COW fork 和惰性分配一样，内核可以对应用程序透明地实现此特性。

在计算机上运行的程序可能需要的内存比计算机拥有的 RAM 多。为了优雅地应对这种情况，操作系统可以实现**分页到磁盘**。其思想是只将用户页面的一小部分存储在 RAM 中，其余的存储在磁盘上的**交换区**中。内核将对应于存储在交换区中（因此不在 RAM 中）的内存的 PTE 标记为无效。如果应用程序尝试使用已换出到磁盘的页面之一，应用程序将发生缺页异常，并且必须将该页面换入：内核陷阱处理程序将分配一页物理 RAM，将页面从磁盘读入 RAM，并修改相关的 PTE 以指向 RAM。

如果需要换入一个页面，但没有空闲的物理 RAM 怎么办？在这种情况下，内核必须首先通过将其换出或驱逐到磁盘上的交换区，并标记引用该物理页的 PTE 为无效，来释放一个物理页。驱逐开销很大，因此如果这种情况不频繁发生，分页性能最好：如果应用程序只使用其内存页的一个子集，并且这些子集的并集适合 RAM。此属性通常称为具有良好的**引用局部性**。与许多虚拟内存技术一样，内核通常以对应用程序透明的方式实现分页到磁盘。

无论硬件提供多少 RAM，计算机通常在空闲物理内存很少或没有的情况下运行。例如，云提供商为了经济高效地使用硬件，将许多客户多路复用到一台机器上。另一个例子是，用户在少量物理内存的智能手机上运行许多应用程序。在这种环境中分配一个页面可能首先需要驱逐一个现有页面。因此，当空闲物理内存稀缺时，分配成本很高。

当空闲内存稀缺时，惰性分配和按需分页尤其有利。在 `sbrk` 或 `exec` 中急切地分配内存会带来驱逐内存以使其可用的额外成本。此外，急切的工作有被浪费的风险，因为在应用程序

### 4.7 现实世界

跳板和陷阱帧可能看起来过于复杂。一个驱动因素是 RISC-V 在强制陷阱时故意尽可能少做事情，以允许非常快速的陷阱处理，事实证明这很重要。因此，内核陷阱处理程序的最初几条指令实际上必须在用户环境中执行：用户页表和用户寄存器内容。而且陷阱处理程序最初不知道一些有用的事实，例如正在运行的进程的身份或内核页表的地址。解决方案之所以成为可能，是因为 RISC-V 提供了受保护的地方，内核可以在进入用户空间之前在其中存放信息：`sscratch` 寄存器，以及指向内核内存但因缺少 `PTE_U` 而受保护的用户页表条目。Xv6 的跳板和陷阱帧利用了这些 RISC-V 特性。

如果内核内存被映射到每个进程的用户页表中（带有适当的 PTE 权限标志），则可以消除对特殊跳板页面的需求。这也将消除从用户空间陷入内核时切换页表的需要。这反过来将允许内核中的系统调用实现利用当前进程的用户内存被映射的优势，允许内核代码直接解引用用户指针。许多操作系统使用这些想法来提高效率。Xv6 避免使用它们，以减少因意外使用用户指针而导致内核出现安全漏洞的风险，并减少为确保用户和内核虚拟地址不重叠而需要的一些复杂性。

生产操作系统实现了写时复制 fork、惰性分配、按需分页、分页到磁盘、内存映射文件等。此外，生产操作系统将尝试使用所有物理内存，无论是用于应用程序还是缓存（例如，文件系统的缓冲区缓存，我们将在后面的 8.2 节中介绍）。Xv6 在这方面很幼稚：您希望您的操作系统使用您花钱购买的物理内存，但 xv6 不会。此外，如果 xv6 内存不足，它会向正在运行的应用程序返回错误或杀死它，而不是，例如，驱逐另一个应用程序的页面。

# 5 中断与设备驱动

**驱动程序**是操作系统中管理特定设备的代码：它配置设备硬件，告诉设备执行操作，处理由此产生的中断，并与可能正在等待设备 I/O 的进程进行交互。驱动程序代码可能很棘手，因为驱动程序与其管理的设备并发执行。此外，驱动程序必须理解设备的硬件接口，这可能很复杂且文档不全。

需要注意的操作系统设备通常可以配置为产生**中断**，中断是陷阱的一种类型。内核陷阱处理代码识别出设备何时引发中断，并调用驱动程序的中断处理程序；在 xv6 中，这个分派发生在 `devintr` (kernel/trap.c:177) 中。

许多设备驱动程序在两个上下文中执行代码：在进程的内核线程中运行的**上半部分**，以及在中断时间执行的**下半部分**。上半部分通过系统调用（如希望设备执行 I/O 的 `read` 和 `write`）被调用。此代码可能要求硬件启动一个操作（例如，要求磁盘读取一个块）；然后代码等待操作完成。最终设备完成操作并引发中断。驱动程序的中断处理程序（作为下半部分）确定哪个操作已完成，如果合适则唤醒等待的进程，并告诉硬件开始任何等待的下一个操作。

### 5.1 代码：控制台输入

控制台驱动程序 (kernel/console.c) 是驱动程序结构的一个简单示例。控制台驱动程序通过连接到 RISC-V 的 UART 串行端口硬件接收人类键入的字符。控制台驱动程序一次累积一行输入，处理特殊输入字符，如退格键和 control-u。用户进程（如 shell）使用 `read` 系统调用从控制台获取输入行。当您在 QEMU 中向 xv6 键入输入时，您的按键通过 QEMU 模拟的 UART 硬件传递给 xv6。

驱动程序与之通信的 UART 硬件是由 QEMU 模拟的 16550 芯片 [12]。在真实的计算机上，16550 将管理连接到终端或其他计算机的 RS232 串行链路。在运行 QEMU 时，它连接到您的键盘和显示器。

UART 硬件以一组**内存映射控制寄存器**的形式呈现给软件。这

意味着，存在一些物理地址，RISC-V 硬件将其连接到 UART 设备，因此加载和存储与设备硬件交互，而不是与 RAM 交互。UART 的内存映射地址从 `0x10000000` 开始，即 `UART0` (kernel/memlayout.h:21)。有少数几个 UART 控制寄存器，每个宽度为一个字节。它们相对于 `UART0` 的偏移量在 (kernel/uart.c:22) 中定义。例如，`LSR` 寄存器包含指示是否有输入字符等待软件读取的位。这些字符（如果有）可以从 `RHR` 寄存器读取。每次读取一个字符时，UART 硬件会将其从等待字符的内部 FIFO 中删除，并在 FIFO 为空时清除 `LSR` 中的“就绪”位。UART 发送硬件在很大程度上独立于接收硬件；如果软件向 `THR` 写入一个字节，UART 将发送该字节。

Xv6 的 `main` 调用 `consoleinit` (kernel/console.c:182) 来初始化 UART 硬件。此代码将 UART 配置为当 UART 接收到每个输入字节时产生接收中断，并在 UART 每次完成发送一个输出字节时产生发送完成中断 (kernel/uart.c:53)。

xv6 shell 通过 `init.c` (user/init.c:19) 打开的文件描述符从控制台读取。对 `read` 系统调用的调用通过内核到达 `consoleread` (kernel/console.c:80)。`consoleread` 等待输入到达（通过中断）并缓冲在 `cons.buf` 中，将输入复制到用户空间，并在（整行到达后）返回到用户进程。如果用户尚未键入整行，任何读取进程将在 `sleep` 调用中等待 (kernel/console.c:96)（第 7 章详细解释了 `sleep`）。

当用户键入一个字符时，UART 硬件要求 RISC-V 引发一个中断，这会激活 xv6 的陷阱处理程序。陷阱处理程序调用 `devintr` (kernel/trap.c:177)，它查看 RISC-V `scause` 寄存器，发现中断来自外部设备。然后它询问一个称为 PLIC [1] 的硬件单元，告诉它哪个设备中断了 (kernel/trap.c:186)。如果是 UART，`devintr` 调用 `uartintr`。

`uartintr` (kernel/uart.c:180) 从 UART 硬件读取任何等待的输入字符，并将它们交给 `consoleintr` (kernel/console.c:136)；它不等待字符，因为未来的输入将引发新的中断。`consoleintr` 的工作是累积输入字符到 `cons.buf` 中，直到整行到达。`consoleintr` 特别处理退格键和其他几个字符。当换行符到达时，`consoleintr` 唤醒正在等待的 `consoleread`（如果有的话）。

一旦被唤醒，`consoleread` 将观察到 `cons.buf` 中有一整行，将其复制到用户空间，并（通过系统调用机制）返回到用户空间。

### 5.2 代码：控制台输出

在连接到控制台的文件描述符上的 `write` 系统调用最终会到达 `uartputc` (kernel/uart.c:87)。设备驱动程序维护一个输出缓冲区 (`uart_tx_buf`)，这样写入进程就不必等待 UART 完成发送；相反，`uartputc` 将每个字符附加到缓冲区，调用 `uartstart` 启动设备发送（如果尚未发送），然后返回。`uartputc` 唯一等待的情况是缓冲区已满。

每次 UART 完成发送一个字节时，它都会产生一个中断。`uartintr` 调用 `uartstart`，

===== 第 55 页 =====

它检查设备是否确实已完成发送，并将下一个缓冲的输出字符交给设备。因此，如果一个进程向控制台写入多个字节，通常第一个字节将由 `uartputc` 调用 `uartstart` 发送，而剩余的缓冲字节将在发送完成中断到达时由来自 `uartintr` 的 `uartstart` 调用发送。

需要注意的一个普遍模式是通过缓冲和中断将设备活动与进程活动解耦。即使没有进程等待读取输入，控制台驱动程序也可以处理输入；随后的 `read` 将看到输入。类似地，进程可以发送输出，而不必等待设备。这种解耦可以通过允许进程与设备 I/O 并发执行来提高性能，并且当设备速度慢（如 UART）或需要立即关注（如回显键入的字符）时尤其重要。这种思想有时被称为**I/O 并发**。

### 5.3 驱动中的并发

您可能已经注意到在 `consoleread` 和 `consoleintr` 中调用了 `acquire`。这些调用获取一个锁，该锁保护控制台驱动程序的数据结构免受并发访问。这里存在三个并发危险：不同 CPU 上的两个进程可能同时调用 `consoleread`；硬件可能要求一个 CPU 在它已经在 `consoleread` 内部执行时传递控制台（实际上是 UART）中断；并且硬件可能在不同的 CPU 上传递控制台中断，而 `consoleread` 正在执行。这些危险可能导致竞争条件或死锁。第 6 章探讨了这些问题以及锁如何解决它们。

并发在驱动程序中需要注意的另一个方面是，一个进程可能正在等待来自设备的数据，但指示输入到达的中断可能在不同的进程（或根本没有进程）运行时到达。因此，中断处理程序不允许考虑它们中断的进程或代码。例如，中断处理程序不能安全地使用当前进程的页表调用 `copyout`。中断处理程序通常做相对较少的工作（例如，仅将输入数据复制到缓冲区），并唤醒上半部分代码来完成其余工作。

### 5.4 时钟中断

Xv6 使用时钟中断来维护其时钟并使其能够在计算密集型进程之间切换；`usertrap` 和 `kerneltrap` 中的 `yield` 调用导致了这种切换。时钟中断来自连接到每个 RISC-V CPU 的时钟硬件。Xv6 对此时钟硬件进行编程，使其定期中断每个 CPU。

RISC-V 要求在机器模式下处理时钟中断，而不是监督模式。RISC-V 机器模式在没有分页的情况下执行，并带有一组独立的控制寄存器，因此在实际中不可能在机器模式下运行普通的 xv6 内核代码。因此，xv6 完全独立于上述陷阱机制来处理时钟中断。

在 `main` 之前，`start.c` 中以机器模式执行的代码设置接收时钟中断 (kernel/start.c:62)。工作的一部分是对 CLINT 硬件进行编程，使其在一定的延迟后产生中断。另一部分是设置一个临时区域，类似于

===== 第 56 页 =====

陷阱帧，以帮助中断处理程序保存寄存器（kernel/start.c:56）。最后，它设置 `mtvec` 指向 `timervec` 并启用时钟中断。

当计时器到期时，CLINT 触发中断，RISC-V 在机器模式下（而不是监督模式下）执行 `timervec`。`timervec` 将一些寄存器保存在先前由 `start` 准备的临时区域中，告诉 CLINT 生成下一个时钟中断（kernel/start.c:47），通知中断应该由监督模式代码处理，调用 `devintr`，并返回。

时钟中断通过 `devintr` (kernel/trap.c:189) 重定向回监督模式。它调用 `clockintr`，后者递增 `ticks` 变量（如果进程正在运行，则调用 `yield`），然后返回。

时钟中断处理程序必须小心，因为 xv6 可能正在内核中执行一些敏感的代码（如在调度器中），此时进程被切换出去可能不安全。为了确保安全，`kerneltrap` 在遇到时钟中断时如果当前 CPU 正在运行一个进程，则调用 `yield`。中断处理程序是简洁的，因为它们需要谨慎地执行相对较少的工作，以确保内核代码能够干净地运行。

### 5.5 现实世界

在第 1 章中，我们看到 shell 使用 `read` 和 `write` 与文件描述符交互，包括控制台。驱动程序和控制台接口比本意要复杂，这主要是为了支持并发输入和输出（通过中断）以及同时的 `read` 和 `write`。

UART 驱动程序通过使用缓冲区处理传入数据，并在输入到达时唤醒 `read`。这种模式非常普遍，以至于许多操作系统都包含了类似的功能。许多驱动程序使用直接内存访问（DMA），它在将数据移入和移出设备方面比编程 I/O（PIO）效率高得多，但仍然需要中断和缓冲区。DMA 允许设备硬件直接将数据读入 RAM 或从 RAM 写出，而 CPU 只需在开始和结束时参与。这在涉及大量数据的设备（如磁盘和网络接口）中很常见。

xv6 的控制台驱动程序使用轮询（一种非中断驱动 I/O）处理输出。这简化了事情，但可能很慢，尤其是在输出很多时。对于时间关键型应用程序，使用中断处理输出也很重要，这样就不会因为等待设备发送输出而浪费 CPU 时间。UART 发送完成中断使驱动程序能够避免轮询等待。虽然 xv6 的 `uartputc` 在发送字符时旋转等待 UART 的发送保持寄存器（THR）为空，但 `uartstart` 无需旋转即可发送缓冲的数据。一些驱动程序还使用 DMA 实现输出，从而允许设备从内存中读取数据，而 CPU 无需处理每个字节的 I/O。一些设备允许将它们的一些寄存器映射到用户空间，允许应用程序代码直接与设备交互，而无需内核的干预。

在 xv6 中，控制台是唯一的交互设备，网络和音频等更复杂的 I/O 设备作为外围组件互连（PCI）总线上的设备出现。PCI 总线枚举在启动时执行，允许内核查找并与之交互连接到系统的设备。

PLIC 是将来自外部设备的中断路由到 CPU 的标准 RISC-V 组件。计时器不是通过 PLIC 路由，而是直接连接到 CPU。同样，核间中断（IPI）用于在多核系统中协调 CPU，也直接连接到 CPU。

根据当前的设备负载，I/O 并发允许设备在准备好时中断，但也会导致复杂性和开销。

UART 驱动程序首先将输入数据复制到内核中的缓冲区，然后再复制到用户空间。这在低数据速率下是有意义的，但对于生成或消耗数据非常快的设备来说，这种双重复制会显著降低性能。一些操作系统能够使用 DMA 在用户空间缓冲区和设备硬件之间直接移动数据。

某些计算机的使用要求系统必须在有界时间内响应。例如，在安全关键型系统中，错过截止时间可能导致灾难。Xv6 不适用于硬实时环境。用于硬实时的操作系统往往是链接到应用程序的库，其方式允许分析确定最坏情况响应时间。Xv6 也不适用于软实时应用程序（偶尔错过截止时间是可以接受的），因为 xv6 的调度器过于简单，并且存在禁用中断时间较长的内核代码路径。

# 6 锁

大多数内核（包括 xv6）会交错执行多个活动。交错的一个来源是多处理器硬件：具有多个独立执行 CPU 的计算机，例如 xv6 的 RISC-V。这些多个 CPU 共享物理 RAM，xv6 利用这种共享来维护所有 CPU 都读取和写入的数据结构。这种共享带来了一个 CPU 读取数据结构而另一个 CPU 正在更新它的可能性，甚至多个 CPU 同时更新相同数据的可能性；如果没有仔细的设计，这种并行访问很可能会产生不正确的结果或损坏的数据结构。即使在单处理器上，内核也可能在多个线程之间切换 CPU，导致它们的执行交错。最后，一个设备中断处理程序如果与某些可中断代码修改相同的数据，那么如果中断在不恰当的时刻发生，就可能会损坏数据。**并发**一词指的是多个指令流由于多处理器并行、线程切换或中断而交错的情况。

内核充满了并发访问的数据。例如，两个 CPU 可能同时调用 `kalloc`，从而同时从空闲列表的头部弹出。内核设计者喜欢允许大量并发，因为它可以通过并行性提高性能，并提高响应能力。然而，结果是内核设计者花费大量精力来证明在这种并发情况下的正确性。有许多方法可以实现正确的代码，有些比其他方法更容易推理。旨在实现并发下正确性的策略以及支持它们的抽象被称为**并发控制**技术。

Xv6 根据情况使用多种并发控制技术；还有许多其他可能的技术。本章关注一种广泛使用的技术：**锁**。锁提供了**互斥**，确保一次只有一个 CPU 可以持有该锁。如果程序员为每个共享数据项关联一个锁，并且代码在使用某项时始终持有相关联的锁，那么该项一次将只被一个 CPU 使用。在这种情况下，我们说锁保护数据项。尽管锁是一种易于理解的并发控制机制，但锁的缺点是它们可能会扼杀性能，因为它们串行化了并发操作。

本章的其余部分解释了为什么 xv6 需要锁，xv6 如何实现它们，以及如何使用它们。

### 6.1 竞争条件

作为为什么需要锁的一个例子，考虑两个进程在两个不同的 CPU 上调用 `wait`。`wait` 释放子进程的内存。因此，在每个 CPU 上，内核将调用 `kfree` 来释放子进程的页面。内核分配器维护一个链表：`kalloc()` (kernel/kalloc.c:69) 从空闲页面链表中弹出一个内存页，而 `kfree()` (kernel/kalloc.c:47) 将一个页面推入空闲链表。为了获得最佳性能，我们希望两个父进程的 `kfree` 可以并行执行，而无需相互等待，但鉴于 xv6 的 `kfree` 实现，这是不正确的。

如果单独执行，这个实现是正确的。但是，如果多个副本并发执行，代码就不正确了。如果两个 CPU 同时执行 `push`，它们可能在执行第 16 行之前都执行第 15 行，如图 6.1 所示，这会导致如图 6.2 所示的不正确结果。然后会有两个列表元素，它们的 `next` 都设置为 `list` 的旧值。当第 16 行的两个对 `list` 的赋值发生时，第二个赋值将覆盖第一个；第一个赋值涉及的元素将丢失。

第 16 行的丢失更新是**竞争条件**的一个例子。竞争条件是指一个内存位置被并发访问，并且至少有一次访问是写操作。竞争通常是一个错误的征兆，要么是丢失更新（如果访问都是写操作），要么是读取了未完全更新的数据结构。竞争的结果取决于所涉及的两个 CPU 的确切时间以及内存系统对其内存操作的排序，这可能使竞争引起的错误难以重现和调试。例如，在调试 `push` 时添加 `printf` 语句可能会改变执行的时序，足以使竞争消失。

避免竞争的常用方法是使用**锁**。锁确保互斥，因此一次只有一个 CPU 可以执行 `push` 的敏感行；这使得上述场景不可能发生。上述代码的正确加锁版本只添加了几行（用黄色高亮）：

```c
6  struct element *list = 0;
7  struct lock listlock;
8
9  void
10 push(int data)
11 {
12   struct element *l;
13   l = malloc(sizeof *l);
14   l->data = data;
15
16   acquire(&listlock);
17   l->next = list;
18   list = l;
19   release(&listlock);
20 }
```

`acquire` 和 `release` 之间的指令序列通常被称为**临界区**。通常说该锁保护 `list`。

当我们说一个锁保护数据时，我们的意思是该锁保护适用于该数据的一些**不变量**集合。不变量是跨操作维护的数据结构的属性。通常，操作的正确行为取决于操作开始时不变量的真实性。操作可能会暂时违反不变量，但必须在完成前重新建立它们。例如，在链表的情况下，不变量是 `list` 指向列表中的第一个元素，并且每个元素的 `next` 字段指向下一个元素。`push` 的实现暂时违反了这个不变量：在第 17 行，`l` 指向下一个列表元素，但 `list` 尚未指向 `l`（在第 18 行重新建立）。我们上面检查的竞争条件发生是因为第二个 CPU 在不变量（暂时）被违反时执行了依赖于列表不变量的代码。正确使用锁可确保一次只有一个 CPU 可以操作临界区中的数据结构，因此当数据结构的不变量不成立时，没有 CPU 会执行数据结构操作。

您可以将锁视为串行化并发的临界区，以便它们一个接一个地运行，从而保留不变量（假设临界区在单独执行时是正确的）。您也可以将由同一锁保护的临界区视为彼此原子，因此每个临界区只看到先前临界区的完整更改集，而永远不会看到部分完成的更新。

虽然正确使用锁可以使不正确的代码变得正确，但锁会限制性能。例如，如果两个进程同时调用 `kfree`，锁将串行化这两个调用，并且我们无法从它们在不同 CPU 上运行中获得任何好处。我们说，如果多个进程同时想要相同的锁，则它们会发生**冲突**，或者说该锁经历**争用**。内核设计中的一个主要挑战是避免锁争用。Xv6 在这方面做得很少，但复杂的内核会专门组织数据结构和算法以避免锁争用。在列表示例中，内核可以为每个 CPU 维护一个空闲列表，并且仅当 CPU 的列表为空且必须从另一个 CPU 窃取内存时，才接触另一个 CPU 的空闲列表。其他用例可能需要更复杂的设计。

锁的放置位置对于性能也很重要。例如，将 `acquire` 移到 `push` 中更早的位置是正确的：将 `acquire` 调用向上移动到第 13 行之前是可以的。这可能会降低性能，因为对 `malloc` 的调用也被串行化了。下面的“使用锁”一节提供了一些关于在何处插入 `acquire` 和 `release` 调用的指南。

### 6.2 代码：锁

Xv6 有两种类型的锁：**自旋锁**和**睡眠锁**。我们将从自旋锁开始。Xv6 将自旋锁表示为一个 `struct spinlock` (kernel/spinlock.h:2)。该结构中的重要字段是

不幸的是，这种实现不能保证多处理器上的互斥。可能发生两个 CPU 同时到达第 25 行，看到 `lk->locked` 为零，然后都通过执行第 26 行获得锁。此时，两个不同的 CPU 持有该锁，违反了互斥属性。我们需要一种方法使第 25 行和第 26 行作为一个**原子**（即不可分割的）步骤执行。

由于锁被广泛使用，多核处理器通常提供实现第 25 行和第 26 行的原子版本的指令。在 RISC-V 上，这条指令是 `amoswap r, a`。`amoswap` 读取内存地址 `a` 处的值，将寄存器 `r` 的内容写入该地址，并将它读取的值放入 `r`。也就是说，它交换寄存器和内存地址的内容。它原子地执行这个序列，使用特殊的硬件防止任何其他 CPU 在读取和写入之间使用该内存地址。

Xv6 的 `acquire` (kernel/spinlock.c:22) 使用可移植的 C 库调用 `__sync_lock_test_and_set`，它归结为 `amoswap` 指令；返回值是 `lk->locked` 的旧（被交换的）内容。`acquire` 函数将交换操作包装在一个循环中，重试（自旋）直到它获得锁。每次迭代将 1 交换到 `lk->locked` 并检查先前的值；如果先前的值为零，那么我们已经获得了锁，并且交换已将 `lk->locked` 设置为 1。如果先前的值为 1，则某个其他 CPU 持有该锁，并且我们原子地将 1 交换到 `lk->locked` 这一事实并没有改变其值。

一旦获得锁，`acquire` 会记录获得锁的 CPU，用于调试。`lk->cpu` 字段受锁保护，并且只能在持有锁时更改。

函数 `release` (kernel/spinlock.c:47) 与 `acquire` 相反：它清除 `lk->cpu` 字段，然后释放锁。从概念上讲，`release` 只需要将零赋值给 `lk->locked`。C 标准允许编译器用多个存储指令实现一个赋值，因此 C 赋值可能相对于并发代码是非原子的。相反，`release` 使用 C 库函数 `__sync_lock_release` 执行原子赋值。这个函数也归结为 RISC-V `amoswap` 指令。

### 6.3 代码：使用锁

Xv6 在许多地方使用锁来避免竞争条件。如上所述，`kalloc` (kernel/kalloc.c:69) 和 `kfree` (kernel/kalloc.c:47) 形成了一个很好的例子。尝试练习 1 和 2，看看如果这些函数省略锁会发生什么。您可能会发现很难触发错误行为，这表明可靠地测试代码是否没有锁错误和竞争是很困难的。xv6 很可能存在一些竞争。

使用锁的一个困难部分是决定使用多少锁以及每个锁应该保护哪些数据和不变量。有几个基本原则。首先，任何时候一个变量可以被一个 CPU 写入，同时另一个 CPU 可以读取或写入它，都应该使用锁来防止这两个操作重叠。其次，请记住锁保护不变量：如果不变量涉及多个内存位置，通常所有这些位置都需要由单个锁保护，以确保不变量得到维护。

上述规则说明了何时需要锁，但没有说明何时不需要锁，为了提高效率，不要锁定太多是很重要的，因为锁会降低并行性。如果并行性不重要，那么可以安排只有一个线程，而不必担心锁。一个简单的内核可以通过在进入内核时获取一个必须在退出内核时释放的单一锁来在多处理器上做到这一点（尽管像管道读取或 `wait` 这样的系统调用会带来问题）。许多单处理器操作系统已使用这种方法（有时称为“大内核锁”）转换为在多处理器上运行，但这种方法牺牲了并行性：一次只有一个 CPU 可以在内核中执行。如果内核执行任何繁重的计算，使用一组更大、更细粒度的锁会更有效率，这样内核可以在多个 CPU 上同时执行。

作为粗粒度锁的一个例子，xv6 的 `kalloc.c` 分配器有一个由单个锁保护的空闲列表。如果不同 CPU 上的多个进程尝试同时分配页面，每个进程都必须在 `acquire` 中自旋等待轮到自己。自旋会降低性能，因为它不是有用的工作。如果锁的争用浪费了很大一部分 CPU 时间，也许可以通过更改分配器设计为具有多个空闲列表（每个都有自己的锁）来提高性能，从而允许真正的并行分配。

作为细粒度锁的一个例子，xv6 为每个文件都有一个单独的锁，因此操作不同文件的进程通常可以继续执行而无需等待彼此的锁。如果希望允许进程同时写入同一文件的不同区域，文件锁定方案可以做得更细粒度。最终，锁粒度决策需要由性能测量以及复杂性考虑来驱动。

遵守全局的避免死锁的顺序可能非常困难。有时锁的顺序与逻辑程序结构冲突，例如，代码模块 M1 调用模块 M2，但锁的顺序要求 M2 中的锁必须在 M1 中的锁之前获取。有时锁的身份无法提前知道，也许是因为必须持有一个锁才能发现下一个要获取的锁的身份。这种情况出现在文件系统查找路径名中的连续组件时，以及出现在 `wait` 和 `exit` 搜索进程表时。

### 6.5 可重入锁

看起来，使用**可重入锁**（也称为**递归锁**）可以避免某些死锁和锁顺序挑战。其思想是，如果一个锁被一个进程持有，并且该进程再次尝试获取同一个锁，那么内核可以允许这样做（因为该进程已经拥有该锁），而不是像 xv6 内核那样调用 `panic`。

然而，事实证明，可重入锁使得推理并发更加困难：可重入锁打破了临界区相对于其他临界区是原子的直觉。考虑以下两个函数 `f` 和 `g`：

```c
struct spinlock lock;
int data = 0; // 由 lock 保护

f() {
    acquire(&lock);
    if(data == 0) {
        call_once();
        h();
        data = 1;
    }
    release(&lock);
}

g() {
    acquire(&lock);
    if(data == 0) {
        call_once();
        data = 1;
    }
    release(&lock);
}
```

看这段代码片段，直觉是 `call_once` 只会被调用一次：要么由 `f` 调用，要么由 `g` 调用，但不会两者都调用。

但是，如果允许可重入锁，并且 `h` 恰好调用了 `g`，那么 `call_once` 将被调用两次。

如果不允许可重入锁，那么 `h` 调用 `g` 会导致死锁，这也不太好。但是，假设调用 `call_once` 是一个严重错误，那么死锁更可取。内核开发者将观察到死锁（内核崩溃）并可以修复代码以避免它，而两次调用 `call_once` 可能会默默地导致一个难以追踪的错误。

出于这个原因，xv6 使用了更易于理解的非可重入锁。只要程序员牢记锁定规则，无论哪种方法都可以工作。如果 xv6 要

### 6.6 锁与中断处理程序

一些 xv6 自旋锁保护由线程和中断处理程序共同使用的数据。例如，`clockintr` 时钟中断处理程序可能会在某个内核线程在 `sys_sleep` (kernel/sysproc.c:64) 中读取 `ticks` 的同时递增 `ticks` (kernel/trap.c:163)。锁 `tickslock` 串行化这两个访问。

自旋锁和中断的交互引入了一个潜在的危险。假设 `sys_sleep` 持有 `tickslock`，并且其 CPU 被一个时钟中断打断。`clockintr` 将尝试获取 `tickslock`，看到它被持有，并等待它被释放。在这种情况下，`tickslock` 将永远不会被释放：只有 `sys_sleep` 可以释放它，但 `sys_sleep` 在 `clockintr` 返回之前不会继续运行。所以 CPU 将死锁，任何需要这两个锁的代码也将冻结。

为了避免这种情况，如果自旋锁被中断处理程序使用，那么 CPU 绝对不能启用中断的情况下持有该锁。Xv6 更保守：当一个 CPU 获取任何锁时，xv6 总是禁用该 CPU 上的中断。中断仍可能在其他 CPU 上发生，因此中断的 `acquire` 可以等待线程释放自旋锁；只是不能在同一个 CPU 上。

xv6 当一个 CPU 不持有自旋锁时重新启用中断；它必须做一些簿记来处理嵌套的临界区。`acquire` 调用 `push_off` (kernel/spinlock.c:89)，`release` 调用 `pop_off` (kernel/spinlock.c:100) 来跟踪当前 CPU 上锁的嵌套级别。当该计数达到零时，`pop_off` 恢复在最外层临界区开始时存在的中断启用状态。`intr_off` 和 `intr_on` 函数执行 RISC-V 指令来分别禁用和启用中断。

重要的是，`acquire` 必须在设置 `lk->locked` 之前严格调用 `push_off` (kernel/spinlock.c:28)。如果两者顺序颠倒，将有一个短暂的窗口，在此期间锁被持有而中断是启用的，一个不幸的定时中断将使系统死锁。类似地，`release` 在释放锁之后调用 `pop_off` 也很重要 (kernel/spinlock.c:66)。

### 6.7 指令与内存顺序

很自然地认为程序按照源代码语句出现的顺序执行。然而，许多编译器和 CPU 会乱序执行代码以实现更高的性能。如果一条指令需要多个周期才能完成，CPU 可能会提前发出该指令，以便它可以与其他指令重叠并避免 CPU 停顿。例如，CPU 可能会注意到在一个串行序列中，指令 A 和 B 彼此不依赖。CPU 可能会先启动指令 B，要么因为它的输入比 A 的输入先准备好，要么为了重叠 A 和 B 的执行。编译器可能会通过为一个语句发出指令，使其在源代码中位于它之前的语句的指令之前，来执行类似的重排序。

### 6.8 睡眠锁

有时 xv6 需要长时间持有一个锁。例如，文件系统（第 8 章）在磁盘上读写其内容时保持文件锁定，这些磁盘操作可能需要几十毫秒。长时间持有自旋锁会导致浪费，如果另一个进程想要获取它，因为获取进程会在旋转中浪费 CPU 很长时间。自旋锁的另一个缺点是，进程在持有自旋锁时不能让出 CPU；我们希望这样做，以便当持有锁的进程等待磁盘时，其他进程可以使用 CPU。在持有自旋锁时让出 CPU 是不合法的，因为如果第二个线程随后试图获取自旋锁，可能导致死锁；由于 `acquire` 不会让出 CPU，第二个线程的自旋可能会阻止第一个线程运行并释放锁。在持有自旋锁时让出 CPU 也会违反中断必须在持有自旋锁时禁用的要求。因此，我们需要一种锁类型，它在等待获取时让出 CPU，并且在锁被持有时允许让出（和中断）。

Xv6 以**睡眠锁**的形式提供了这种锁。`acquiresleep` (kernel/sleeplock.c:22) 在等待时让出 CPU，使用第 7 章将解释的技术。在高层，一个睡眠锁有一个受自旋锁保护的 `locked` 字段，并且 `acquiresleep` 对 `sleep` 的调用原子地让出 CPU 并释放自旋锁。结果是，当 `acquiresleep` 等待时，其他线程可以执行。

由于睡眠锁保持中断启用，它们不能在中断处理程序中使用。由于 `acquiresleep` 可能让出 CPU，睡眠锁不能在自旋锁临界区内使用（尽管自旋锁可以在睡眠锁临界区内使用）。

### 6.9 现实世界

尽管锁提供了正确性，但它们也限制了性能。操作系统设计者面临着设计高效数据结构和算法的持续挑战，这些数据结构和算法可以限制锁争用。许多操作系统使用无锁数据结构来避免锁的开销，同时保留正确性。例如，可以使用原子指令实现无锁链表，在插入元素时不需要锁。然而，无锁编程比有锁编程更复杂；例如，必须处理乱序内存访问。尽管 xv6 演示了锁的基本用法，但真正的操作系统通常使用比 xv6 更复杂的并发控制。

在单处理器上，可以使用禁用中断来代替锁。许多操作系统，包括 xv6，都使用禁用中断作为实现低级别同步的原语（例如，在 `push_off` 中）。在多个处理器上，禁用中断不足以保护共享数据结构，因为中断可能由另一个 CPU 处理。为了支持多处理器，xv6 使用锁结合禁用中断。

现代 CPU 提供了相当复杂的内存模型。它们提供的内存排序保证通常比 xv6 所依赖的更强。RISC-V 有一个相对较弱的内存模型 [1]：它允许 CPU 在硬件中执行比 xv6 通过内存屏障防止的更多重新排序。特别是，RISC-V 的“宽松内存模型”允许在写入之间重新排序读取。x86 的内存模型更强，通常允许较少的重新排序。

# 7 调度

任何操作系统都可能运行比计算机 CPU 更多的进程，因此需要一个计划来在进程之间对 CPU 进行分时共享。理想情况下，共享对用户进程应该是透明的。一种常见的方法是通过将进程多路复用到硬件 CPU 上，为每个进程提供它拥有自己虚拟 CPU 的错觉。本章解释 xv6 如何实现这种多路复用。

### 7.1 多路复用

Xv6 通过在两种情况下将每个 CPU 从一个进程切换到另一个进程来实现多路复用。首先，xv6 的 `sleep` 和 `wakeup` 机制在进程等待设备或管道 I/O 完成、等待子进程退出或在 `sleep` 系统调用中等待时进行切换。其次，xv6 定期强制切换，以应对长时间计算而不睡眠的进程。这种多路复用创造了每个进程都有自己的 CPU 的错觉，就像 xv6 使用内存分配器和硬件页表来创造每个进程都有自己的内存的错觉一样。

实现多路复用带来了一些挑战。首先，如何从一个进程切换到另一个进程？虽然上下文切换的思想很简单，但其实现是 xv6 中最不透明的代码。其次，如何以对用户进程透明的方式强制切换？Xv6 使用标准技术，其中硬件时钟的中断驱动上下文切换。第三，所有 CPU 在同一组共享进程之间切换，需要一个锁定计划来避免竞争。第四，当进程退出时，其内存和其他资源必须被释放，但它不能自己完成所有这些工作，因为（例如）它在仍在使用自己的内核栈时无法释放它。第五，多核机器的每个核心必须记住它正在执行哪个进程，以便系统调用影响正确进程的内核状态。最后，`sleep` 和 `wakeup` 允许进程放弃 CPU 并等待被另一个进程或中断唤醒。需要小心避免导致丢失唤醒通知的竞争。Xv6 试图尽可能简单地解决这些问题，但最终代码仍然很棘手。

图 7.1 概述了从一个用户进程切换到另一个用户进程所涉及的步骤：用户-内核转换（系统调用或中断）到旧进程的内核线程，上下文切换到当前 CPU 的调度器线程，上下文切换到新进程的内核线程，以及陷阱返回到用户级进程。xv6 调度器为每个 CPU 有一个专用的线程（保存的寄存器和栈），因为调度器在旧进程的内核栈上执行是不安全的：其他核心可能唤醒该进程并运行它，而在两个不同核心上使用同一个栈将是一场灾难。在本节中，我们将研究在内核线程和调度器线程之间切换的机制。

从一个线程切换到另一个线程涉及保存旧线程的 CPU 寄存器，并恢复新线程先前保存的寄存器；栈指针和程序计数器被保存和恢复这一事实意味着 CPU 将切换栈并切换正在执行的代码。

函数 `swtch` 执行内核线程切换的保存和恢复。`swtch` 不直接了解线程；它只是保存和恢复 32 个 RISC-V 寄存器的集合，称为**上下文**。当一个进程需要放弃 CPU 时，该进程的内核线程调用 `swtch` 来保存其自己的上下文并返回到调度器上下文。每个上下文包含在一个 `struct context` (kernel/proc.h:2) 中，该结构本身包含在进程的 `struct proc` 或 CPU 的 `struct cpu` 中。`Swtch` 接受两个参数：`struct context *old` 和 `struct context *new`。它将当前寄存器保存在 `old` 中，从 `new` 加载寄存器，并返回。

让我们跟踪一个进程通过 `swtch` 进入调度器。我们在第 4 章中看到，中断结束时的一种可能性是 `usertrap` 调用 `yield`。`Yield` 又调用 `sched`，后者调用 `swtch` 将当前上下文保存在 `p->context` 中，并切换到先前保存在 `cpu->scheduler` 中的调度器上下文 (kernel/proc.c:490)。

### 7.3 代码：调度

上一节研究了 `swtch` 的底层细节；现在让我们将 `swtch` 视为已知，并检查从一个进程的内核线程通过调度器切换到另一个进程的过程。调度器以每个 CPU 一个特殊线程的形式存在，每个线程运行 `scheduler` 函数。该函数负责选择接下来运行哪个进程。一个想要放弃 CPU 的进程必须获取自己的进程锁 `p->lock`，释放它持有的任何其他锁，更新自己的状态（`p->state`），然后调用 `sched`。您可以在 `yield` (kernel/proc.c:496)、`sleep` 和 `exit` 中看到这个序列。`Sched` 双重检查其中一些要求 (kernel/proc.c:480-485)，然后检查一个隐含条件：由于持有一个锁，中断应该被禁用。最后，`sched` 调用 `swtch` 将当前上下文保存在 `p->context` 中，并切换到 `cpu->scheduler` 中的调度器上下文。`Swtch` 在调度器的栈上返回，就像调度器的 `swtch` 返回一样。调度器继续其 for 循环，找到一个要运行的进程，切换到它，然后循环重复。

我们刚刚看到 xv6 在调用 `swtch` 时持有 `p->lock`：`swtch` 的调用者必须已经持有该锁，并且锁的控制权传递给切换到的代码。这种约定对于锁来说是不寻常的；通常，获取锁的线程也负责释放锁，这使得推理正确性更容易。对于上下文切换，必须打破这种约定，因为 `p->lock` 保护进程状态和上下文字段的不变量，这些在 `swtch` 执行期间不成立。如果在 `swtch` 期间不持有 `p->lock`，可能发生问题的一个例子是：在 `yield` 将其状态设置为 `RUNNABLE` 之后，但在 `swtch` 使其停止使用自己的内核栈之前，不同的 CPU 可能决定运行该进程。结果将是两个 CPU 在同一个栈上运行，这将导致混乱。

内核线程放弃 CPU 的唯一地方是在 `sched` 中，并且它总是切换到调度器中的同一位置，而调度器（几乎总是）切换到先前调用过 `sched` 的某个内核线程。因此，如果打印出 xv6 切换线程的行号，人们会观察到以下简单模式：(kernel/proc.c:456)、(kernel/proc.c:490)、(kernel/proc.c:456)、(kernel/proc.c:490)，等等。通过线程切换有目的地相互传递控制权的过程有时被称为**协程**；在这个例子中，`sched` 和 `scheduler` 是彼此的协程。

“返回”到该函数的开头。`Forkret` 的存在是为了释放 `p->lock`；否则，由于新进程需要像从 `fork` 返回一样返回到用户空间，它可以直接从 `usertrapret` 开始。

调度器 (kernel/proc.c:438) 运行一个循环：找到一个要运行的进程，运行它直到它让出 CPU，重复。调度器遍历进程表，寻找一个可运行的进程，即 `p->state == RUNNABLE` 的进程。一旦找到进程，它设置每 CPU 当前进程变量 `c->proc`，将进程标记为 `RUNNING`，然后调用 `swtch` 开始运行它 (kernel/proc.c:451-456)。

思考调度代码结构的一种方式是，它强制实施一组关于每个进程的不变量，并在这些不变量不成立时持有 `p->lock`。一个不变量是，如果一个进程是 `RUNNING`，则时钟中断的 `yield` 必须能够安全地切换离开该进程；这意味着 CPU 寄存器必须保存进程的寄存器值（即 `swtch` 尚未将它们移动到上下文），并且 `c->proc` 必须指向该进程。另一个不变量是，如果一个进程是 `RUNNABLE`，那么空闲 CPU 的调度器运行它必须是安全的；这意味着 `p->context` 必须保存进程的寄存器（即，它们实际上不在真实寄存器中），没有 CPU 在该进程的内核栈上执行，并且没有 CPU 的 `c->proc` 指向该进程。观察到，当持有 `p->lock` 时，这些属性通常不成立。

维护上述不变量是 xv6 经常在一个线程中获取 `p->lock` 而在另一个线程中释放它的原因，例如在 `yield` 中获取并在 `scheduler` 中释放。一旦 `yield` 开始修改一个正在运行的进程的状态以使其变为 `RUNNABLE`，锁必须保持直到不变量恢复：最早的正确释放点是调度器（在自己的栈上运行）清除 `c->proc` 之后。类似地，一旦调度器开始将 `RUNNABLE` 进程转换为 `RUNNING`，锁就不能释放，直到内核线程完全运行（在 `swtch` 之后，例如在 `yield` 中）。

### 7.4 代码：mycpu 和 myproc

Xv6 经常需要指向当前进程的 `proc` 结构的指针。在单处理器上，可以有一个指向当前 `proc` 的全局变量。这在多核机器上不起作用，因为每个核心执行不同的进程。解决这个问题的方法是利用每个核心都有一组自己的寄存器这一事实；我们可以使用其中一个寄存器来帮助查找每核心信息。

Xv6 为每个 CPU 维护一个 `struct cpu` (kernel/proc.h:22)，它记录当前在该 CPU 上运行的进程（如果有）、该 CPU 调度器线程的保存寄存器，以及管理中断禁用所需的嵌套自旋锁计数。函数 `mycpu` (kernel/proc.c:72) 返回指向当前 CPU 的 `struct cpu` 的指针。RISC-V 为其 CPU 编号，给每个 CPU 一个 `hartid`。Xv6 确保在内核中，每个 CPU 的 `hartid` 存储在该 CPU 的 `tp` 寄存器中。这允许 `mycpu` 使用 `tp` 索引一个 `cpu` 结构数组以找到正确的那个。

确保 CPU 的 `tp` 始终保存 CPU 的 `hartid` 有点复杂。`mstart` 在 CPU 启动序列的早期，仍在机器模式下时，设置 `tp` 寄存器 (kernel/start.c:51)。`usertrapret` 将 `tp` 保存在跳板页中，因为用户进程可能会修改 `tp`。最后，当从用户空间进入内核时，`uservec` 恢复保存的 `tp` (kernel/trampoline.S:70)。编译器保证永远不会使用 `tp` 寄存器。如果 xv6 可以在需要时向 RISC-V 硬件询问当前的 `hartid` 会更方便，但 RISC-V 只允许在机器模式下进行，不允许在监督模式下进行。

### 7.5 睡眠与唤醒

调度和锁有助于将一个线程的行为隐藏起来不让另一个线程看到，但我们也需要有助于线程有意交互的抽象。例如，xv6 中管道的读者可能需要等待写者进程产生数据；父进程对 `wait` 的调用可能需要等待子进程退出；而读取磁盘的进程需要等待磁盘硬件完成读取。xv6 内核在这些情况下（以及许多其他情况下）使用称为 **`sleep`** 和 **`wakeup`** 的机制。`Sleep` 允许内核线程等待一个特定事件；另一个线程可以调用 `wakeup` 来指示等待某个事件的线程应该恢复。`Sleep` 和 `wakeup` 通常被称为**顺序协调**或**条件同步**机制。

`Sleep` 和 `wakeup` 提供了一个相对低级的同步接口。为了激发它们在 xv6 中的工作方式，我们将使用它们来构建一个更高级的同步机制，称为**信号量** [4]，用于协调生产者和消费者（xv6 不使用信号量）。一个信号量维护一个计数并提供两个操作。“V”操作（对于生产者）递增计数。“P”操作（对于消费者）等待直到计数非零，然后递减它并返回。如果只有一个生产者线程和一个消费者线程，并且它们在不同的 CPU 上执行，并且编译器没有过于激进地优化，那么这个实现将是正确的：

### 7.6 代码：睡眠与唤醒

Xv6 的 `sleep` (kernel/proc.c:529) 和 `wakeup` (kernel/proc.c:560) 提供了上面最后一个例子中所示的接口，并且它们的实现（加上使用它们的规则）确保没有丢失唤醒。基本思想是让 `sleep` 将当前进程标记为 `SLEEPING`，然后调用 `sched` 以释放 CPU；`wakeup` 寻找在给定等待通道上睡眠的进程，并将其标记为 `RUNNABLE`。`sleep` 和 `wakeup` 的调用者可以使用任何双方约定好的数字作为通道。Xv6 经常使用涉及等待的内核数据结构的地址。

`Sleep` 获取 `p->lock` (kernel/proc.c:540)。现在，要进入睡眠的进程同时持有 `p->lock` 和 `lk`。在调用者中持有 `lk` 是必要的（在示例中，是 `P`）：它确保没有其他进程（在示例中，是运行 `V` 的进程）可以开始对 `wakeup(chan)` 的调用。现在 `sleep` 持有 `p->lock`，释放 `lk` 是安全的：其他进程可能开始调用 `wakeup(chan)`，但 `wakeup` 将等待获取 `p->lock`，因此将等待直到 `sleep` 完成将进程置于睡眠状态，从而防止 `wakeup` 错过睡眠。

现在 `sleep` 持有 `p->lock` 而没有其他锁，它可以通过记录睡眠通道、将进程状态更改为 `SLEEPING` 并调用 `sched` 来将进程置于睡眠状态 (kernel/proc.c:544-547)。稍后将清楚为什么在进程被标记为 `SLEEPING` 之前不释放 `p->lock`（由调度器释放）至关重要。

在某个时刻，一个进程将获取条件锁，设置睡眠者在等待的条件，并调用 `wakeup(chan)`。重要的是，`wakeup` 在持有条件锁¹ 时被调用。`Wakeup` 遍历进程表 (kernel/proc.c:560)。它获取它检查的每个进程的 `p->lock`，这既因为它可能操纵该进程的状态，也因为 `p->lock` 确保 `sleep` 和 `wakeup` 不会错过彼此。当 `wakeup` 找到一个状态为 `SLEEPING` 且 `chan` 匹配的进程时，它将进程的状态更改为 `RUNNABLE`。下次调度器运行时，它将看到该进程已准备好运行。

1. xv6 的 `wakeup` 在调用时不持有条件锁，因为它不需要。调用者负责在调用 `wakeup` 之前获取条件锁。

为什么 `sleep` 和 `wakeup` 的锁定规则能确保睡眠进程不会错过唤醒？睡眠进程从检查条件之前到被标记为 `SLEEPING` 之后，要么持有条件锁，要么持有自己的 `p->lock`，或两者都持有。调用 `wakeup` 的进程在 `wakeup` 的循环中持有这两个锁。因此，要么唤醒者在消费线程检查条件之前使条件成立；要么唤醒者的 `wakeup` 在睡眠线程被标记为 `SLEEPING` 之后严格地检查它。然后 `wakeup` 将看到睡眠进程并唤醒它（除非其他东西先将其唤醒）。

有时多个进程在同一个通道上睡眠；例如，多个进程从管道读取。对 `wakeup` 的单个调用将唤醒它们全部。其中一个将首先运行并获取 `sleep` 被调用时持有的锁，并且（在管道的情况下）读取管道中等待的任何数据。其他进程将发现，尽管被唤醒，但没有数据可读。从他们的角度来看，这次唤醒是“虚假的”，他们必须再次睡眠。因此，`sleep` 总是在检查条件的循环内被调用。

如果 `sleep`/`wakeup` 的两次使用意外选择了相同的通道，则不会造成伤害：它们将看到虚假的唤醒，但如上所述的循环将容忍这个问题。`sleep`/`wakeup` 的许多魅力在于它既轻量级（无需创建特殊数据结构作为睡眠通道），又提供了一个间接层（调用者不需要知道他们正在与哪个特定进程交互）。

### 7.7 代码：管道

一个更复杂的例子，使用 `sleep` 和 `wakeup` 来同步生产者和消费者，是 xv6 的管道实现。我们在第 1 章中看到了管道的接口：写入管道一端的数据被复制到内核缓冲区中，然后可以从管道的另一端读取。后续章节将检查围绕管道的文件描述符支持，但让我们现在看看 `pipewrite` 和 `piperead` 的实现。

每个管道由一个 `struct pipe` 表示，其中包含一个锁和一个数据缓冲区。字段 `nread` 和 `nwrite` 计算从缓冲区读取和写入缓冲区的总字节数。缓冲区环绕：在 `buf[PIPESIZE-1]` 之后写入的下一个字节是 `buf[0]`。计数不环绕。这种约定让实现区分满缓冲区（`nwrite == nread+PIPESIZE`）和空缓冲区（`nwrite == nread`），但意味着索引缓冲区必须使用 `buf[nread % PIPESIZE]` 而不是仅仅 `buf[nread]`（对于 `nwrite` 也是如此）。

假设 `piperead` 和 `pipewrite` 的调用同时发生在两个不同的 CPU 上。`Pipewrite` (kernel/pipe.c:77) 首先获取管道的锁，该锁保护计数、数据及其相关不变量。`Piperead` (kernel/pipe.c:106) 然后也尝试获取锁，但不能。它在 `acquire` (kernel/spinlock.c:22) 中自旋等待锁。当 `piperead` 等待时，`pipewrite` 循环遍历正在写入的字节（`addr[0..n-1]`），依次将每个字节添加到管道中 (kernel/pipe.c:95)。在此循环期间，可能发生缓冲区填满的情况 (kernel/pipe.c:88)。在这种情况下，`pipewrite` 调用 `wakeup` 提醒任何睡眠的读者缓冲区中有数据在等待，然后在 `&pi->nwrite` 上睡眠，等待读者从缓冲区中取出一些字节。`Sleep` 作为将 `pipewrite` 的进程置于睡眠状态的一部分，释放 `pi->lock`。

现在 `pi->lock` 可用，`piperead` 设法获取它并进入其临界区：它发现 `pi->nread != pi->nwrite` (kernel/pipe.c:113)（`pipewrite` 因 `pi->nwrite == pi->nread+PIPESIZE` 而进入睡眠 (kernel/pipe.c:88)），因此它进入 for 循环，从管道复制数据 (kernel/pipe.c:120)，并通过复制的字节数递增 `nread`。现在有那么多字节可用于写入，因此 `piperead` 在返回前调用 `wakeup` (kernel/pipe.c:127) 来唤醒任何睡眠的写者。`Wakeup` 找到一个在 `&pi->nwrite` 上睡眠的进程，即正在运行 `pipewrite` 但在缓冲区填满时停止的进程。它将该进程标记为 `RUNNABLE`。

管道代码为读者和写者使用不同的睡眠通道（`pi->nread` 和 `pi->nwrite`）；在不太可能发生的情况下，如果有许多读者和写者等待同一个管道，这可能会使系统更高效。管道代码在检查睡眠条件的循环内睡眠；如果有多个读者或写者，除了第一个被唤醒的进程外，其他进程都会发现条件仍然为假，并再次睡眠。

### 7.8 代码：wait、exit 和 kill

`Sleep` 和 `wakeup` 可用于多种类型的等待。第 1 章介绍的一个有趣例子是子进程的 `exit` 与其父进程的 `wait` 之间的交互。在子进程死亡时，父进程可能已经在 `wait` 中睡眠，或者可能正在做其他事情；在后一种情况下，后续对 `wait` 的调用必须观察到子进程的死亡，这可能是在它调用 `exit` 之后很久。xv6 记录子进程死亡直到 `wait` 注意到它的方式是，`exit` 将调用者放入 `ZOMBIE` 状态，它一直保持该状态，直到父进程的 `wait` 注意到它，将子进程的状态更改为 `UNUSED`，复制子进程的退出状态，并将子进程的进程 ID 返回给父进程。如果父进程在子进程之前退出，父进程将子进程交给 `init` 进程，`init` 进程永远调用 `wait`；因此每个子进程都有一个父进程来清理它。一个挑战是避免同时进行的父进程和子进程的 `wait` 和 `exit` 以及同时的 `exit` 和 `exit` 之间的竞争和死锁。

`Wait` 首先获取 `wait_lock` (kernel/proc.c:384)。原因是 `wait_lock` 充当条件锁，有助于确保父进程不会错过退出子进程的唤醒。然后 `wait` 扫描进程表。如果它找到一个 `ZOMBIE` 状态的子进程，它释放该子进程的资源及其 `proc` 结构，将子进程的退出状态复制到提供给 `wait` 的地址（如果它不为 0），并返回子进程的进程 ID。如果 `wait` 找到子进程但没有退出，它调用 `sleep` 等待其中任何一个退出 (kernel/proc.c:426)，然后再次扫描。`wait` 通常持有两个锁，`wait_lock` 和某个子进程的 `np->lock`；避免死锁的顺序是先 `wait_lock` 后 `np->lock`。

`Exit` (kernel/proc.c:340) 记录退出状态，释放一些资源，调用 `reparent` 将其子进程交给 `init` 进程，唤醒父进程（以防它在 `wait` 中），将调用者标记为僵尸，并永久让出 CPU。`exit` 在此序列期间持有 `wait_lock` 和 `p->lock`。它持有 `wait_lock`，因为它是唤醒的条件锁（`p->parent`），防止 `wait` 中的父进程丢失唤醒。`exit` 还必须为此序列持有 `p->lock`，以防止 `wait` 中的父进程在子进程最终调用 `swtch` 之前看到子进程处于 `ZOMBIE` 状态。`Exit` 以与 `wait` 相同的顺序获取这些锁以避免死锁。

看起来 `exit` 在将状态设置为 `ZOMBIE` 之前唤醒父进程可能是不正确的，但这是安全的：尽管 `wakeup` 可能导致父进程运行，但 `wait` 中的循环无法检查子进程，直到调度器释放子进程的 `p->lock`，因此 `wait` 直到 `exit` 将其状态设置为 `ZOMBIE` 之后很久才能查看退出进程 (kernel/proc.c:372)。

虽然 `exit` 允许进程终止自身，但 `kill` (kernel/proc.c:579) 允许一个进程请求终止另一个进程。让 `kill` 直接销毁目标进程太复杂了，因为目标进程可能正在另一个 CPU 上执行，也许正处于对内核数据结构进行敏感更新的过程中。因此 `kill` 做的很少：它只是设置受害者的 `p->killed`，如果它在睡眠，则唤醒它。最终，受害者将进入或离开内核，此时 `usertrap` 中的代码如果设置了 `p->killed`，将调用 `exit`。如果受害者在用户空间中运行，它将很快通过系统调用或因为时钟（或其他设备）中断而进入内核。

如果受害进程在 `sleep` 中，`kill` 对 `wakeup` 的调用将导致受害者从 `sleep` 返回。这可能是危险的，因为等待的条件可能不成立。然而，xv6 对 `sleep` 的调用总是包装在一个 `while` 循环中，该循环在 `sleep` 返回后重新测试条件。一些对 `sleep` 的调用还会在循环中测试 `p->killed`，如果设置了就放弃当前活动。这仅在放弃是正确的做法时才这样做。例如，管道读写代码在 `killed` 标志被设置时返回；最终代码将返回到陷阱，陷阱将再次检查 `p->killed` 并退出。

一些 xv6 `sleep` 循环不检查 `p->killed`，因为代码处于多步系统调用的中间，该调用应该是原子的。`virtio` 驱动程序 (kernel/virtio_disk.c:273) 是一个例子：它不检查 `p->killed`，因为磁盘操作可能是一组写入操作中的一个，所有这些操作都是为了文件系统保持正确状态所必需的。一个在等待磁盘 I/O 时被杀死的进程在完成当前系统调用并且 `usertrap` 看到 `killed` 标志之前不会退出。

### 7.9 进程锁

与每个进程关联的锁（`p->lock`）是 xv6 中最复杂的锁。思考 `p->lock` 的一个简单方式是，在读取或写入以下任何 `struct proc` 字段时必须持有它：`p->state`、`p->chan`、`p->killed`、`p->xstate` 和 `p->pid`。这些字段可能被其他进程或其他核心上的调度器线程使用，因此它们必须受锁保护是很自然的。

然而，`p->lock` 的大多数用途是保护 xv6 进程数据结构和算法的更高级方面。以下是 `p->lock` 所做的全部事情：

- 与 `p->state` 一起，它防止了在分配新进程的 `proc[]` 槽位时出现竞争。
- 它在进程被创建或销毁时将其隐藏起来。
- 它防止父进程的 `wait` 回收一个已将其状态设置为 `ZOMBIE` 但尚未让出 CPU 的进程。
- 它防止另一个核心的调度器在某个进程将其状态设置为 `RUNNABLE` 之后但在完成 `swtch` 之前决定运行该进程。

- 它确保只有一个核心的调度器决定运行一个 `RUNNABLE` 进程。
- 它防止时钟中断导致进程在 `swtch` 中时让出 CPU。
- 与条件锁一起，它有助于防止 `wakeup` 忽略一个正在调用 `sleep` 但尚未完成让出 CPU 的进程。
- 它防止 `kill` 的目标进程在 `kill` 检查 `p->pid` 和设置 `p->killed` 之间退出并可能被重新分配。
- 它使 `kill` 对 `p->state` 的检查和写入成为原子操作。

`p->parent` 字段受全局锁 `wait_lock` 保护，而不是 `p->lock`。只有进程的父进程修改 `p->parent`，尽管该字段既被进程自身读取，也被其他搜索其子进程的进程读取。`wait_lock` 的目的是在 `wait` 睡眠等待任何子进程退出时充当条件锁。退出的子进程在将其状态设置为 `ZOMBIE`、唤醒其父进程并让出 CPU 之前，持有 `wait_lock` 或 `p->lock`。`wait_lock` 还串行化了父进程和子进程的并发 `exit`，以便保证 `init` 进程（继承子进程）从其 `wait` 中被唤醒。`wait_lock` 是一个全局锁而不是每个父进程的锁，因为进程在获取它之前不知道自己的父进程是谁。

### 7.10 现实世界

xv6 调度器实现了一个简单的调度策略，依次运行每个进程。此策略称为**轮转**。真正的操作系统实现了更复杂的策略，例如，允许进程具有优先级。其思想是，可运行的高优先级进程将比可运行的低优先级进程更受调度器青睐。这些策略可能会很快变得复杂，因为通常存在相互竞争的目标：例如，操作系统可能还想保证公平性和高吞吐量。此外，复杂的策略可能导致意想不到的相互作用，例如**优先级反转**和**拥塞**。当低优先级和高优先级进程都使用某个特定锁时，可能会发生优先级反转，当低优先级进程获取该锁时，可能会阻止高优先级进程取得进展。当许多高优先级进程等待一个持有共享锁的低优先级进程时，可能会形成一长队等待的进程；一旦形成拥塞，它可能会持续很长时间。为了避免这类问题，复杂的调度器中需要额外的机制。

`Sleep` 和 `wakeup` 是一种简单而有效的同步方法，但还有许多其他方法。所有方法中的第一个挑战是避免我们在本章开头看到的“丢失唤醒”问题。原始 Unix 内核的 `sleep` 只是禁用了中断，这在 Unix 运行在单 CPU 系统上时是足够的。由于 xv6 在多处理器上运行，它向 `sleep` 添加了一个显式锁。FreeBSD 的 `msleep` 采用了类似的方法。Plan 9 的 `sleep` 使用了一个回调函数，该函数在即将进入睡眠前持有调度锁时运行；该函数作为对睡眠条件的最后检查，以避免丢失唤醒。Linux 内核的

在 `wakeup` 中扫描整个进程集是低效的。更好的解决方案是用一个包含在该结构上睡眠的进程列表的数据结构替换 `sleep` 和 `wakeup` 中的 `chan`，例如 Linux 的等待队列。Plan 9 的 `sleep` 和 `wakeup` 将该结构称为**汇合点**或 `Rendez`。许多线程库将相同的结构称为**条件变量**；在这种情况下，操作 `sleep` 和 `wakeup` 被称为 `wait` 和 `signal`。所有这些机制都具有相同的风格：睡眠条件受某种锁保护，该锁在睡眠期间原子地释放。

`wakeup` 的实现会唤醒所有在特定通道上等待的进程，可能有很多进程在等待该特定通道。操作系统将调度所有这些进程，它们将竞争检查睡眠条件。以这种方式表现的进程有时被称为**惊群**，最好避免。大多数条件变量有两种唤醒原语：`signal`，唤醒一个进程；以及 `broadcast`，唤醒所有等待的进程。

终止进程并清理它们给 xv6 带来了很多复杂性。在大多数操作系统中，这甚至更复杂，因为，例如，目标进程可能在内核深处睡眠，并且展开其栈需要小心，因为调用栈上的每个函数可能都需要做一些清理工作。一些语言通过提供异常机制来提供帮助，但 C 不行。此外，还有其他事件可能导致睡眠进程被唤醒，即使它正在等待的事件尚未发生。例如，当一个 Unix 进程正在睡眠时，另一个进程可能向它发送一个信号。在这种情况下，进程将从被中断的系统调用返回，返回值为 -1，错误码设置为 `EINTR`。应用程序可以检查这些值并决定做什么。Xv6 不支持信号，因此不会出现这种复杂性。

Xv6 对 `kill` 的支持并不完全令人满意：存在一些 `sleep` 循环，它们可能应该检查 `p->killed`。一个相关的问题是，即使对于检查 `p->killed` 的 `sleep` 循环，在 `sleep` 和 `kill` 之间也存在竞争；后者可能刚好在受害者的循环检查 `p->killed` 之后但在它调用 `sleep` 之前设置 `p->killed` 并尝试唤醒受害者。如果发生此问题，受害者直到它等待的条件发生才会注意到 `p->killed`。这可能是相当长的时间，甚至永远不会发生（例如，如果受害者正在等待控制台输入，但用户没有键入任何输入）。

真正的操作系统会使用常量时间的显式空闲列表来查找空闲的 `proc` 结构，而不是 `allocproc` 中的线性时间搜索；xv6 为了简单而使用线性扫描。

# 第 8 章

## 文件系统

文件系统的目的是组织和存储数据。文件系统通常支持用户和应用程序之间的数据共享，以及持久性，以便在重启后数据仍然可用。

xv6 文件系统提供了类 Unix 的文件、目录和路径名（参见第 1 章），并将其数据存储在 virtio 磁盘上以实现持久性。文件系统解决了几个挑战：

- 文件系统需要**磁盘上的数据结构**来表示命名目录和文件的树，记录保存每个文件内容的块的身份，并记录磁盘的哪些区域是空闲的。
- 文件系统必须支持**崩溃恢复**。也就是说，如果发生崩溃（例如，电源故障），文件系统在重启后必须仍然正常工作。风险在于崩溃可能会中断一系列更新，并留下不一致的磁盘数据结构（例如，一个块既在文件中使用又被标记为空闲）。
- 不同的进程可能同时操作文件系统，因此文件系统代码必须协调以维护不变量。
- 访问磁盘比访问内存慢几个数量级，因此文件系统必须维护一个热门块的内存**缓存**。

本章的其余部分解释了 xv6 如何应对这些挑战。

### 8.1 概述

xv6 文件系统实现组织为七个层，如图 8.1 所示。**磁盘层**在 virtio 硬盘上读取和写入块。**缓冲区缓存层**缓存磁盘块并同步对它们的访问，确保一次只有一个内核进程可以修改任何特定块中存储的数据。**日志层**允许更高层将对多个块的更新包装在一个**事务**中，并确保在面对崩溃时原子地更新块（即，所有块都更新或都不更新）。**Inode 层**提供单个文件，每个文件

### 8.2 缓冲区缓存层

缓冲区缓存有两个任务：（1）同步对磁盘块的访问，以确保内存中只有一个块的副本，并且一次只有一个内核线程使用该副本；（2）缓存热门块，以便无需从慢速磁盘重新读取。代码在 `bio.c` 中。

缓冲区缓存导出的主要接口是 `bread` 和 `bwrite`；前者获取一个 `buf`，其中包含一个块的可读或可修改的内存副本，后者将修改后的缓冲区写入磁盘上的相应块。内核线程在使用完缓冲区后必须调用 `brelse` 释放它。缓冲区缓存使用每缓冲区的睡眠锁来确保一次只有一个线程使用每个缓冲区（以及每个磁盘块）；`bread` 返回一个锁定的缓冲区，`brelse` 释放该锁。

让我们回到缓冲区缓存。缓冲区缓存有固定数量的缓冲区来保存磁盘块，这意味着如果文件系统请求一个尚未在缓存中的块，缓冲区缓存必须回收一个当前保存其他块的缓冲区。缓冲区缓存回收**最近最少使用**的缓冲区用于新块。假设是最近最少使用的缓冲区是最不可能很快再次使用的缓冲区。

### 8.3 代码：缓冲区缓存

缓冲区缓存是一个双向链表。由 `main` (kernel/main.c:27) 调用的函数 `binit`，用静态数组 `buf` (kernel/bio.c:43-52) 中的 `NBUF` 个缓冲区初始化链表。对缓冲区缓存的所有其他访问都通过 `bcache.head` 引用链表，而不是 `buf` 数组。

一个缓冲区有两个与之关联的状态字段。字段 `valid` 指示缓冲区包含块的有效副本。字段 `disk` 指示缓冲区内容已交给磁盘，磁盘可能会更改缓冲区（例如，将数据从磁盘写入 `data`）。

`Bread` (kernel/bio.c:93) 调用 `bget` 来获取给定扇区的缓冲区 (kernel/bio.c:97)。如果需要从磁盘读取缓冲区，`bread` 在返回缓冲区之前调用 `virtio_disk_rw` 来完成读取。

`Bget` (kernel/bio.c:59) 扫描缓冲区列表，寻找具有给定设备和扇区号的缓冲区 (kernel/bio.c:65-73)。如果存在这样的缓冲区，`bget` 获取该缓冲区的睡眠锁。然后 `bget` 返回锁定的缓冲区。

如果没有给定扇区的缓存缓冲区，`bget` 必须创建一个，可能重用保存不同扇区的缓冲区。它第二次扫描缓冲区列表，寻找一个未使用的缓冲区（`b->refcnt = 0`）；任何这样的缓冲区都可以使用。`Bget` 编辑缓冲区元数据以记录新的设备和扇区号，并获取其睡眠锁。注意，赋值 `b->valid = 0` 确保 `bread` 将从磁盘读取块数据，而不是错误地使用缓冲区的

先前内容。

重要的是，每个磁盘扇区最多只有一个缓存的缓冲区，以确保读者看到写入，并且因为文件系统使用缓冲区上的锁进行同步。`Bget` 通过在第一循环检查块是否被缓存的整个过程中，到第二循环声明块现在已被缓存（通过设置 `dev`、`blockno` 和 `refcnt`）期间，持续持有 `bcache.lock` 来保证这个不变量。这使得检查块是否存在以及（如果不存在）指定一个缓冲区来保存该块成为原子操作。

`bget` 在 `bcache.lock` 临界区之外获取缓冲区的睡眠锁是安全的，因为非零的 `b->refcnt` 阻止了缓冲区被重新用于不同的磁盘块。睡眠锁保护对缓冲区缓存内容的读写，而 `bcache.lock` 保护关于哪些块被缓存的信息。

如果所有缓冲区都忙，则意味着太多进程同时执行文件系统调用；`bget` 会崩溃。更优雅的响应可能是睡眠直到缓冲区空闲，尽管那样会有死锁的可能性。

一旦 `bread` 读取了磁盘（如果需要）并将缓冲区返回给其调用者，调用者就独占使用该缓冲区，并且可以读取或写入数据字节。如果调用者修改了缓冲区，它必须在释放缓冲区之前调用 `bwrite` 将更改的数据写入磁盘。`Bwrite` (kernel/bio.c:107) 调用 `virtio_disk_rw` 与磁盘硬件通信。

当调用者使用完缓冲区后，它必须调用 `brelse` 来释放它。（`brelse` 这个名字是 `b-release` 的缩写，虽然晦涩但值得学习：它起源于 Unix，并在 BSD、Linux 和 Solaris 中使用。）`Brelse` (kernel/bio.c:117) 释放睡眠锁，并将缓冲区移动到链表的前面 (kernel/bio.c:128-133)。移动缓冲区使得链表按缓冲区最近使用的顺序排序（意思是释放）：链表中的第一个缓冲区是最近使用的，最后一个是最近最少使用的。`bget` 中的两个循环利用了这一点：对现有缓冲区的扫描在最坏情况下必须处理整个列表，但首先检查最近使用的缓冲区（从 `bcache.head` 开始并沿着 `next` 指针）将在有良好局部性时减少扫描时间。选择要重用的缓冲区的扫描通过向后扫描（沿着 `prev` 指针）选择最近最少使用的缓冲区。

### 8.4 日志层

文件系统设计中最有趣的问题之一是**崩溃恢复**。问题出现是因为许多文件系统操作涉及对磁盘的多次写入，并且在一部分写入之后发生的崩溃可能会使磁盘上的文件系统处于不一致状态。例如，假设在文件截断（将文件长度设置为零并释放其内容块）期间发生崩溃。根据磁盘写入的顺序，崩溃可能留下一个引用了一个被标记为空闲的内容块的 inode，或者可能留下一个已分配但未被引用的内容块。

后者相对良性，但引用一个已释放块的 inode 可能会在重启后导致严重问题。重启后，内核可能会将该块分配给另一个文件，现在我们有两个不同的文件无意中指向同一个块。如果 xv6 支持多用户，这种情况可能是一个安全问题，因为旧文件的所有者将能够读取

===== 第 89 页 =====

和写入新文件中的块，而新文件由不同的用户拥有。

Xv6 通过一种简单的**日志**形式解决了文件系统操作期间崩溃的问题。一个 xv6 系统调用不会直接写入磁盘上的文件系统数据结构。相反，它将希望进行的所有磁盘写入的描述放在磁盘上的**日志**中。一旦系统调用记录了其所有写入，它会在磁盘上写入一个特殊的**提交记录**，指示日志包含一个完整的操作。此时，系统调用将写入复制到磁盘上的文件系统数据结构。在这些写入完成后，系统调用会擦除磁盘上的日志。

如果系统崩溃并重启，文件系统代码会在运行任何进程之前从崩溃中恢复如下。如果日志被标记为包含一个完整的操作，则恢复代码会将写入复制到磁盘上文件系统中的相应位置。如果日志未被标记为包含完整操作，则恢复代码将忽略该日志。恢复代码最后擦除日志。

为什么 xv6 的日志解决了文件系统操作期间崩溃的问题？如果崩溃发生在操作提交之前，那么磁盘上的日志将不会被标记为完成，恢复代码将忽略它，并且磁盘的状态就像操作从未开始一样。如果崩溃发生在操作提交之后，那么恢复将重放该操作的所有写入，如果操作已经开始将其写入磁盘数据结构，则可能会重复这些写入。在任何一种情况下，日志使操作对于崩溃是原子的：恢复后，要么操作的所有写入都出现在磁盘上，要么都不出现。

### 8.5 日志设计

日志位于一个已知的固定位置，在超级块中指定。它由一个**头块**后跟一系列更新的块副本（“日志块”）组成。头块包含一个扇区号数组，每个日志块一个，以及日志块的数量。磁盘上头块中的计数要么为零，表示日志中没有事务，要么非零，表示日志包含一个完整的已提交事务，具有指示数量的日志块。Xv6 在事务提交时写入头块，但在此之前不写，并在将日志块复制到文件系统后将计数设置为零。因此，事务中途发生的崩溃将导致日志头块中的计数为零；提交后发生的崩溃将导致非零计数。

每个系统调用的代码指示对于崩溃必须是原子的写入序列的开始和结束。为了允许不同进程并发执行文件系统操作，日志系统可以将多个系统调用的写入累积到一个事务中。因此，单个提交可能涉及多个完整系统调用的写入。为了避免将系统调用拆分到多个事务中，日志系统只在没有文件系统系统调用正在进行时提交。

将多个事务一起提交的想法称为**组提交**。组提交减少了磁盘操作的数量，因为它将提交的固定成本分摊到多个操作上。组提交还同时向磁盘系统提供更多的并发写入，可能允许磁盘在一次磁盘旋转中写入它们。Xv6 的 virtio 驱动程序不支持这种批处理，但 xv6 的文件系统设计允许它。

Xv6 在磁盘上分配固定大小的空间来保存日志。一个事务中系统调用写入的块总数必须能放入该空间。这有两个后果。不允许任何单个系统调用写入比日志空间更多的不同块。这对大多数系统调用来说不是问题，但其中两个可能会写入许多块：`write` 和 `unlink`。大的文件写入可能会写入许多数据块、许多位图块以及一个 inode 块；取消链接一个大文件可能会写入许多位图块和一个 inode。Xv6 的 `write` 系统调用将大的写入分解成多个适合日志的较小写入，而 `unlink` 不会造成问题，因为在实践中 xv6 文件系统只使用一个位图块。日志空间有限的另一个后果是，除非确定系统调用的写入将适合日志中的剩余空间，否则日志系统不允许系统调用启动。

### 8.6 代码：日志

系统调用中日志的典型使用如下所示：

```c

end_op();
```

`begin_op` (kernel/log.c:127) 等待，直到日志系统当前未在提交，并且有足够的未保留日志空间来容纳此次调用的写入。`log.outstanding` 计数已保留日志空间的系统调用数量；总保留空间是 `log.outstanding` 乘以 `MAXOPBLOCKS`。增加 `log.outstanding` 既保留了空间，又防止在此系统调用期间发生提交。代码保守地假设每个系统调用可能写入多达 `MAXOPBLOCKS` 个不同的块。

`log_write` (kernel/log.c:215) 充当 `bwrite` 的代理。它记录块的扇区号在内存中，为它在磁盘上的日志中保留一个槽位，并将缓冲区固定在块缓存中，以防止块缓存驱逐它。块必须一直留在缓存中直到提交：在此之前，缓存的副本是修改的唯一记录；它不能在提交之前被写入到它在磁盘上的位置；并且同一事务中的其他读取必须看到修改。`log_write` 注意到一个块在单个事务中被多次写入时，会为该块分配日志中相同的槽位。这种优化通常称为**吸收**。例如，包含几个文件的 inode 的磁盘块在一个事务中被多次写入是很常见的。通过将多次磁盘写入吸收为一次，文件系统可以节省日志空间，并可以实现更好的性能，因为只有一个磁盘块的副本必须写入磁盘。

`end_op` (kernel/log.c:147) 首先减少未完成系统调用的计数。如果计数现在为零，它将通过调用 `commit()` 提交当前事务。此过程有四个阶段。`write_log()` (kernel/log.c:179) 将事务中修改的每个块从缓冲区缓存复制到其在磁盘日志中的槽位。`write_head()` (kernel/log.c:103) 将头块写入磁盘：这是提交点，此次写入之后的崩溃将导致恢复重放

### 8.7 代码：块分配器

文件和目录的内容存储在磁盘块中，这些块必须从空闲池中分配。Xv6 的块分配器在磁盘上维护一个空闲**位图**，每个块一个位。零位表示对应的块空闲；一位表示正在使用。程序 `mkfs` 设置与引导扇区、超级块、日志块、inode 块和位图块对应的位。

块分配器提供两个函数：`balloc` 分配一个新的磁盘块，`bfree` 释放一个块。`Balloc` 在 (kernel/fs.c:71) 中的循环考虑每个块，从块 0 开始到 `sb.size`（文件系统中的块数）。它寻找一个位图位为零的块，指示它是空闲的。如果 `balloc` 找到这样一个块，它会更新位图并返回该块。为了提高效率，循环被分成两部分。外部循环读取每个位图位块。内部循环检查单个位图块中的所有 `BPB` 位。如果两个进程同时尝试分配块可能发生的竞争被缓冲区缓存一次只允许一个进程使用任何一个位图块这一事实所阻止。

`Bfree` (kernel/fs.c:90) 找到正确的位图块并清除正确的位。同样，`bread` 和 `brelse` 暗示的独占使用避免了显式锁定的需要。

### 8.8 Inode 层

术语 **inode** 可以有两种相关的含义。它可能指的是磁盘上的数据结构，包含文件的大小和数据块编号列表。或者“inode”可能指的是内存中的 inode，它包含磁盘上 inode 的副本以及内核需要的额外信息。

磁盘上的 inode 被紧凑地排列在磁盘上一个称为 **inode 块**的连续区域中。每个 inode 大小相同，因此给定一个数字 n，很容易找到磁盘上的第 n 个 inode。实际上，这个数字 n 称为 **inode 编号**或 **i-number**，是实现在中识别 inode 的方式。

磁盘上的 inode 由 `struct dinode` (kernel/fs.h:32) 定义。`type` 字段区分文件、目录和特殊文件（设备）。类型为零表示磁盘 inode 是空闲的。`nlink` 字段计数引用此 inode 的目录条目数，以便识别何时应释放磁盘 inode 及其数据块。`size` 字段记录文件内容的字节数。`addr` 数组记录保存文件内容的磁盘块的块号。

内核将活动 inode 的集合保存在一个称为 `itable` 的内存表中；`struct inode` (kernel/file.h:17) 是磁盘上 `struct dinode` 的内存副本。仅当有 C 指针指向该 inode 时，内核才将该 inode 保存在内存中。`ref` 字段计数指向内存中 inode 的 C 指针数量，当引用计数降至零时，内核从内存中丢弃该 inode。函数 `iget` 和 `iput` 获取和释放指向 inode 的指针，修改引用计数。指向 inode 的指针可以来自文件描述符、当前工作目录和临时内核代码（如 `exec`）。

在 xv6 的 inode 代码中有四种锁或类锁机制。`itable.lock` 保护一个不变量，即一个 inode 在 inode 表中最多出现一次，以及内存中 inode 的 `ref` 字段计数指向该 inode 的内存指针数量。每个内存中的 inode 有一个 `lock` 字段，包含一个睡眠锁，它确保对该 inode 的字段（如文件长度）以及该 inode 的文件或目录内容块的独占访问。一个 inode 的 `ref` 如果大于零，会导致系统将该 inode 保留在表中，并且不会将表项重新用于不同的 inode。最后，每个 inode 包含一个 `nlink` 字段（在磁盘上，如果在内存中则复制到内存），计数引用文件的目录条目数；如果其链接计数大于零，xv6 不会释放 inode。

由 `iget()` 返回的 `struct inode` 指针保证有效，直到相应的 `iput()` 调用；inode 不会被删除，并且指针指向的内存不会被重新用于不同的 inode。`iget()` 提供对 inode 的非独占访问，因此可以有多个指针指向同一个 inode。文件系统代码的许多部分都依赖于 `iget()` 的这种行为，既可以持有对 inode 的长期引用（如打开的文件和当前目录），又可以在操作多个 inode（如路径名查找）的代码中避免死锁的同时防止竞争。

`iget` 返回的 `struct inode` 可能没有任何有用的内容。为了确保它包含磁盘上 inode 的副本，代码必须调用 `ilock`。这会锁定 inode（以便没有其他进程可以 `ilock` 它），并从磁盘读取 inode（如果尚未读取）。`iunlock` 释放 inode 上的锁。将 inode 指针的获取与锁定分离有助于在某些情况下避免死锁，例如在目录查找期间。多个进程可以持有一个指向由 `iget` 返回的 inode 的指针

### 8.9 代码：Inode

为了分配一个新的 inode（例如，在创建文件时），xv6 调用 `ialloc` (kernel/fs.c:196)。`Ialloc` 类似于 `balloc`：它遍历磁盘上的 inode 结构，一次一个块，寻找一个被标记为空闲的。当它找到一个时，它通过将新的类型写入磁盘来声明它，然后通过对 `iget` 的尾调用来返回 inode 表中的一个条目 (kernel/fs.c:210)。`ialloc` 的正确操作依赖于这样一个事实：一次只有一个进程可以持有对 `bp` 的引用：`ialloc` 可以确信没有其他进程同时看到该 inode 可用并试图声明它。

`Iget` (kernel/fs.c:243) 在 inode 表中查找一个具有所需设备和 inode 编号的活动条目（`ip->ref > 0`）。如果找到，它返回对该 inode 的新引用 (kernel/fs.c:252-256)。在 `iget` 扫描时，它记录第一个空槽位的位置 (kernel/fs.c:257-258)，如果它需要分配一个新的表项，就会使用该位置。

在读取或写入其元数据或内容之前，代码必须使用 `ilock` 锁定 inode。`Ilock` (kernel/fs.c:289) 为此目的使用睡眠锁。一旦 `ilock` 独占访问 inode，它会从磁盘（更可能从缓冲区缓存）读取 inode（如果需要）。函数 `iunlock` (kernel/fs.c:317) 释放睡眠锁，这可能导致任何正在睡眠的进程被唤醒。

`Iput` (kernel/fs.c:333) 通过递减引用计数来释放指向 inode 的 C 指针 (kernel/fs.c:356)。如果这是最后一个引用，那么 inode 在 inode 表中的槽位现在是空闲的，可以重新用于不同的 inode。

如果 `iput` 看到没有 C 指针引用一个 inode，并且该 inode 没有指向它的链接（不在任何目录中），那么该 inode 及其数据块必须被释放。`Iput` 调用 `itrunc` 将文件截断为零字节，释放数据块；将 inode 类型设置为 0（未分配）；并将 inode 写入磁盘 (kernel/fs.c:338)。

在 `iput` 释放 inode 的情况下，其锁定协议值得仔细研究。一个危险是，一个并发线程可能正在 `ilock` 中等待使用此 inode（例如，读取文件或列出目录），并且不会准备好发现 inode 不再被分配。这不可能发生，因为如果 inode 没有指向它的链接并且 `ip->ref` 为 1，则系统调用无法获得指向内存中 inode 的指针。这一个引用是调用 `iput` 的线程拥有的引用。确实，`iput` 在其 `itable.lock` 临界区之外检查引用计数是否为 1，但在那时链接计数已知为零，因此没有线程会尝试获取新的引用。另一个主要危险是，对 `ialloc` 的并发调用可能会选择与 `iput` 正在释放的相同的 inode。这只有在 `iupdate` 将 inode 写入磁盘使其类型为零之后才会发生。这种竞争是良性的；分配线程会在读取或写入 inode 之前礼貌地等待获取 inode 的睡眠锁，此时 `iput` 已经完成了它。

`Iput()` 可以写入磁盘。这意味着任何使用文件系统的系统调用都可能写入磁盘，因为该系统调用可能是最后一个持有文件引用的调用。即使是像 `read()` 这样看起来是只读的调用，也可能最终调用 `iput()`。这反过来意味着，如果使用文件系统，即使是只读系统调用也必须包装在事务中。

`iput()` 和崩溃之间存在一个具有挑战性的交互。当文件的链接计数降至零时，`iput()` 不会立即截断文件，因为某个进程可能仍然在内存中持有对 inode 的引用：一个进程可能仍然在读取和写入文件，因为它成功打开了它。但是，如果在最后一个进程关闭文件的文件描述符之前发生崩溃，那么该文件将在磁盘上被标记为已分配，但没有目录条目指向它。

文件系统以两种方式之一处理这种情况。简单的解决方案是，在重启后恢复时，文件系统扫描整个文件系统，寻找标记为已分配但没有目录条目指向它们的文件。如果存在任何这样的文件，那么它可以释放这些文件。

第二种解决方案不需要扫描文件系统。在这种解决方案中，文件系统在磁盘上（例如，在超级块中）记录一个文件的 inode 编号，该文件的链接计数降至零但引用计数不为零。如果文件系统在其引用计数达到 0 时删除文件，则通过从列表中删除该 inode 来更新磁盘上的列表。在恢复时，文件系统释放列表中的任何文件。

Xv6 两种解决方案都没有实现，这意味着即使不再使用 inode，它们也可能在磁盘上被标记为已分配。这意味着随着时间的推移，xv6 面临磁盘空间耗尽的风险。

### 8.11 代码：目录层

目录在内部实现得与文件非常相似。其 inode 类型为 `T_DIR`，其数据是一系列目录条目。每个条目是一个 `struct dirent` (kernel/fs.h:56)，包含一个名称和一个 inode 编号。名称最多 `DIRSIZ`（14）个字符；如果较短，则以 NUL（0）字节终止。inode 编号为零的目录条目是空闲的。

函数 `dirlookup` (kernel/fs.c:530) 在目录中搜索具有给定名称的条目。如果找到，它返回指向相应 inode 的指针（未锁定），并将 `*poff` 设置为目录中条目的字节偏移量，以防调用者希望编辑它。如果 `dirlookup` 找到具有正确名称的条目，它会更新 `*poff` 并返回通过 `iget` 获得的未锁定 inode。`Dirlookup` 是 `iget` 返回未锁定 inode 的原因。调用者已锁定 `dp`，因此如果查找的是 `.`（当前目录的别名），在返回前尝试锁定 inode 将尝试重新锁定 `dp` 并导致死锁。（存在涉及多个进程和 `..`（父目录的别名）的更复杂的死锁场景；`.` 不是唯一的问题。）调用者可以解锁 `dp`，然后锁定 `ip`，确保一次只持有一个锁。

函数 `dirlink` (kernel/fs.c:557) 将具有给定名称和 inode 编号的新目录条目写入目录 `dp`。如果名称已存在，`dirlink` 返回错误 (kernel/fs.c:563-567)。主循环读取目录条目，寻找一个未分配的条目。当它找到一个时，它会提前停止循环 (kernel/fs.c:541-542)，`off` 设置为可用条目的偏移量。否则，循环结束，`off` 设置为 `dp->size`。无论哪种方式，`dirlink` 然后通过在偏移量 `off` 处写入 (kernel/fs.c:577-580) 向目录添加一个新条目。

### 8.12 代码：路径名

路径名查找涉及对 `dirlookup` 的一系列调用，每个路径分量一次。`Namei` (kernel/fs.c:664) 计算 `path` 并返回相应的 inode。函数 `nameiparent` 是一个变体：它在最后一个元素之前停止，返回父目录的 inode 并将最后一个元素复制到 `name` 中。两者都调用通用函数 `namex` 来完成实际工作。

`Namex` (kernel/fs.c:629) 首先决定路径评估从哪里开始。如果路径以斜杠开头，评估从根目录开始；否则，从当前目录开始 (kernel/fs.c:633-636)。然后它使用 `skipelem` 依次考虑路径的每个元素 (kernel/fs.c:638)。循环的每次迭代必须在当前 inode `ip` 中查找名称。迭代开始时锁定 `ip` 并检查它是否是一个目录。如果不是，查找失败 (kernel/fs.c:639-643)。（锁定 `ip` 是必要的，不是因为 `ip->type` 会在脚下改变——它不能——而是因为在运行锁之前，不能保证 `ip->type` 已从磁盘加载。）如果调用是 `nameiparent` 并且这是最后一个路径元素，则循环提前停止，按照 `nameiparent` 的定义；最后一个路径元素已被复制到 `name` 中，因此 `namex` 只需要返回未锁定的 `ip` (kernel/fs.c:644-648)。最后，循环使用 `dirlookup` 查找路径元素，并通过设置 `ip = next` 为下一次迭代做准备 (kernel/fs.c:649-654)。当循环用完路径元素时，它返回 `ip`。

### 8.13 文件描述符层

Unix 接口的一个很酷的方面是，Unix 中的大多数资源都表示为文件，包括控制台、管道等设备，当然还有真正的文件。文件描述符层是实现这种统一性的层。

如第 1 章所述，Xv6 为每个进程提供了自己的打开文件表，即文件描述符。每个打开的文件由一个 `struct file` (kernel/file.h:1) 表示，它是 inode 或管道以及 I/O 偏移量的包装器。每次调用 `open` 都会创建一个新的打开文件（一个新的 `struct file`）：如果多个进程独立打开同一个文件，不同的实例将有不同的 I/O 偏移量。另一方面，一个单一的打开文件（同一个 `struct file`）可以在一个进程的文件表中多次出现，也可以在多个进程的文件表中出现。如果一个进程使用 `open` 打开文件，然后使用 `dup` 创建别名，或者使用 `fork` 与子进程共享它，就会发生这种情况。引用计数跟踪对特定打开文件的引用数。文件可以以只读、只写或读写方式打开。`readable` 和 `writable` 字段跟踪这一点。

系统中所有打开的文件都保存在一个全局文件表 `ftable` 中。文件表具有分配文件（`filealloc`）、创建重复引用（`filedup`）、释放引用（`fileclose`）以及读写数据（`fileread` 和 `filewrite`）的函数。

前三个遵循现在熟悉的形式。`Filealloc` (kernel/file.c:30) 扫描文件表以寻找未引用的文件（`f->ref == 0`）并返回一个新引用；`filedup` (kernel/file.c:48) 递增引用计数；`fileclose` (kernel/file.c:60) 递减引用计数。当文件的引用计数达到零时，`fileclose` 根据类型释放底层的管道或 inode。

函数 `filestat`、`fileread` 和 `filewrite` 实现对文件的 `stat`、`read` 和 `write` 操作。`Filestat` (kernel/file.c:88) 仅允许在 inode 上使用，并调用 `stati`。`Fileread`

### 8.14 代码：系统调用

有了较低层提供的函数，大多数系统调用的实现都很简单（参见 (kernel/sysfile.c)）。有几个调用值得仔细看看。

函数 `sys_link` 和 `sys_unlink` 编辑目录，创建或删除对 inode 的引用。它们是使用事务力量的另一个很好的例子。`Sys_link` (kernel/sysfile.c:120) 首先获取其两个参数，两个字符串 `old` 和 `new` (kernel/sysfile.c:125)。假设 `old` 存在并且不是一个目录 (kernel/sysfile.c:129-132)，`sys_link` 递增其 `ip->nlink` 计数。然后 `sys_link` 调用 `nameiparent` 来找到 `new` 的父目录和最后一个路径元素 (kernel/sysfile.c:145)，并创建一个指向 `old` 的 inode 的新目录条目 (kernel/sysfile.c:148)。新父目录必须存在并且与现有 inode 在同一设备上：inode 编号仅在单个磁盘上具有唯一含义。如果发生这样的错误，`sys_link` 必须返回并递减 `ip->nlink`。

事务简化了实现，因为它需要更新多个磁盘块，但我们不必担心我们做它们的顺序。它们要么全部成功，要么都不成功。例如，没有事务，在创建链接之前更新 `ip->nlink` 会使文件系统暂时处于不安全状态，并且在此期间发生的崩溃可能会造成严重破坏。有了事务，我们不必担心这个。

`Sys_link` 为现有的 inode 创建一个新名称。函数 `create` (kernel/sysfile.c:242) 为一个新的 inode 创建一个新名称。它是三个文件创建系统调用的概括：带有 `O_CREATE` 标志的 `open` 创建一个新的普通文件，`mkdir` 创建一个新目录，`mknod` 创建一个新设备文件。与 `sys_link` 一样，`create` 首先调用 `nameiparent` 来获取父目录的 inode。然后它调用 `dirlookup` 来检查名称是否已经存在 (kernel/sysfile.c:252)。如果名称存在，`create` 的行为取决于它被用于哪个系统调用：`open` 的语义与 `mkdir` 和 `mknod` 不同。如果 `create` 代表 `open`（`type == T_FILE`）使用，并且存在的名称本身是一个普通文件，那么 `open` 将其视为成功，因此 `create` 也将其视为成功 (kernel/sysfile.c:256)。否则，这是一个错误 (kernel/sysfile.c:257-258)。如果名称不存在，`create` 现在使用 `ialloc` 分配一个新的 inode (kernel/sysfile.c:261)。如果新 inode 是一个目录，`create` 使用 `.` 和 `..` 条目初始化它。最后，现在数据已正确初始化，`create` 可以将其链接到父目录中 (kernel/sysfile.c:274)。`Create` 与 `sys_link` 一样，同时持有两个 inode 锁：`ip` 和 `dp`。没有死锁的可能性，因为 inode `ip` 是新分配的：系统中的任何其他进程都不会持有 `ip` 的锁然后试图锁定 `dp`。

使用 `create`，很容易实现 `sys_open`、`sys_mkdir` 和 `sys_mknod`。`Sys_open` (kernel/sysfile.c:287) 是最复杂的，因为创建新文件只是它所能做的一小部分。如果 `open` 传递了 `O_CREATE` 标志，它将调用 `create` (kernel/sysfile.c:301)。否则，它将调用 `namei` (kernel/sysfile.c:307)。`Create` 返回一个锁定的 inode，但 `namei` 不会，因此 `sys_open` 必须自己锁定 inode。这提供了一个方便的地方来检查目录是否仅以只读方式打开，而不是写入。假设以某种方式获得了 inode，`sys_open` 分配一个文件和一个文件描述符 (kernel/sysfile.c:325)，然后填充该文件 (kernel/sysfile.c:337-342)。请注意，没有其他进程可以访问部分初始化的文件，因为它仅在当前进程的表中。

### 8.15 现实世界

真实操作系统中的缓冲区缓存比 xv6 的缓冲区缓存复杂得多，但它服务于相同两个目的：缓存和同步对磁盘的访问。Xv6 的缓冲区缓存，像 V6 的一样，使用简单的**最近最少使用（LRU）** 驱逐策略；还有更多复杂的策略可以实现，每个策略对某些工作负载有好处，对其他工作负载则不然。更高效的 LRU 缓存将消除链表，而是使用哈希表进行查找和使用堆进行 LRU 驱逐。现代缓冲区缓存通常与虚拟内存系统集成以支持内存映射文件。

Xv6 的日志系统效率低下。提交不能与文件系统系统调用并发发生。系统记录整个块，即使一个块中只有几个字节被更改。它执行同步日志写入，一次一个块，每个块可能需要整个磁盘旋转时间。真实的日志系统解决了所有这些问题。

日志不是提供崩溃恢复的唯一方法。早期的文件系统在重启期间使用清理程序（例如，UNIX `fsck` 程序）来检查每个文件和目录以及块和 inode 空闲列表，寻找并解决不一致性。对于大型文件系统，清理可能需要数小时，并且在某些情况下，无法以使原始系统调用原子化的方式解决不一致性。从日志中恢复要快得多，并且使系统调用在面对崩溃时是原子的。

Xv6 使用与早期 UNIX 相同的基本磁盘 inode 和目录布局；这个方案多年来一直非常持久。BSD 的 UFS/FFS 和 Linux 的 ext2/ext3 基本上使用相同的数据结构。文件系统布局中最低效的部分是目录，它在每次查找时需要线性扫描所有磁盘块。当目录只有几个磁盘块时，这是合理的，但对于包含许多文件的目录，这很昂贵。Microsoft Windows 的 NTFS、macOS 的 HFS 和 Solaris 的 ZFS 等等，将目录实现为磁盘上块的平衡树。这很复杂，但保证了对数时间的目录查找。

# 第 9 章

## 并发再探

同时获得良好的并行性能、并发下的正确性以及可理解的代码是内核设计中的一大挑战。直接使用锁是实现正确性的最佳途径，但并不总是可行的。本章重点介绍 xv6 被迫以复杂方式使用锁的例子，以及 xv6 使用类锁技术但不是锁的例子。

### 9.1 锁模式

缓存项通常是锁定的挑战。例如，文件系统的块缓存 (kernel/bio.c:26) 存储多达 `NBUF` 个磁盘块的副本。至关重要的是，一个给定的磁盘块在缓存中最多只能有一个副本；否则，不同的进程可能对应该是同一个块的不同副本进行冲突的更改。每个缓存的块存储在一个 `struct buf` (kernel/buf.h:1) 中。一个 `struct buf` 有一个 `lock` 字段，有助于确保一次只有一个进程使用给定的磁盘块。然而，这个锁还不够：如果一个块根本不在缓存中，并且两个进程同时想要使用它怎么办？没有 `struct buf`（因为块尚未缓存），因此没有东西可以锁定。Xv6 通过将一个额外的锁（`bcache.lock`）与一组缓存块的身份相关联来处理这种情况。需要检查块是否被缓存（例如，`bget` (kernel/bio.c:59)）或更改缓存块集的代码必须持有 `bcache.lock`；在该代码找到它需要的块和 `struct buf` 之后，它可以释放 `bcache.lock` 并仅锁定特定的块。这是一种常见的模式：一个用于项集的锁，加上每个项一个锁。

通常，获取锁的同一函数会释放它。但是，更精确的看待方式是，锁在必须表现为原子的序列开始时被获取，并在该序列结束时被释放。如果序列在不同的函数、不同的线程或不同的 CPU 上开始和结束，那么锁的获取和释放也必须如此。锁的功能是强制其他使用等待，而不是将一块数据固定给特定的代理。一个例子是 `yield` (kernel/proc.c:496) 中的 `acquire`，它在调度器线程中释放，而不是在获取进程。另一个例子是 `ilock` (kernel/fs.c:289) 中的 `acquiresleep`；此代码通常在读取磁盘时睡眠；它可能在不同的 CPU 上醒来，这意味着锁可能在不同的 CPU 上被获取和释放。

释放一个受嵌入在对象中的锁保护的对象是一件微妙的事情，因为拥有锁不足以保证释放是正确的。问题情况发生在其他线程在 `acquire` 中等待使用该对象时；释放该对象会隐式释放嵌入的锁，这将导致等待线程发生故障。一种解决方案是跟踪对对象的引用数量，以便仅在最后一个引用消失时才释放它。参见 `pipeclose` (kernel/pipe.c:59) 的例子；`pi->readopen` 和 `pi->writeopen` 跟踪管道是否有文件描述符指向它。

### 9.2 类锁模式

在许多地方，xv6 以类锁的方式使用引用计数或标志来指示对象已分配且不应被释放或重用。进程的 `p->state` 以此方式作用，`file`、`inode` 和 `buf` 结构中的引用计数也是如此。虽然在每种情况下，锁都保护标志或引用计数，但是后者阻止了对象被过早释放。

文件系统使用 `struct inode` 引用计数作为一种共享锁，可以被多个进程持有，以避免如果代码使用普通锁会发生的死锁。例如，`namex` (kernel/fs.c:629) 中的循环依次锁定每个路径名组件命名的目录。然而，`namex` 必须在循环结束时释放每个锁，因为如果它持有多个锁，并且路径名包含点（例如 `a/./b`），它可能会与自身死锁。它也可能与涉及该目录和 `..` 的并发查找死锁。正如第 8 章所解释的，解决方案是让循环将其引用计数增加（但未锁定）的情况下将目录 inode 带到下一次迭代。

某些数据项在不同时间受到不同机制的保护，并且有时通过 xv6 代码的结构隐式地防止并发访问，而不是通过显式锁。例如，当一个物理页空闲时，它受 `kmem.lock` (kernel/kalloc.c:24) 保护。如果该页面随后被分配为管道 (kernel/pipe.c:23)，它会受到一个不同的锁（嵌入的 `pi->lock`）保护。如果该页面被重新分配给新进程的用户内存，它根本不受锁保护。相反，分配器不会将该页面给任何其他进程（直到它被释放）这一事实保护它免受并发访问。新进程内存的所有权很复杂：首先父进程在 `fork` 中分配和操作它，然后子进程使用它，并且（在子进程退出后）父进程再次拥有内存并将其传递给 `kfree`。这里有两个教训：一个数据对象在其生命周期的不同点可能以不同的方式受到并发的保护，并且保护可能采取隐式结构而非显式锁的形式。

最后一个类锁示例是需要在调用 `mycpu()` (kernel/proc.c:80) 时禁用中断。禁用中断使调用代码相对于可能强制上下文切换的时钟中断成为原子操作，因此将进程移动到不同的 CPU。

### 9.4 并行性

锁定主要是为了正确性而抑制并行性。由于性能也很重要，内核设计者通常不得不考虑如何使用锁来既实现正确性又允许并行性。虽然 xv6 并非为高性能而系统设计，但仍然值得考虑 xv6 的哪些操作可以并行执行，哪些可能会在锁上冲突。

xv6 中的管道是相当好的并行性的一个例子。每个管道都有自己的锁，因此不同的进程可以在不同的 CPU 上并行地读写不同的管道。然而，对于给定的管道，写者和读者必须等待对方释放锁；他们不能同时读/写同一个管道。此外，从空管道读取（或向满管道写入）必须阻塞，但这并非由于锁定方案。

上下文切换是一个更复杂的例子。两个内核线程，每个在其自己的 CPU 上执行，可以同时调用 `yield`、`sched` 和 `swtch`，并且这些调用将并行执行。每个线程都持有一个锁，但它们是不同的锁，因此它们不必互相等待。然而，一旦进入调度器，两个 CPU 在搜索进程表以寻找 `RUNNABLE` 进程时可能会在锁上发生冲突。也就是说，xv6 在上下文切换期间可能会从多个 CPU 获得性能提升，但可能不如它本可以做到的那样多。

另一个例子是不同 CPU 上的不同进程并发调用 `fork`。这些调用可能不得不为了 `pid_lock` 和 `kmem.lock` 以及为了搜索进程表以寻找 `UNUSED` 进程而需要的每进程锁而相互等待。另一方面，两个 `fork` 进程可以完全并行地复制用户内存页和格式化页表页。

上述每个例子中的锁定方案在某些情况下牺牲了并行性能。在每种情况下，都可以通过更精细的设计获得更多的并行性。是否值得取决于细节：相关操作被调用的频率、代码在争用锁上花费的时间、有多少 CPU 可能同时运行冲突的操作，以及其他部分的代码是否是更具限制性的瓶颈。可能很难

## 总结

本文通过逐行研究一个操作系统 xv6，介绍了操作系统的主要思想。一些代码行体现了主要思想的本质（例如，上下文切换、用户/内核边界、锁等），每一行都很重要；其他代码行说明了如何实现特定操作系统思想的例子，并且可以很容易地以不同的方式完成（例如，更好的调度算法、更好的磁盘数据结构来表示文件、更好的日志记录以允许并发事务等）。所有这些思想都是在一个特定的、非常成功的系统调用接口（Unix 接口）的背景下说明的，但这些思想可以推广到其他操作系统的设计中。
