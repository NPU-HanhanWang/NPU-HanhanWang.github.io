# 第一部分：缓存基本原理

## 1.1 局部性原理回顾

**局部性原理**：程序倾向于使用与最近使用过的地址相近或相同的地址。

### 时间局部性（Temporal Locality）
- 最近引用过的数据项很可能在不久的将来再次被引用
- 例如：循环中的计数器变量 `i`

### 空间局部性（Spatial Locality）
- 地址相近的数据项倾向于在相近的时间被引用
- 例如：顺序遍历数组元素

> **意义**：局部性是缓存能够大幅提升性能的根本原因，程序利用局部性可以显著减少慢速内存访问。

---

## 1.2 通用缓存概念

### 缓存的基本工作流程

```
                    CPU请求数据
                        ↓
                ┌───────────────┐
                │ 在缓存中查找？  │
                └───────────────┘
                   ↓        ↓
                  是        否
                   ↓        ↓
              【缓存命中】 【缓存未命中】
                   ↓        ↓
              直接返回数据  从内存加载数据块
                          到缓存中
```

### 三种缓存未命中类型

| 类型                                          | 定义                                                                       | 能否避免                         |
| --------------------------------------------- | -------------------------------------------------------------------------- | -------------------------------- |
| **冷启动/强制未命中（Cold/Compulsory Miss）** | 首次访问某数据块时必然发生的未命中。即使无限大缓存也无法避免。             | 无法避免                         |
| **容量未命中（Capacity Miss）**               | 活动缓存块集合大于缓存容量时发生的未命中。缓存足够大但无法容纳所有工作集。 | 通过增加缓存大小或减小工作集     |
| **冲突未命中（Conflict Miss）**               | 缓存大小足够，但太多数据对象映射到同一组导致的未命中。由具体放置策略引起。 | 通过改进映射策略（如提高关联度） |

<details>
<summary><b>💡 理解要点：</b></summary>

- 冷启动未命中：程序刚开始运行时的必然代价
- 容量未命中：工作集（工作集指程序在一段时间内频繁访问的内存数据集合）超出缓存容量时的代价
- 冲突未命中：映射策略导致的"人为"未命中，可以通过设计缓解
</details>

## 1.3 通用缓存组织（S, E, B）

```
    ┌─────────────────────────────────────────────────────┐
    │                   缓存整体结构                        │
    │  ┌─────────────────────────────────────────────┐    │
    │  │              缓存大小 C = S × E × B         │    │
    │  └─────────────────────────────────────────────┘    │
    │                                                     │
    │  S = 组数（Sets）                                    │
    │  E = 每组的行数（Lines per set）/ 关联度             │
    │  B = 块大小（Block size in bytes）                  │
    │                                                     │
    │  内存地址划分：                                     │
    │  ┌──────┬─────────┬─────────────┐                   │
    │  │ Tag  │ Set索引  │ 块内偏移    │                   │
    │  │ t位  │ s位      │ b位        │                   │
    │  └──────┴─────────┴─────────────┘                   │
    │                                                     │
    │  缓存行（Cache Line）结构：                         │
    │  ┌──────────┬────────┬────────────┐                 │
    │  │ Valid位  │ Tag   │ 数据块(B字节)│                 │
    │  │  1 bit   │ t bits │  B bytes   │                 │
    │  └──────────┴────────┴────────────┘                 │
    └─────────────────────────────────────────────────────┘
```

### 各部分说明
- **Tag（标记位）**：用于判断当前块是否包含请求的数据，与地址高位比较
- **Set Index（组索引）**：决定数据映射到哪一组
- **Offset（块内偏移）**：确定数据在块内的位置
- **Valid Bit（有效位）**：指示该行是否包含有效数据
- **Dirty Bit（脏位）**：用于写回策略，标记数据是否被修改过

### 组索引的作用
组索引是从内存地址中提取的中间位（详见补充材料），用于快速定位数据可能所在的缓存组。采用中间位作为索引可以利用空间局部性，避免连续地址映射到同一组导致的冲突。

---

## 1.4 缓存读取流程

1. 从地址中提取 **组索引（s位）** 定位到具体组
2. 从地址中提取 **标记（t位）**
3. 在组内并行查找：
   - 检查所有行的有效位
   - 比较标记位是否匹配
   - 若匹配，则 **缓存命中**，从块中偏移位置取出数据
4. 若无匹配行，则 **缓存未命中**：
   - 若组内有空行，填充新块
   - 若无空行，按替换策略驱逐一行

> **关键点**：组相联缓存中，缓存未命中时只需检查特定组，无需遍历整个缓存，查找效率极高。

---

## 1.5 直接映射缓存（E = 1）

### 结构特点
- 每组只有 **1行**（E = 1）
- 每个内存块只能映射到唯一一个缓存位置
- 实现简单，但冲突未命中率高

### 地址映射示例

假设参数：S = 4组，E = 1行/组，B = 2字节/块，地址空间 M = 16字节（4位地址）

```
地址位划分（4位地址）：
┌──────┬─────────┬─────────────┐
│ Tag  │ Set索引  │ 块内偏移    │
│ 2位  │ 2位     │ 0位         │
└──────┴─────────┴─────────────┘
注意：由于 B=2，b=1，但示例中地址位4位，块内偏移1位，索引1位，标签2位
```

**地址跟踪示例**（读操作，每次1字节）：

| 地址 | 二进制 | 操作 | 结果   | 说明                  |
| ---- | ------ | ---- | ------ | --------------------- |
| 0    | 0000   | 读   | 未命中 | 冷启动，加载块 M[0-1] |
| 1    | 0001   | 读   | 命中   | 已在缓存中            |
| 7    | 0111   | 读   | 未命中 | 加载块 M[6-7]         |
| 8    | 1000   | 读   | 未命中 | 加载块 M[8-9]         |
| 0    | 0000   | 读   | 未命中 | 原来的块已被替换      |

**缓存状态变化**：

```
初始状态（所有行无效）：
Set 0: [V=0] [Tag=--] [数据=--]
Set 1: [V=0] [Tag=--] [数据=--]
Set 2: [V=0] [Tag=--] [数据=--]
Set 3: [V=0] [Tag=--] [数据=--]

读地址0后：
Set 0: [V=1] [Tag=00] [数据=M[0-1]]

读地址7后（地址0111，Set索引=1，Tag=01）：
Set 1: [V=1] [Tag=01] [数据=M[6-7]]

读地址8后（地址1000，Set索引=0，Tag=10）：
Set 0: [V=1] [Tag=10] [数据=M[8-9]]  ← 覆盖了原来的M[0-1]

读地址0后（地址0000，Set索引=0，Tag=00）：
Set 0: [V=1] [Tag=00] [数据=M[0-1]]  ← 再次覆盖
```

## 1.6 E路组相联缓存（E = 2示例）

### 结构特点
- 每组有 **E行**（E ≥ 2）
- 每个内存块可以映射到组内任意一行
- 通过增加关联度减少冲突未命中

### 组相联缓存模拟

参数：S = 2组，E = 2行/组，B = 2字节/块，4位地址（t=2, s=1, b=1）

```
地址位划分：
┌──────┬─────────┬─────────────┐
│ Tag  │ Set索引  │ 块内偏移    │
│ 2位  │ 1位     │ 1位         │
└──────┴─────────┴─────────────┘
```

**地址跟踪**：

| 地址 | 二进制 | Set索引 | Tag | 操作 | 结果   |
| ---- | ------ | ------- | --- | ---- | ------ |
| 0    | 0000   | 0       | 00  | 读   | 未命中 |
| 1    | 0001   | 0       | 00  | 读   | 命中   |
| 7    | 0111   | 1       | 01  | 读   | 未命中 |
| 8    | 1000   | 0       | 10  | 读   | 未命中 |
| 0    | 0000   | 0       | 00  | 读   | 命中   |

**关键点**：由于 E=2，Set 0 可以同时保存 Tag=00 和 Tag=10 的两块数据，避免了直接映射中的冲突替换！

```
Set 0: [V=1 Tag=00 数据=M[0-1]] [V=1 Tag=10 数据=M[8-9]]
Set 1: [V=1 Tag=01 数据=M[6-7]] [V=0 Tag=-- 数据=--]
```

## 1.7 替换策略

当组内所有行都被占用时，需要选择一行被驱逐：

| 策略                    | 描述                     | 优缺点                       |
| ----------------------- | ------------------------ | ---------------------------- |
| **LRU（最近最少使用）** | 替换最长时间未被访问的行 | 效果好，但硬件开销大         |
| **随机**                | 随机选择一行替换         | 实现简单，效果尚可           |
| **FIFO（先进先出）**    | 替换最早加载的行         | 实现简单，但可能替换常用数据 |

## 1.8 写操作处理

### 写操作的核心问题
- 数据在多个层级存在副本：L1、L2、L3、主存
- 需要保证数据一致性

### 写命中策略

| 策略                        | 描述                 | 特点                                   |
| --------------------------- | -------------------- | -------------------------------------- |
| **Write-Through（写直达）** | 立即写入内存         | 简单，但带宽消耗大                     |
| **Write-Back（写回）**      | 延迟到被替换时才写回 | 带宽效率高，需要 **脏位（Dirty Bit）** |

### 写未命中策略

| 策略                              | 描述                                 |
| --------------------------------- | ------------------------------------ |
| **Write-Allocate（写分配）**      | 加载到缓存后更新，适合后续有多次写入 |
| **No-Write-Allocate（非写分配）** | 直接写入内存，不加载到缓存           |

### 典型组合

1. **Write-Through + No-Write-Allocate**（简单，易实现）
2. **Write-Back + Write-Allocate**（高性能，现代处理器常用）

### Write-Back + Write-Allocate 工作流程

```
    写操作请求
        ↓
┌───────────────────┐
│  地址在缓存中？    │
└───────────────────┘
    ↓ 是           ↓ 否
【写命中】      【写未命中】
    ↓              ↓
更新块数据    从内存读取块（读未命中处理）
设置脏位=1        ↓
    ↓         执行写命中操作
    完成          ↓
             设置脏位=1

    【被替换时】
        ↓
   检查脏位
    ↓是     ↓否
写回内存  直接丢弃
清除脏位
```

## 1.10 缓存性能指标

### 关键指标

| 指标           | 定义                                   | 典型值                             |
| -------------- | -------------------------------------- | ---------------------------------- |
| **未命中率**   | 未命中次数/总访问次数 = 1 - 命中率     | L1: 3-10%，L2: < 1%                |
| **命中时间**   | 从缓存获取数据的时间（含判断是否命中） | L1: 4周期，L2: 10周期              |
| **未命中惩罚** | 未命中导致的额外时间开销               | 主存: 50-200周期（且趋势在增加！） |

### 性能差异的惊人影响

```
示例：缓存命中时间 = 1周期，未命中惩罚 = 100周期

情况1：97% 命中率
平均访问时间 = 1 + 0.03 × 100 = 4周期

情况2：99% 命中率
平均访问时间 = 1 + 0.01 × 100 = 2周期

结论：99% 命中率是 97% 的 2 倍性能！
这解释了为什么未命中率比命中率更受关注
```

> **核心观点**：未命中率的小幅改善（从3%降到1%）能让性能翻倍。优化缓存命中率是性能优化的关键。

# 第二部分：缓存友好代码

## 2.1 编写缓存友好代码的原则

### 核心准则

1. **让常见情况快起来**（Make the common case go fast）
   - 重点关注核心函数的内层循环

2. **最小化内层循环的未命中次数**
   - ✅ 重复引用变量 → 好的时间局部性
   - ✅ 步长1的引用模式 → 好的空间局部性

> 关键思想：我们对局部性的定性理解通过缓存内存的定量分析得到量化。

---

## 2.2 内存山（Memory Mountain）

### 什么是内存山？

**读吞吐量/带宽**：每秒从内存读取的字节数（MB/s）

**内存山**：衡量读吞吐量随空间和时间局部性变化的3D图，用于表征内存系统性能。

### 测试函数原理

```c
long data[MAXELEMS];    // 全局数组

// test - 以指定步长遍历数组前 elems 个元素
int test(int elems, int stride) {
    long i, sx2 = stride*2, sx3 = stride*3, sx4 = stride*4;
    long acc0 = 0, acc1 = 0, acc2 = 0, acc3 = 0;
    long length = elems, limit = length - sx4;

    // 4x4 循环展开：每次处理4个元素
    for (i = 0; i < limit; i += sx4) {
        acc0 += data[i];
        acc1 += data[i + stride];
        acc2 += data[i + sx2];
        acc3 += data[i + sx3];
    }
    // 处理剩余元素
    for (; i < length; i++) {
        acc0 += data[i];
    }
    return ((acc0 + acc1) + (acc2 + acc3));
}
```

### 内存山揭示的规律

- **小工作集（高时间局部性）**：数据在L1缓存中 → 最高吞吐量
- **中等工作集**：数据在L2/L3缓存中 → 中等吞吐量
- **大工作集**：数据在主存中 → 最低吞吐量
- **小步长（高空间局部性）**：充分利用缓存块 → 高吞吐量
- **大步长**：跨越多个缓存块 → 低吞吐量

# 第三部分：循环重排与空间局部性

## 3.1 矩阵乘法回顾

```
C[i][j] = A[i][0] × B[0][j] + A[i][1] × B[1][j] + ... + A[i][n-1] × B[n-1][j]

即：C[i][j] = Σ(k=0 到 n-1) A[i][k] × B[k][j]
```

### C数组的行主序存储

```
行主序布局（Row-Major Order）：
┌─────────────────────────────────────────────┐
│ A[0][0] A[0][1] ... A[0][N-1] A[1][0] ... │
└─────────────────────────────────────────────┘

遍历同一行的列（步长1）：
for (i = 0; i < N; i++) sum += a[0][i];
✅ 好的空间局部性
未命中率 = sizeof(元素) / B

遍历同一列的行（步长N）：
for (i = 0; i < M; i++) sum += a[i][0];
❌ 无空间局部性
未命中率 = 1（100%）
```

## 3.2 不同循环顺序的未命中率分析

**假设条件**：
- 块大小 = 32B（可容纳4个double）
- 矩阵维度 N 非常大（1/N ≈ 0）
- 缓存不足以容纳多行

### 实现1：ijk（i-j-k顺序）

```c
for (i = 0; i < n; i++) {
    for (j = 0; j < n; j++) {
        for (k = 0; k < n; k++) {
            C[i][j] += A[i][k] * B[k][j];
        }
    }
}
```

**内层循环模式**：
- **A**：按行访问，步长1 → ✅ 空间局部性好（每个块4个double，未命中率0.25）
- **B**：按列访问，步长n → ❌ 无空间局部性（未命中率1.0）
- **C**：固定元素 → ✅ 时间局部性好

**总未命中率**：1.25（每次内层迭代）

### 实现2：kij（k-i-j顺序）

```c
for (k = 0; k < n; k++) {
    for (i = 0; i < n; i++) {
        r = A[i][k];
        for (j = 0; j < n; j++) {
            C[i][j] += r * B[k][j];
        }
    }
}
```

**内层循环模式**：
- **B**：按行访问，步长1 → ✅ 空间局部性好（未命中率0.25）
- **C**：按行访问，步长1 → ✅ 空间局部性好（未命中率0.25）
- **A**：固定元素 → ✅ 时间局部性好

**总未命中率**：0.5（每次内层迭代）⭐ 最优

### 实现3：jki（j-k-i顺序）

```c
for (j = 0; j < n; j++) {
    for (k = 0; k < n; k++) {
        r = B[k][j];
        for (i = 0; i < n; i++) {
            C[i][j] += A[i][k] * r;
        }
    }
}
```

**内层循环模式**：
- **C**：按列访问，步长n → ❌ 无空间局部性（未命中率1.0）
- **A**：按列访问，步长n → ❌ 无空间局部性（未命中率1.0）
- **B**：固定元素 → ✅ 时间局部性好

**总未命中率**：2.0（最差）

## 3.3 六种循环顺序总结

| 顺序      | 加载/存储    | 每次迭代平均未命中 | 性能排名 |
| --------- | ------------ | ------------------ | -------- |
| ijk (jik) | 2加载, 0存储 | 1.25               | 🟡 中等   |
| kij (ikj) | 2加载, 1存储 | 0.5                | 🟢 最优   |
| jki (kji) | 2加载, 1存储 | 2.0                | 🔴 最差   |

**结论**：循环顺序对性能影响巨大！选择正确的循环顺序可以显著提高缓存利用率。

# 第四部分：分块技术（Blocking）

## 4.1 问题：朴素矩阵乘法的缓存表现

### 朴素实现

```c
void mmm(double* a, double* b, double* c, int n) {
    int i, j, k;
    for (i = 0; i < n; i++)
        for (j = 0; j < n; j++)
            for (k = 0; k < n; k++)
                c[i*n + j] += a[i*n + k] * b[k*n + j];
}
```

### 未命中分析

**假设**：
- 矩阵元素为 double（8字节）
- 缓存块 = 8个double
- 缓存大小 C << n（远小于矩阵维度）

**第一次迭代**（i=0, j=0）：
- 读取 A 的第0行：n/8 次未命中（空间局部性）
- 读取 B 的第0列：n 次未命中（每次访问不同块）
- 总计：9n/8 次未命中

**每次迭代**：9n/8 次未命中

**总未命中**：(9n/8) × n² = (9/8)n³ ⚠️ 过高！

### 根本问题
- 每对 (i, j) 都需要完整扫描B的一列
- B 的一列在内存中不连续，跨越 n 个不同的缓存块
- 即使重用 A 的行，B 的列访问模式破坏空间局部性

---

## 4.2 分块矩阵乘法

### 核心思想
将矩阵分成 **B × B** 的子块，每次只处理一个子块矩阵乘法，确保三个子块（A块、B块、C块）都能放入缓存。

### 分块实现

```c
void mmm(double* a, double* b, double* c, int n) {
    int i, j, k, i1, j1, k1;

    // 分块循环
    for (i = 0; i < n; i += B)
        for (j = 0; j < n; j += B)
            for (k = 0; k < n; k += B)
                // B × B 小矩阵乘法
                for (i1 = i; i1 < i + B; i1++)
                    for (j1 = j; j1 < j + B; j1++)
                        for (k1 = k; k1 < k + B; k1++)
                            c[i1*n + j1] += a[i1*n + k1] * b[k1*n + j1];
}
```

> **注意**：这里的 B 是分块大小，不是缓存块大小！

### 分块示意图

```
    ┌─────────────┐
    │  k方向分块   │
    │  ┌──┬──┬──┐  │
    │  │  │  │  │  │
    │  ├──┼──┼──┤  │
    │  │  │  │  │  │  ← i方向分块
    │  ├──┼──┼──┤  │
    │  │  │  │  │  │
    │  └──┴──┴──┘  │
    └─────────────┘
        ↑
      j方向分块
```

## 4.3 分块未命中分析

**假设**：
- 缓存块 = 8个double
- 缓存大小 C << n
- 三个块能同时放入缓存：3B² < C

### 第一次分块迭代（单个B×B块）

- A块未命中：B²/8（按行访问，空间局部性）
- B块未命中：B²/8（按行访问，空间局部性）
- C块未命中：B²/8（按行访问，空间局部性）
- 总计：3B²/8 次未命中

### 整体计算

- 每个B×B块需 2n/B × B²/8 = nB/4 次未命中（忽略C块）
- 总共有 (n/B)² 个块
- **总未命中**：nB/4 × (n/B)² = n³/(4B)

### 性能对比

| 方法     | 未命中次数 | 加速比             |
| -------- | ---------- | ------------------ |
| 朴素实现 | (9/8)n³    | 1×                 |
| 分块实现 | n³/(4B)    | (9/8) × 4B = 4.5B× |

### 最优分块大小
选择最大的 B，满足 **3B² < C**（三个块能同时放入缓存）

### 为什么分块如此有效？

矩阵乘法的本质具有内在的时间局部性：
- 输入数据：3n²
- 计算量：2n³
- **每个数组元素被使用 O(n) 次！**

但程序必须被正确编写才能利用这一特性。分块技术通过重组计算顺序，确保数据在被缓存期间被充分利用。

## B. 为什么使用中间位作为组索引？

### 中间位索引

```
地址格式（中间位索引）：
┌──────────────┬─────────────┬──────────────┐
│   Tag (TT)   │  Set Index  │   Offset     │
│   高位       │   (SS)      │   (BB)       │
└──────────────┴─────────────┴──────────────┘
```

**优点**：相邻地址映射到不同组，充分利用空间局部性

### 高位索引（对比）

```
地址格式（高位索引）：
┌──────────────┬─────────────┬──────────────┐
│  Set Index   │   Tag (TT)  │   Offset     │
│   (SS)       │   高位      │   (BB)       │
└──────────────┴─────────────┴──────────────┘
```

**缺点**：连续地址映射到同一组，导致大量冲突未命中

### 实际例子：Core i7 L1 数据缓存

```
参数：
- 大小：32KB
- 关联度：8路组相联
- 块大小：64B
- 地址范围：47位

计算：
- B（块大小）= 64 → b = 6位
- S（组数）= 32KB / (64 × 8) = 64 → s = 6位
- E（每组合数）= 8 → e = 3位
- t（标记位数）= 47 - 6 - 6 = 35位

地址示例：0x00007f7262a1e010
- 块偏移：0x10（低6位）
- 组索引：0x0（接下来6位）
- 标记：0x7f7262a1e（剩余35位）
```

## D. 矩阵乘法循环顺序详细分析

### ikj 实现

```c
/* ikj */
for (i = 0; i < n; i++) {
    for (k = 0; k < n; k++) {
        r = a[i][k];
        for (j = 0; j < n; j++)
            c[i][j] += r * b[k][j];
    }
}
```

**内层循环未命中率**：
- A：1.0（固定元素，每次访问）
- B：0.0（已缓存）
- C：0.25（空间局部性）

**总计**：1.25

### jki 实现

```c
/* jki */
for (j = 0; j < n; j++) {
    for (k = 0; k < n; k++) {
        r = b[k][j];
        for (i = 0; i < n; i++)
            c[i][j] += a[i][k] * r;
    }
}
```

**内层循环未命中率**：
- A：1.0（步长n访问）
- B：0.0（固定元素）
- C：1.0（步长n访问）

**总计**：2.0

### 为什么 kij 最好？

```
kij 内层循环：
for (j = 0; j < n; j++)
    C[i][j] += A[i][k] * B[k][j];
              ↑          ↑
         固定元素    连续访问
         (时间局部性) (空间局部性)
```

- A 的元素在寄存器中保持不变
- B 和 C 都在按行连续访问
- 三个矩阵中两个有好的空间局部性，一个有好的时间局部性
- 完美匹配缓存的特性

## E. 常见面试/考试问题

<details>
<summary><b>Q1: 为什么 99% 命中率比 97% 命中率性能好一倍？</b></summary>

假设命中时间 = 1周期，未命中惩罚 = 100周期：
- 97%命中：平均访问时间 = 1 + 0.03×100 = 4周期
- 99%命中：平均访问时间 = 1 + 0.01×100 = 2周期

未命中率降低66%（从3%到1%），性能翻倍。
</details>

<details>
<summary><b>Q2: 分块技术为什么能提高缓存性能？</b></summary>

分块通过将大矩阵乘法分解为小矩阵乘法，确保工作集（三个子块）能同时放入缓存。这样，数据在被替换前被多次使用（O(n)次），大大提高了时间局部性。朴素实现中，B矩阵的列访问破坏了空间局部性，每次迭代都产生未命中。
</details>

<details>
<summary><b>Q3: Write-Back 和 Write-Through 各有什么优缺点？</b></summary>

**Write-Through**：
- ✅ 简单，数据一致性容易保证
- ❌ 每次写操作都访问内存，带宽消耗大

**Write-Back**：
- ✅ 减少内存写操作，带宽效率高
- ❌ 需要脏位，实现更复杂，替换时需要检查并写回
</details>

<details>
<summary><b>Q4: 如何选择最优的分块大小？</b></summary>

选择最大满足 `3B² < C` 的 B，其中 C 是缓存大小。同时要考虑：
- 所有层次的缓存（L1、L2、L3）
- 实际应用中可能需要在不同层次间权衡
- 过大B会导致容量未命中，过小B则分块开销增加
</details>
