## 一、从自动补全开始

### 试试看，你能补全这些句子吗？

- Tepper School of **Business**
- College of Fine **Arts**
- Don't forget to tie your **shoes**
- Put the fries in the **bag**
- Have a good **day / night / evening / flight / trip...**

### 关键洞察

如果你自信地回答了"bag"，可能是因为你见过特定的网络迷因——但**不是每个人都见过**。而"Have a good ___"则有很多可能，你通常会选择**最可能的词**。

> 📱 **你的手机自动补全做的就是同一件事**：基于经验预测最可能的词。

## 二、最简模型：基于前一个词的预测

### 实现方式

阅读大量文本，统计词A后面跟着词B的次数：

```python
counts = {
    "fuzzy": {"bunny": 1402, "sock": 349, "bread": 78, "phone": 1},
    "potato": {"chip": 1809, "bread": 802, "bug": 283, "steven": 23},
}
```

然后将计数转换为概率：
- 给定"potato"，出现"chip"的概率很高
- 给定"potato"，出现"steven"的概率极低

> 📈 更多示例文本意味着更好的预测——但这种方法永远不会完美。

## 三、词嵌入（Embeddings）

### 什么是嵌入？

将单词的**含义和上下文**数学表示为**向量**：

```python
embeddings = {
    "cat":     [0.9, 0.1, 0.8, 0.0, ...],
    "dog":     [0.9, 0.1, 0.7, 0.0, ...],
    "apple":   [0.1, 0.0, 0.1, 0.9, ...],
    "student": [0.2, 0.9, 0.3, 0.1, ...],
}
```

- 也许向量的第0维表示"动物性程度"
- 第1维表示"知识性程度"

### 关键特性

在N维向量空间中，**相似单词在空间中的距离很近**——即使不是真正的同义词（如"dog"和"cat"）。

### 向量的加减运算

一个好的嵌入模型可以实现直观的向量运算：

```
king - man + woman ≈ queen
paris - france + italy ≈ rome
```

> 🔧 嵌入是在**海量数据**上训练出来的！

## 四、注意力机制（Attention）

### 从看一个词到看多个词

与其只基于最后一个词预测下一个词，不如**基于最后N个词**进行预测。

- 嵌入技术让这变得可行——不需要在训练数据中找到精确的词匹配

### 注意力的作用

**注意力决定句子中哪些词对彼此最重要。**

例子：
> "I mention a **king sized mattress**"

"size"和"mattress"的存在告诉我，这里的"king"很可能**不是在谈论皇室**。注意力机制会用这个信息来更新句子中"king"的向量。

> 💡 同时，"a"在这里几乎不贡献任何意义。

## 五、LLM：大型语言模型

### 目标

给定前面的文本，**预测下一段文本**。

### 三步流程

```
步骤1：将文本转换为嵌入（向量）
         ↓
步骤2：使用Transformer，基于注意力和上下文更新嵌入
         ↓
    （重复多次，不断丰富嵌入）
         ↓
步骤3：基于更新后的嵌入预测下一个文本
```

> 🧠 这些步骤中的许多环节使用**神经网络**从训练数据中学习复杂模式。

## 六、从自动补全到对话

### 问题

自动补全模型**不是对话系统**。

### 解决方案

将自动补全转变为聊天机器人需要：
- 施加额外的结构
- 进行**微调（Fine-tuning）**

### 关键概念：RLHF

> **Reinforcement Learning from Human Feedback（基于人类反馈的强化学习）**

这是将基础语言模型转变为有用对话助手的核心技术之一。



## 七、需要澄清的简化

以上是**简化版解释**。实际情况更复杂：

| 简化        | 现实                                      |
| ----------- | ----------------------------------------- |
| LLM处理单词 | LLM处理**token**（不等于单词）            |
| 基础预测    | 有 护栏 增加操作复杂性                    |
| 纯文本生成  | **Agent**使用工具获取上下文和操作其他系统 |
| 固定流程    | 新模型在训练和运行中有额外步骤            |

> 但这个解释应该让这一切**少一点魔法感**。

---

## 八、可解释性问题

### 核心困境

即使这些系统不是魔法，**它们的行为也不是严格可解释的**。

| 组件                    | 问题                                                             |
| ----------------------- | ---------------------------------------------------------------- |
| 嵌入                    | 是**学习出来的**，不是设计出来的。向量维度不一定对应可描述的概念 |
| Transformer增强后的嵌入 | 对人类来说更难理解                                               |
| 神经网络                | 由无数权重组成，自动调整以捕捉我们可能注意不到的模式             |

### 问题的本质

- 如果AI给出的回答**对我们来说有意义** → 我们可以相信它找到了指示合理回答的模式，但**无法窥视LLM内部发现这些模式是什么**
- 如果回答**没有意义** → 这就是**幻觉（Hallucination）**
- 让AI解释自己 → 最多只能给出**似乎合理的解释的幻觉**，而非真正的解释

## 九、实际案例：用AI Agent团队做游戏

### 实验设置

用Claude Agent团队制作角色扮演游戏：
- 第一版在课程结束时才完成
- 两个提示就用完了Claude Pro额度：
  - 30%用于构建团队
  - 70%用于初步尝试创建游戏

### 结果评估

**"闪亮但不好"**：

| 方面     | 评价                                           |
| -------- | ---------------------------------------------- |
| 标题画面 | ✅ 不错                                         |
| 故事叙述 | 大量内容，但角色名直接从老游戏《梦幻之星》复制 |
| 游戏玩法 | 无自由 roaming；只有一场战斗，狂按Enter就能赢  |
| 代码质量 | 生成了大量未使用的代码                         |

### 教训

> 作者又做了三次简单的"修bug并继续"提示来完成演示。**每次提示立即耗尽5小时的Claude额度限制**。

最终结果：
- 技术上是游戏，从"AI现在能创建基础游戏"的角度令人印象深刻
- 但**不会用有趣、巧妙或设计良好**来形容

### 结论

> **"最多只能称之为'形状像游戏'。是对真正工艺的拙劣模仿。"**
>
> 在这种复杂程度下，即使这些强大工具也难以做出好东西——**除非我们带来自己在规划、深思熟虑和创造力方面的技能。**

## 十、关键要点

1. **LLM本质上是高级自动补全**——在巨大规模上预测下一个token
2. **嵌入**让模型理解词与词之间的语义相似性
3. **注意力机制**让模型理解上下文中的重要性关系
4. **RLHF**将基础模型转变为有用的对话助手
5. **可解释性是根本挑战**——我们无法真正知道模型为什么给出特定回答
6. **当前AI的能力边界**：在复杂任务上仍需要人类的规划、思考和创造力
