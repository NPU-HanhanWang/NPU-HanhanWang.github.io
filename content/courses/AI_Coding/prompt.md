# AI提示词工程综合指南

## 文档说明

本文档整合了三份权威来源的提示词工程最佳实践：

- **Claude Opus 4.8 专项指南**（Anthropic）
- **Gemini API 提示设计策略**（Google）
- **MIT Sloan 课程精要**（学术视角）

> 📌 **核心理念**：提示词工程的本质不是寻找“魔法咒语”，而是学会清晰表达你的问题和期望。

---

## 第一部分：核心原则

### 1.1 黄金法则

> **把提示词给一个对任务只有最少背景知识的同事看。如果他们感到困惑，AI也会困惑。**

将AI视为一个**聪明但缺乏上下文的实习生**——它需要明确的方向，不会自动填补信息空白。

### 1.2 提示词工程的本质

| 概念           | 说明                                   |
| -------------- | -------------------------------------- |
| **提示词工程** | 选择合适词汇、句式、标点来优化文本输入 |
| **问题定义**   | 界定问题的焦点、范围、边界             |

> 💡 **长期建议**：培养清晰定义问题和目标的能力，可能比掌握提示词技巧更重要。

### 1.3 迭代思维

> 提示工程是**迭代的**。提供的策略和模板只是起点，需要根据具体使用场景和观察到的模型回答进行实验和优化。

## 第二部分：三大核心策略

### 策略一：提供充足的上下文

**原则**：背景信息越多，响应越精准。

| ❌ 错误做法                                                          | ✅ 正确做法                                                                                                           |
| ------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| "What's the best time of year to enjoy New England's fall foliage?" | "你是一位专攻树木的资深野生生物学家。基于美国近期天气模式，预测新英格兰的最佳秋叶季——并用幼儿园孩子能懂的方式解释。" |
| 通用回答（重启路由器、检查连接...）                                 | "检查以太网线是否已连接到路由器和调制解调器..."（设备特定指导）                                                      |

**进阶技巧**：如果想AI模仿你的写作风格，提供你的写作样本作为上下文。

### 策略二：具体明确

**原则**：输入的精细度与输出的有用性成正比。

| ❌ 模糊提问                      | ✅ 具体提问                                     |
| ------------------------------- | ---------------------------------------------- |
| "Tell me about climate change." | "讨论未来十年气候变化对发展中国家的经济影响。" |

**具体化要素清单**：
- 指定时间范围（如“未来十年”）
- 限定地理区域（如“发展中国家”）
- 明确分析角度（如“经济影响”）
- 提供示例格式
- 列出约束条件

### 策略三：延续对话

基于聊天的AI能**记住对话历史**，无需重复上下文。

```
第1轮：用户提供上下文 + 提问 → AI生成响应
第2轮：用户"make it funnier" → AI生成新版本
第3轮：用户"explain it to college English majors" → AI生成学术化版本
```

⚠️ **注意**：切换话题时**建议开启新会话**，避免上下文干扰。

## 第三部分：提示结构组织

### 3.1 使用XML标签

```xml
<instructions>
  [指令内容]
</instructions>

<context>
  [背景信息]
</context>

<examples>
  <example>示例1</example>
  <example>示例2</example>
</examples>

<input>
  [变量输入]
</input>
```

**最佳实践**：
- 使用一致、描述性的标签名
- 内容有层级关系时嵌套标签

### 3.2 Gemini结构化模板

```xml
<role>
You are a helpful assistant.
</role>

<constraints>
1. Be objective.
2. Cite sources.
</constraints>

<context>
[用户输入 - 模型知道这是数据而非指令]
</context>

<task>
[具体用户请求]
</task>
```

### 3.3 长上下文结构

> ✅ **最佳实践**：将查询放在文档末尾可提升高达**30%**的响应质量

```
[大量上下文内容...]

根据上述信息，请回答：[具体问题]
```

## 第四部分：示例策略（Few-shot vs Zero-shot）

### 4.1 对比

| 类型                    | 定义             | 适用场景           | 示例数量 |
| ----------------------- | ---------------- | ------------------ | -------- |
| **零样本（Zero-shot）** | 无示例，仅指令   | 简单任务、通用查询 | 0        |
| **少样本（Few-shot）**  | 提供示例示范模式 | 格式控制、风格模仿 | 3-5个    |

### 4.2 效果对比

**零样本**（选择较长解释）：
```
Q: How is snow formed?
Explanation1: [详细解释...]
Explanation2: "Water vapor freezes..."
Answer: Explanation1 ← 选择了较长的
```

**少样本**（偏好简洁）：
```
示例1: Q: Why is sky blue? → Answer: Explanation2 (短)
示例2: Q: What causes earthquakes? → Answer: Explanation1 (短)

Now answer: How is snow formed?
Answer: Explanation2 ← 现在选择了较短的
```

### 4.3 最佳实践

✅ 使用**3-5个**具体、多样的示例
✅ 保持示例格式一致（XML标记、空格、换行符）
✅ 单个示例用`<example>`，多个示例放`<examples>`
⚠️ 太多示例可能导致过拟合

## 第五部分：输出格式控制

### 5.1 控制冗长度

> **核心技巧**：告诉AI该做什么，而非不该做什么

| ❌ 错误示例                     | ✅ 正确示例                         |
| ------------------------------ | ---------------------------------- |
| “不要在你的回复中使用Markdown” | “你的回复应由流畅的散文段落组成。” |

**Claude Opus 4.8行为特征**：
- 简单查询 → 更短的答案
- 开放式分析 → 更长的答案
- 动态校准响应长度

### 5.2 补全策略格式化

通过**提供开头模式**引导模型格式：

```
提示：
Create an outline for an essay about hummingbirds.
I. Introduction
*

模型回答：
I. Introduction
* Hook: Start with a captivating fact...
* Background: Briefly introduce hummingbirds...
```

### 5.3 最小化Markdown

```
在撰写报告、文档、技术说明、分析或任何长篇内容时，
使用完整的段落和句子，以清晰流畅的散文形式呈现。

除非符合以下情况，否则不要使用有序列表或无序列表：
a) 呈现真正离散的项目，且列表格式最为合适
b) 用户明确要求使用列表
```

## 第六部分：模型特定配置

### 6.1 Claude Effort参数

| 级别     | 适用场景         | 推荐度           |
| -------- | ---------------- | ---------------- |
| `xhigh`  | 编码、Agent任务  | ✅ 推荐默认       |
| `high`   | 智能敏感任务     | ⚠️ 最低推荐       |
| `medium` | 成本敏感场景     |                  |
| `low`    | 短任务、延迟敏感 | ❌ 有思考不足风险 |

> ⚠️ **关键提醒**：遇到复杂问题思考不足时，**优先提高effort**而非修改提示词。

### 6.2 Claude自适应思维

```python
message = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=64000,
    thinking={"type": "adaptive"},
    output_config={"effort": "high"},
    messages=[{"role": "user", "content": "..."}]
)
```

### 6.3 Gemini参数调优

| 参数                | 作用            | 取值范围   | 说明                  |
| ------------------- | --------------- | ---------- | --------------------- |
| `temperature`       | 随机性/创造性   | 0.0 - 2.0  | 低=确定性，高=多样性  |
| `topK`              | 候选token池大小 | 1 - 40     | 1=贪心解码            |
| `max_output_tokens` | 回答长度上限    | 取决于模型 | 100 token ≈ 60-80单词 |

> ⚠️ **Gemini 3特殊说明**：强烈建议将`temperature`保留为默认值**1.0**。更改可能导致意外行为（循环或性能下降），特别是在复杂数学或推理任务中。

## 第七部分：工具与行动控制

### 7.1 工具使用模式

| 模式         | XML标签                            | 行为                             |
| ------------ | ---------------------------------- | -------------------------------- |
| **积极模式** | `<default_to_action>`              | 默认实施变更，意图不明时推断行动 |
| **保守模式** | `<do_not_act_before_instructions>` | 默认提供信息，仅明确要求时行动   |

### 7.2 并行工具调用

```
如果你打算调用多个工具，且这些工具调用之间没有依赖关系，
请并行执行所有独立的工具调用。

例如，在读取3个文件时，并行运行3个工具调用，
同时将这3个文件读入上下文。

切勿在工具调用中使用占位符或猜测缺失的参数。
```

### 7.3 高风险操作确认

需要确认的操作类型：

| 操作类型       | 示例                               |
| -------------- | ---------------------------------- |
| 破坏性操作     | 删除文件、删除分支、rm -rf         |
| 难以逆转的操作 | git push --force、git reset --hard |
| 对外可见的操作 | 推送代码、评论PR/Issue、发送消息   |

## 第八部分：提示分解策略

### 三种分解方法

| 方法         | 说明                       | 流程                             |
| ------------ | -------------------------- | -------------------------------- |
| **细分指令** | 复杂提示拆分为多个简单提示 | 复杂提示 → 拆分 → 按需选择       |
| **串联提示** | 分步骤依次处理             | 步骤1 → 步骤2 → 步骤3 → 最终输出 |
| **汇总回答** | 并行处理多个数据部分       | 数据A→任务A ↘ 汇总→最终输出      |

## 第九部分：AI的三大局限与应对

### 9.1 幻觉（Hallucination）

**定义**：AI生成表面上连贯可信、实则完全虚构的内容。

**应对策略**：
- 始终对AI输出保持**批判性审视**
- 对重要信息进行**人工验证**
- 意识到“可信度≠准确性”

**防幻觉提示词**：
```
对于你未打开过的代码，切勿猜测。如果用户引用了一个具体文件，
你必须在回答前读取该文件。

在调查之前，切勿对代码做出任何断言——除非你确信答案正确，
始终给出有根据、无幻觉的回答。
```

### 9.2 有害偏见

**典型案例**：MIT亚裔学生让AI将照片转为专业头像，结果生成的是**蓝眼睛、浅肤色**的版本。

**应对**：通过迭代开发和公众反馈来识别和消除偏见。

### 9.3 上下文遗忘

**问题**：AI可能在长对话中“忘记”之前的内容。

**应对**：
- 同一话题：正常延续
- 切换话题：**开启新会话**

## 第十部分：前端设计专项

### Claude Opus 4.8默认美学

| 元素     | 默认风格                                |
| -------- | --------------------------------------- |
| 背景色   | 暖奶油色/米白色（#F4F1EA）              |
| 标题字体 | Georgia / Fraunces / Playfair（衬线体） |
| 强调色   | 赤陶色/琥珀色                           |

### 改变默认风格的方法

**方法一：指定具体替代方案**
> 视觉方向应源自冷峻的单色氛围，使用浅银灰色调，逐渐加深为蓝灰色和近黑色……配色：#E9ECEC、#C9D2D4、#8C9A9E、#44545B、#11171B

**方法二：让AI先提议选项**
> 在开始构建之前，针对此需求提出4种不同的视觉方向。让用户选择一个，然后仅实现所选方向。

### 防“AI俗气”提示词

```
切勿使用通用的AI生成美学，例如过度使用的字体族
（Inter、Roboto、Arial、系统字体）、陈腐的配色方案
（特别是在白色或深色背景上的紫色渐变）、
可预测的千篇一律的布局和设计。

使用独特的字体、协调的色彩与主题，以及动画来实现效果。
```

## 第十一部分：提示词类型速查

| 类型              | 描述                     | 示例                                   | 适用场景          |
| ----------------- | ------------------------ | -------------------------------------- | ----------------- |
| **Zero-Shot**     | 直接给出清晰指令，无示例 | "用5个要点总结这篇文章"                | 快速获取通用答案  |
| **Few-Shot**      | 提供几个示例让AI模仿     | "这是2个示例总结，请按相同风格写第3个" | 需要特定格式/风格 |
| **Instructional** | 使用动词开头的直接命令   | "写一份执行摘要，100字以内"            | 任务明确时        |
| **Role-Based**    | 让AI扮演特定角色         | "你是一位MBA教授，准备讲座大纲..."     | 领域专业化输出    |
| **Contextual**    | 先提供背景再提问         | "本文档用于本科课程，请用通俗语言改写" | 针对特定受众      |

## 第十二部分：完整检查清单

```
□ 提供了足够的背景上下文？
□ 指令是否具体、明确？
□ 将查询/任务放在了文档末尾？（长上下文场景）
□ 指定了期望的输出格式？
□ 是否需要提供3-5个示例（Few-shot）？
□ 设定了合适的角色身份？
□ 明确了目标受众？
□ 添加了约束条件（长度、风格等）？
□ 使用了XML标签组织复杂提示？
□ 要求AI在不确定时主动提问？
□ 对于编码任务，设置了合适的effort级别？
□ 检查了temperature参数是否合适？
□ 如果需要工具调用，明确了何时以及如何使用？
```
